## ✅ 一句话先给你结论

> “在租户情况下请求的” ≠ “MQ Consumer 里有租户”

👉 你现在的请求，在「下单那一刻」确实是在租户上下文里  
👉 但 MQ Consumer 是“另一个线程、另一个世界”

---

## ✅ 二、你这个请求的真实时间线（非常关键）

### ✅ 1️⃣ 下单请求（✅ 有租户）

```
浏览器
 → 网关
 → 订单服务
 → 创建订单（tenant_id = 3）
 → 发起支付
```

✅ 这里 MP 自动插 tenant_id
✅ 一切正常

---

### ✅ 2️⃣ 支付成功（✅ 有租户）

```
支付系统
 → 支付成功
 → 发送 MQ
```

✅ Producer 可能也有租户
✅ 消息里带了 tenantId

---

### ❌ 3️⃣ MQ Consumer（❌ 没有租户）

```
RabbitMQ
 → Consumer 线程
 → 没有 HTTP 请求
 → 没有 JWT
 → 没有 SecurityContext
```

👉 MP 的 `TenantLineHandler.getTenantId()` = null

---

## ✅ 三、为什么“明明是租户请求，Consumer 却没有？”

因为这是 两个完全不同的执行上下文：

| 维度                   | 下单请求 | MQ Consumer |
| ---------------------- | -------- | ----------- |
| 请求来源               | HTTP     | MQ          |
| 是否有 JWT             | ✅        | ❌           |
| 是否有 SecurityContext | ✅        | ❌           |
| 是否有 ThreadLocal     | ✅        | ❌           |
| MP 能否拿到 tenantId   | ✅        | ❌           |

👉 它们只是“逻辑相关”，不是“上下文共享”

## ✅ 五、✅ 正确的理解方式（非常重要）

> MP 的租户插件：
>
> ❌ 不是“记住你是谁”
> ✅ 而是“每次执行 SQL 时临时问你：tenantId 是多少？”

👉 Consumer 执行 SQL 时，它再问一次
👉 你没给，它就填 null

## ✅ 六、✅ 正确、唯一的解决方案

### ✅ 消息里带 tenantId（你已经做到了 ✅）

```java
PaymentSuccessMessage {
    Long tenantId; // ✅ 必须有
}
```

---

### ✅ Consumer 手动“喂”给 MP

```java
@RabbitListener(queues = ...)
public void onMessage(PaymentSuccessMessage msg, Channel channel) {

    TenantContext.setTenantId(msg.getTenantId()); // ✅ 关键

    try {
        service.handlePaymentSuccess(msg);
    } finally {
        TenantContext.clear();
    }
}
```

---

### ✅ TenantLineHandler 从 ThreadLocal 读

```java
@Override
public Long getTenantId() {
    return TenantContext.getTenantId();
}
```

## ✅ 七、一句话终局（给你定心丸）

> 你没理解错，也不是 MP 的锅  
> 而是：MQ Consumer 是一个“全新的请求入口”

👉 只要是新入口
👉 就必须重新 setTenantId

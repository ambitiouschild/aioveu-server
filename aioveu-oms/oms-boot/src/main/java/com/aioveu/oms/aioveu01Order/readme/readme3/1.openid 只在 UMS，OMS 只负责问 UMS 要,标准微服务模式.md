> ✅ 不要在订单微服务里存 openid
>
> ✅ 统一去 ums（用户中心）查

## 一、为什么你的想法是「对的」

| 方案              | 问题           |
| ----------------- | -------------- |
| order 表存 openid | ❌ 数据冗余     |
| 多处维护 openid   | ❌ 极易不一致   |
| 用户换微信 / 解绑 | ❌ 订单表脏数据 |

✅ openid 是用户属性，不是订单属性

## 二、标准微服务架构（你现在的模型）

```
┌──────────────┐
│  UMS 用户中心 │  ← ✅ 唯一持有 openid
└──────┬───────┘
       │ Feign
┌──────▼───────┐
│ OMS 订单服务 │
└──────┬───────┘
       │
┌──────▼───────┐
│ 微信发货同步 │
└──────────────┘
```

## 三、✅ 正确实现方式（Feign 调用 UMS）

### 1️⃣ UMS 提供接口

```java
@GetMapping("/internal/openid/{memberId}")
public String getOpenidByMemberId(@PathVariable Long memberId) {
    return userService.getOpenidByMemberId(memberId);
}
```

---

### 2️⃣ OMS 定义 Feign Client

```java
@FeignClient(name = "aioveu-user")
public interface UserFeignClient {

    @GetMapping("/internal/openid/{memberId}")
    String getOpenidByMemberId(@PathVariable("memberId") Long memberId);
}
```

---

### 3️⃣ 发货 Service 中调用（✅ 推荐写法）

```java
// 1. 订单
OmsOrder order = orderService.getById(orderId);

// 2. 通过 memberId 查 openid（✅ 统一入口）
String openid = userFeignClient
        .getOpenidByMemberId(order.getMemberId());

if (StringUtils.isBlank(openid)) {
    throw new RuntimeException(
        "用户openid不存在，memberId=" + order.getMemberId()
    );
}

// 3. 微信发货参数
ObjectNode payer = body.putObject("payer");
payer.put("openid", openid);
```

## 四、⚠️ 必须考虑的 3 个边界问题（非常重要）

### ✅ 1. 用户未绑定微信

| 场景     | 处理               |
| -------- | ------------------ |
| 余额支付 | ❌ 不能同步微信发货 |
| 后台发货 | ❌ 同步失败         |

✅ 必须抛异常 + 告警

---

### ✅ 2. UMS 不可用（熔断）

```java
@FeignClient(
  name = "aioveu-user",
  fallback = UserFeignFallback.class
)
```

```java
@Component
public class UserFeignFallback implements UserFeignClient {

    @Override
    public String getOpenidByMemberId(Long memberId) {
        log.error("UMS 调用失败，memberId={}", memberId);
        return null;
    }
}
```

---

### ✅ 3. 性能问题（你担心的）

✅ 发货是低频操作（不是下单）
✅ Feign 调用一次完全可以接受

> 真到百万级订单再说缓存，现在不用过度设计

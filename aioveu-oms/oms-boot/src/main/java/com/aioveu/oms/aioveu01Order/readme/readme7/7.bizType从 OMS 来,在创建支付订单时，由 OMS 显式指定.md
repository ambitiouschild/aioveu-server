## 二、`bizType` 到底是什么？（必须先想清楚）

### ✅ `bizType` = 业务类型（Business Type）

它回答的问题是：

> “这笔支付，是为了什么业务场景？”

| bizType  | 含义       |
| -------- | ---------- |
| ORDER    | 商品订单   |
| RECHARGE | 账户充值   |
| DEPOSIT  | 押金       |
| REFUND   | 退款补差价 |
| VIP      | 会员购买   |

👉 Pay 不关心业务细节
👉 Pay 只关心“怎么收钱、怎么退钱”
👉 OMS / 其他服务关心“为什么收钱”

---

## 三、在你这套系统里，`bizType` 从哪里来？（标准答案）

### ✅ 来源链路（100% 正确）

```
用户下单
  ↓
OMS 创建订单
  ↓
OMS 明确知道：这是【商品订单】
  ↓
OMS 调用 PayFeignClient.createPayOrder(...)
  ↓
OMS 在 PayOrderForm 中显式指定 bizType = ORDER
  ↓
Pay 保存 pay_order.biz_type = ORDER
```

✅ bizType 是 OMS 的“业务声明”
✅ Pay 只做“存储 + 校验 + 路由”

## 四、代码层面：怎么传？（你这套系统的正确写法）

### ✅ 1️⃣ `BizTypeEnum`（放在 `pay-api`，唯一真理源）

```java
@Getter
public enum BizTypeEnum {

    ORDER("ORDER", "商品订单"),
    RECHARGE("RECHARGE", "账户充值"),
    DEPOSIT("DEPOSIT", "押金"),
    REFUND("REFUND", "退款"),
    VIP("VIP", "会员购买");

    @EnumValue
    private final String code;
    private final String desc;

    BizTypeEnum(String code, String desc) {
        this.code = code;
        this.desc = desc;
    }
}
```

✅ String 类型
✅ 语义清晰
✅ 扩展性强

---

### ✅ 2️⃣ `PayOrderForm`（pay-api，OMS 调用用）

```java
@Data
public class PayOrderForm {

    @NotBlank
    private String orderSn;

    @NotNull
    private BigDecimal amount;

    @NotNull
    private PaymentChannelEnum paymentChannel;

    @NotNull
    private PaymentMethodEnum paymentMethod;

    /** 业务类型（必须显式指定） */
    @NotNull(message = "业务类型不能为空")
    private BizTypeEnum bizType;
}
```

⚠️ 重点：
> ❌ 没有默认值
> ✅ 必须由 OMS 明确指定

---

### ✅ 3️⃣ OMS 调用 Pay 时（关键代码）

```java
PayOrderForm form = PayOrderForm.builder()
        .orderSn(order.getOrderSn())
        .amount(order.getTotalAmount())
        .paymentChannel(order.getPaymentChannel())
        .paymentMethod(order.getPaymentMethod())
        // ✅ 明确告诉 Pay：这是“商品订单”
        .bizType(BizTypeEnum.ORDER)
        .build();

payFeignClient.createPayOrder(form);
```

✅ OMS 拥有业务语义
✅ Pay 不做任何推断

## 五、Pay 微服务拿到 `bizType` 后干什么？

### ✅ Pay 的正确职责

| 行为             | 是否允许 |
| ---------------- | -------- |
| 存储 bizType     | ✅ 必须   |
| 校验合法性       | ✅ 必须   |
| 路由不同回调逻辑 | ✅ 推荐   |
| 推断 bizType     | ❌ 禁止   |
| 兜底默认值       | ❌ 禁止   |

### ✅ Pay 回调分发示例

```java
switch (payOrder.getBizType()) {
    case ORDER -> omsMqSender.send(payOrder);
    case RECHARGE -> accountMqSender.send(payOrder);
    case DEPOSIT -> depositMqSender.send(payOrder);
    default -> log.error("未知 bizType");
}
```

✅ Pay 不关心业务细节
✅ 只负责“把钱的结果告诉对的人”

## 六、为什么 Pay 不能自己“猜” bizType？（非常关键）

### ❌ 错误想法示例

```java
// ❌ 千万别这么干
if (orderSn.startsWith("ORD")) {
    payOrder.setBizType(BizTypeEnum.ORDER);
} else if (orderSn.startsWith("REC")) {
    payOrder.setBizType(BizTypeEnum.RECHARGE);
}
```

### ❌ 后果

• 规则一变，Pay 全炸

• 新业务必须改 Pay

• 支付中台失去“中立性”

• 资损风险


👉 Pay 一旦开始“理解业务”，它就不再是中台

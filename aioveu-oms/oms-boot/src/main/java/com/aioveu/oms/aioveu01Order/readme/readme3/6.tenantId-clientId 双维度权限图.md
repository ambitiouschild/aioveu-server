---

## ✅ 一句话结论（先给你）

> ✅ 数据隔离用 `tenantId`
>
> ✅ 微信身份用 `clientId`
>
> ✅ 两者必须共存，不是二选一

---

## 一、你现在纠结的本质

| 你担心的问题                  | 本质                 |
| ----------------------------- | -------------------- |
| 用 clientId 怎么过滤？        | ❌ 搞混了「隔离维度」 |
| tenantId 还能不能用？         | ✅ 当然能             |
| clientId 会不会破坏租户隔离？ | ❌ 不会               |

---

## 二、✅ 正确的数据模型（这是关键）

### ✅ 订单表（OmsOrder）

```sql
id
order_sn
tenant_id     ✅ 数据隔离
client_id     ✅ 微信身份
member_id
transaction_id
```

✅ tenantId 负责“我能看到哪些数据”
✅ clientId 负责“我用哪个微信 app 发货”

---

## 三、✅ 查询 / 过滤时怎么用？

### ✅ 场景 1：后台订单列表（租户隔离）

```java
LambdaQueryWrapper<OmsOrder> wrapper =
        new LambdaQueryWrapper<>();

wrapper.eq(OmsOrder::getTenantId, currentTenantId);
```

✅ 完全不受影响

---

### ✅ 场景 2：发货（微信身份）

```java
weChatApiClient.uploadShippingInfo(
        order.getClientId(),
        body
);
```

✅ tenantId 不参与微信调用

---

## 四、✅ 核心原则（记住这句）

> tenantId = 数据权限
>
> clientId = 业务能力

| 场景     | 用谁       |
| -------- | ---------- |
| 查订单   | ✅ tenantId |
| 改订单   | ✅ tenantId |
| 发货     | ✅ clientId |
| 微信退款 | ✅ clientId |
| 报表统计 | ✅ tenantId |

---

```
└─────┬──────┘
      │

│ tenantId   │ ✅
│ clientId   │ ✅
└─────┬──────┘
      │
┌─────▼──────┐
│ 微信发货   │
│ clientId   │ ✅
└────────────┘
```

---

## 六、✅ 你担心的“过滤失效”不会发生

| 错误理解                           | 正确理解 |
| ---------------------------------- | -------- |
| 有 clientId 就不能用 tenantId 过滤 | ❌        |
| clientId 会破坏租户隔离            | ❌        |
| 发货和查询要用同一个 ID            | ❌        |

✅ 它们是“正交维度”

---

## 七、✅ 给你一句“不会后悔”的设计口诀

> “租户管数据，应用管能力”

| 维度     | 管什么                 |
| -------- | ---------------------- |
| tenantId | 你能不能看见这条数据   |
| clientId | 你能不能用这个微信 app |

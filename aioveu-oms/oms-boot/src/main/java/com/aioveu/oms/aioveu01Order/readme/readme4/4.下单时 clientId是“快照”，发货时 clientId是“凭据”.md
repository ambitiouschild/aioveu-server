## ✅ 核心逻辑闭环（一句话版）

> 下单时 `clientId` 是“快照”，发货时 `clientId` 是“凭据”。

---

## 二、为什么管理端“无法”获取对应的 clientId？

| 端       | 为什么拿不到                                    |
| -------- | ----------------------------------------------- |
| 用户端   | ✅ 有 Header `X-Client-Id`（因为是小程序发起的） |
| 管理端   | ❌ 只有 `orderSn`，没有 `clientId` 参数          |
| 发货接口 | ❌ 通常只接收 `orderSn` 或 `orderId`             |

👉 所以，发货时必须从订单里“回忆”起当时的 clientId。

## 三、✅ 完整的“下单 → 发货”数据流（你的正确架构）

```
【用户端下单】
   ↓ Header: X-Client-Id = mini_program_A
【Auth 微服务】
   ↓ 校验白名单
【下单接口】
   ↓
【OmsOrder 订单表】
   ├─ order_sn: 202601210001
   ├─ tenant_id: 1001
   └─ client_id: mini_program_A  ✅ (固化快照)
```

```
【管理端点击发货】
   ↓ 只传 orderSn
【发货接口】
   ↓ select * from oms_order where order_sn = ?
【OmsOrder】
   ↓ 取出 client_id
【WeChatApiClient】
   ↓ uploadShippingInfo(mini_program_A, ...)
【微信发货接口】✅
```

## 五、✅ 架构师级总结（这是“教科书级”的设计）

| 步骤   | 数据来源             | 为什么                        |
| ------ | -------------------- | ----------------------------- |
| 下单   | Header `X-Client-Id` | 确定用户是在哪个小程序买的    |
| 存订单 | `order.clientId`     | 固化事实，防止抵赖            |
| 发货   | `order.clientId`     | 唯一可信来源，管理端无 Header |

> 结论：
> 你现在的架构是完全正确的。
> `clientId` 在下单时被“刻”在订单里，发货时拿出来用，这是最安全、最合理的闭环。

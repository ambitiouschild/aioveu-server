## ✅ 一句话终局结论（请截图）

> `ClientContextHolder.getClientId()` 在 `submitOrder()` 里  
> ❌ 不能保证一定能取到  
> ✅ 必须显式从请求头里拿，或者由 Controller 传进来

---

## ❌ 为什么你这里直接用不行？

### ❌ 1️⃣ `submitOrder()` 被 `@GlobalTransactional` 包裹

Seata 会：
• 开启新线程

• 切换线程上下文

• 清理 ThreadLocal


👉 `ClientContextHolder` 通常是 `ThreadLocal`

结果：
```java
ClientContextHolder.getClientId() == null
```

✅ 这是 Seata + ThreadLocal 的经典坑

---

### ❌ 2️⃣ 你前面自己已经踩过一次坑

你之前在 `confirmOrder()` 里写了这段话👇（非常关键）：

```java
// 解决子线程无法获取HttpServletRequest请求对象中数据的问题
RequestAttributes attributes = RequestContextHolder.getRequestAttributes();
RequestContextHolder.setRequestAttributes(attributes, true);
```

👉 说明你已经意识到：
> 异步 / 分布式事务线程里，ThreadLocal 会丢

---

### ❌ 3️⃣ `submitOrder()` 比 `confirmOrder()` 更危险

| 方法         | 有没有 Seata              |
| ------------ | ------------------------- |
| confirmOrder | ❌ 没有                    |
| submitOrder  | ✅ 有 @GlobalTransactional |

👉 Seata 一定会切线程
👉 ThreadLocal 一定丢

---

## ✅ 正确、安全、平台级写法（终局）

### ✅ ✅ 方案一（✅ 最推荐）：Controller 传进来（你现在这个是对的）

```java
public String submitOrder(OrderSubmitForm submitForm, String clientId)
```

✅ 100% 稳定
✅ 不依赖 ThreadLocal
✅ 不依赖 Seata
✅ 审计清晰

👉 这是平台级标准写法

---

### ✅ ✅ 方案二（✅ 如果你非要 Service 里取）

必须在 Controller 里先存到 ThreadLocal，并且手动继承

```java
// Controller
RequestAttributes attributes = RequestContextHolder.getRequestAttributes();
RequestContextHolder.setRequestAttributes(attributes, true);
ClientContextHolder.setClientId(request.getHeader("X-Client-Id"));
```

然后 Service 里：

```java
String clientId = ClientContextHolder.getClientId();
```

⚠️ 但仍然不推荐
⚠️ Seata 可能还是会清掉

## ✅ 你现在这段代码为什么是对的（重点）

```java
// Controller
public Result<String> submitOrder(
        @RequestHeader("X-Client-Id") String clientId,
        @Validated @RequestBody OrderSubmitForm submitForm
) {
    String orderSn = orderService.submitOrder(submitForm, clientId);
}
```

✅ clientId 来自 HTTP Header
✅ 不是 ThreadLocal
✅ 不受 Seata 影响
✅ 可审计、可测试

> 一律从方法参数里拿上下文

**一句话先给结论：**

> **transactionId ≠ paymentNo / outTradeNo**
>
> **它是“钱已经落袋”之后，微信/支付宝给你的“银行回单号”。**

而且：

> ✅ **订单表里可以有，但不该依赖**
>
> ✅ **真正的“唯一可信源”在支付表**
>
> ✅ **订单表只做“弱引用”**

下面我把这三者的关系一次性给你理到“不会再混”的程度。

------

## 一、三个号的身份对照表（记这张表就够了）

| 名称            | 谁生成         | 什么时候有     | 作用                        | 是否可变 |
| --------------- | -------------- | -------------- | --------------------------- | -------- |
| `paymentNo`     | 你（支付服务） | 创建支付订单时 | 支付系统主键、幂等键        | ❌ 永不   |
| `outTradeNo`    | 你（支付服务） | 创建支付订单时 | 传给微信/支付宝的商户订单号 | ❌ 永不   |
| `transactionId` | 微信/支付宝    | 用户付款成功后 | 第三方交易流水号            | ❌ 永不   |

👉 **订单表里：**

- 

  `outTradeNo`✅ **必须有**

- 

  `transactionId`⚠️ **可以有，但只是“附属信息”**

------

## 二、transactionId 到底是什么？

### ✅ 官方定义（微信）

> **微信支付订单号（transaction_id）**
>
> 是微信侧生成的**唯一交易流水号**

你可以把它理解为：

> **“银行打款回执单号”**

### 出现时机

| 阶段              | 是否有 transactionId  |
| ----------------- | --------------------- |
| 创建支付订单      | ❌                     |
| 用户扫码/拉起支付 | ❌                     |
| 用户付款成功      | ✅                     |
| 微信回调          | ✅                     |
| 退款              | ✅（原 transactionId） |

------

## 三、为什么订单表“不该依赖” transactionId？

你之前那句注释非常专业 👇

```
// 发货时：oms → 查 pay → 拿 transaction_id
```

✅ **这句话本身就是标准答案**

原因如下：

### ❌ 问题 1：订单表拿不到“第一时间”

- 

  回调先到 **支付服务**

- 

  订单服务可能是 **异步消费 MQ**

- 

  在订单服务里：

  - 

    要么还没回调

  - 

    要么回调丢了

👉 **用 transactionId 当条件 = 不稳定**

------

### ❌ 问题 2：transactionId 会变语义

- 

  全额退款 → transactionId 还在

- 

  部分退款 → 还是同一个

- 

  撤销 → 没 transactionId

👉 **订单状态不应该被 transactionId 驱动**

------

### ❌ 问题 3：对账才是它的主场

`transactionId`的真正用途是：

| 场景         | 谁用     |
| ------------ | -------- |
| 微信对账文件 | 财务     |
| 争议处理     | 客服     |
| 退款接口     | 支付服务 |
| 查单接口     | 支付服务 |

❌ **不是订单服务发货的依据**

------

## 四、订单表里 transactionId 的“正确定位”

### ✅ 可以存，但要明确“只读、非业务键”

```
/**
 * 微信/支付宝交易流水号（transaction_id）
 * 1. 支付成功后由回调写入
 * 2. 仅用于查询/对账/客服
 * 3. 不作为发货、状态判断依据
 */
private String transactionId;
```

### ✅ 写入时机

```
// 支付服务回调处理
payOrder.setTransactionId(wxCallback.getTransactionId());
payOrderMapper.updateById(payOrder);

// 订单服务（MQ 消费）
omsOrder.setTransactionId(payInfo.getTransactionId());
omsOrderMapper.updateById(omsOrder);
```

⚠️ **注意：**

- 

  订单表是“被动同步”

- 

  永远不从订单表反查支付

------

## 五、三个号在“发货链路”里的正确分工

### ✅ 正确链路（你现在这套是对的）

```
微信回调
 ↓
支付服务更新 pay_order
 ↓
发送 MQ：paymentNo + orderNo
 ↓
订单服务消费
 ↓
订单服务：
   - 用 paymentNo / outTradeNo 判断是否已支付
   - 忽略 transactionId
 ↓
发货 / 开通权益
```

### ❌ 错误链路（很多人踩）

```
订单服务：
   if (transactionId != null) {
       发货
   }
```

👉 **这是反模式**

------

## 六、退款时 transactionId 的角色（重点）

微信退款接口：

```
transactionId = "4200001234567890";
```

✅ **退款必须用 transactionId**

✅ **但退款一定发生在支付服务**

✅ **订单服务只传 paymentNo**

```
// 订单服务
refundFeign.refund(paymentNo);

// 支付服务
PayOrder payOrder = payOrderMapper.selectByPaymentNo(paymentNo);
wxPay.refund(payOrder.getTransactionId());
```

## 七、一句话终极总结（收藏级）

> **paymentNo / outTradeNo 是“你家的订单号”，
>
> transactionId 是“微信的银行回单号”。**
>
> ✅ 订单表里 **必须有 outTradeNo**
>
> ✅ transactionId **可以存，但只是备注**
>
> ✅ 真正干活、对账、退款，全靠 transactionId，但都在 **支付服务**

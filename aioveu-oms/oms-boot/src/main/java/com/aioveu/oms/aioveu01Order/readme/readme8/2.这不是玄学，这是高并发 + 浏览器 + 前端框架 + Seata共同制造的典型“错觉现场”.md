## 二、为什么你会觉得“不应该”？

### ① 你看到的是「单次点击」
但日志告诉你的是：

```
00:23:39.254  ✅ /orders/payment
00:23:39.306  ❌ /orders/
```

👉 同一个功能，同一个按钮，同一个用户
👉 但 Tomcat 收到了两个 HTTP 请求

你大脑说：“我只点了一下”
Tomcat 说：“你确实发了两次”

---

### ② 前端“没动”，但“行为变了”

最常见、但前端自己都察觉不到的 4 种情况：

| 场景                     | 前端“没改代码”但会多发 |
| ------------------------ | ---------------------- |
| 按钮没防抖               | onClick 触发多次       |
| 页面 onLoad + 按钮 click | 双管齐下               |
| axios 拦截器 retry       | 自动补刀               |
| 支付失败后自动重试       | 第二次 URL 拼错        |

👉 不是“改了代码”
👉 是“原来就有隐患，今天刚好触发”

---

### ③ Seata 放大了伤害

如果是普通 Spring 事务：
• ✅ 第一个请求：成功

• ❌ 第二个请求：404

• ✅ 最终结果是：成功


但你有 Seata GlobalTransactional：
• ✅ 第一个请求：DB 已提交

• ❌ 第二个请求：抛异常

• ❌ Seata：整个全局事务回滚


👉 你看到的是“莫名其妙回滚”
👉 本质是“第二个请求把第一个拖下水”

## 四、你现在的“不应该”，在技术上是怎么成立的？

```
你点一次支付
   ↓
前端 onClick 触发
   ↓
axios 请求发出 ✅ /orders/payment
   ↓
浏览器 / 框架 / 重试机制
   ↓
又发一次 ❌ /orders/
   ↓
Spring：没这个资源 → 抛异常
   ↓
Seata：有异常 → 回滚第一个请求
   ↓
你看到：啥都没动，钱也没扣，订单回滚
```

👉 每一个环节都“合理”
👉 合在一起就是“不应该但发生了”

这是一个非常好的问题，它触及了API设计中的一个常见场景。在订单统计这个例子中，使用 **`POST`** 而非 **`GET`** 主要基于以下几个技术性考量，这通常被认为是更合适和务实的选择：

### 核心原因：查询条件的复杂度

1. **参数复杂，不适合放在URL中** `GET`请求的参数是通过URL的查询字符串（`?key1=value1&key2=value2`）传递的。订单统计查询条件（`OrderPageQuery`）通常包含多个字段，如时间范围、订单状态、商品关键词、用户信息、支付方式等。当这些参数组合起来时，URL会变得非常长，可能超出浏览器或服务器的长度限制，且难以阅读和维护。
2. **请求体支持结构化数据** `POST`请求可以将一个结构化的JSON对象（如 `OrderPageQuery`对象）放在请求体（Request Body）中发送。这种方式能清晰地表达复杂、嵌套的查询条件，并且与后端定义的Java对象模型完美映射，方便框架（如Spring MVC）自动进行反序列化和参数绑定。

### 语义与操作性质的考量

1. **“统计”更接近“执行查询”，而非“获取资源”** 虽然RESTful风格通常建议用 `GET`获取资源，但“获取订单统计”这个操作，本质上不是获取一个预先存在的、有唯一ID的资源。它更像是**向服务器提交一个查询请求，由服务器执行一个计算（聚合、统计）操作，然后返回计算结果**。这种“命令”或“动作”的语义，使用 `POST`在概念上同样合理，甚至更贴切。
2. **与“分页查询”接口保持一致** 在您之前的讨论中，订单分页查询接口也使用了 `POST`。这使得前后端交互模式统一：所有复杂的、带条件的查询请求，都通过 `POST`发送一个查询条件对象（`OrderPageQuery`）到相应的端点（如 `/order/page`和 `/order/statistics`）。这简化了前端调用逻辑和HTTP客户端的配置。

### 安全性与无副作用

1. **安全性无差异** 无论是 `GET`还是 `POST`，获取统计信息的接口都应该是**安全**（Safe）和**幂等**（Idempotent）的，即多次调用不会改变服务器状态。在这一核心特性上，两者没有区别。使用 `POST`并不会改变这个接口的只读属性。
2. **对敏感数据的保护** 虽然查询参数本身可能不敏感，但使用 `POST`可以将查询条件放在请求体中，而非明文的URL里。这在某些场景下（如日志记录）可以减少敏感信息（如复杂筛选条件）的暴露。不过，这通常不是主要原因。

### 与 `GET`的对比

使用 `GET`的主要优势在于可缓存性和易于直接调用（如在浏览器地址栏输入）。但对于**实时性强、参数组合多变**的订单统计接口，其结果缓存价值不大，且复杂的参数使得“直接调用”并不方便。因此，`GET`的优势在此并不明显。

**总结来说**，为订单统计接口选择 `POST`方法，主要是一个**实用性优于教条**的设计决策。它优雅地解决了复杂查询参数的传递问题，与执行聚合计算的语义相符，并保持了与系统中其他复杂查询接口的一致性。这是当前在构建需要接受复杂查询条件的RESTful或类RESTful API时，一种非常常见且被广泛接受的做法。

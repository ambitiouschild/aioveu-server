## 二、OMS 为什么用 INT（现实原因）

### ✅ 1️⃣ OMS 是“订单中心”，不是“支付专家”

OMS 对支付只关心三件事：

• 有没有付？

• 什么时候付的？

• 付了多少？


👉 它不关心：
• 是 JSAPI 还是 NATIVE

• 是 ALIPAY_WAP 还是 ALIPAY_APP


所以 OMS 只需要：

```java
PaymentChannelEnum {
    ALIPAY(1),
    WECHAT(2),
    UNIONPAY(3),
    BALANCE(4)
}
```

✅ 1 / 2 / 3 / 4
• 索引小

• 性能好

• 历史兼容

• 财务对账稳


---

### ✅ 2️⃣ OMS 是“老系统改造”

你前面自己说过：

> OMS 原来就是数字

👉 这是事实约束，不是设计偏好

• 改 VARCHAR = 全量数据迁移

• 改 VARCHAR = 索引重建

• 改 VARCHAR = 历史报表全炸


✅ INT 是对现实的妥协，也是专业

---

### ✅ 3️⃣ OMS 的支付字段是“结果型”

```sql
oms_order_pay.pay_type = 2
```

意思是：

> “这笔订单，最终是用微信付的”

✅ 不需要知道是微信 JSAPI 还是微信 APP
✅ 这是“事后结论”，不是“支付过程”

---

## 三、PAY 为什么用 String（这才是关键）

### ✅ 1️⃣ PAY 是“支付中台”，要对齐三方标准

微信 / 支付宝 / 银联 全是 String：

| 渠道   | 三方返回                           |
| ------ | ---------------------------------- |
| 微信   | `JSAPI / APP / NATIVE / MWEB`      |
| 支付宝 | `QUICK_MSECURITY_PAY / WAP / PAGE` |
| 银联   | `APP / WEB`                        |

👉 你用 INT 试试？
• 微信加一个新场景 = 你改表 + 发版

• 支付宝升级 = 你全链路改 Enum


✅ String = 天然对齐三方

---

### ✅ 2️⃣ PAY 的支付渠道是“过程型”

PAY 关心的是：

• 怎么付的？

• 走的哪个产品码？

• 调的哪个接口？


```java
PaymentChannelEnum {
    ALIPAY("ALIPAY"),
    WECHAT("WECHAT"),
    UNIONPAY("UNIONPAY")
}

PaymentMethodEnum {
    JSAPI("JSAPI"),
    APP("APP"),
    NATIVE("NATIVE"),
    MWEB("MWEB")
}
```

✅ 这是“支付工程问题”
✅ 不是订单统计问题

---

### ✅ 3️⃣ PAY 是“唯一真理源”

> PAY 说了算，OMS 只认结果

• PAY 用 String 定义“什么是支付”

• OMS 用 INT 记录“付没付”

• Enum 在 Java 层统一语义


✅ 这是中台架构的精髓

---

## 四、它们是怎么“和平共存”的？（重点）

### ✅ Java 层：一套 Enum，两种存储

```java
@Getter
public enum PaymentChannelEnum {

    ALIPAY(1, "ALIPAY"),
    WECHAT(2, "WECHAT");

    private final int code;        // OMS
    @EnumValue
    private final String name;     // PAY
}
```

| 模块 | 用哪个           |
| ---- | ---------------- |
| OMS  | `code` → INT     |
| PAY  | `name` → VARCHAR |
| Java | Enum 本身        |

✅ 这是你这套系统“最优雅的点”

---

### ✅ 数据流长这样（你现在的架构）

```
OMS(order)
  └── payment_channel = 2

PAY(pay_order)
  └── payment_channel = 'WECHAT'
  └── payment_method = 'JSAPI'

回调
  └── OMS 写 oms_order_pay.pay_type = 2
```

✅ 没有矛盾
✅ 没有重复造轮子
✅ 这是“专业拆分”

>

>
> ✅ Enum 是 Java 层的唯一真理源

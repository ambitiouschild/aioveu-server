## 二、`oms_order_pay` 本来是干啥的？（历史真相）

从字段设计一眼就能看出来，这是“电商订单系统里的支付快照表”：

| 字段               | 含义                       |
| ------------------ | -------------------------- |
| `order_id`         | 关联 OMS 订单              |
| `pay_sn`           | 支付流水号（来自支付系统） |
| `pay_amount`       | 实际支付金额               |
| `pay_type`         | 支付方式（1/2/3/4）        |
| `pay_status`       | 支付状态                   |
| `pay_subject`      | 交易描述                   |
| `callback_content` | 回调原始报文               |
| `confirm_time`     | 确认时间                   |

👉 它描述的是：
> “某订单，某一次，有没有付钱、怎么付的、付了多少”

✅ 这是 OMS 的“支付结果视图”
❌ 不是 Pay 服务的“支付单”

## 三、它和 `pay_order` 的区别（非常关键）

| 维度         | oms_order_pay    | pay_order         |
| ------------ | ---------------- | ----------------- |
| 所属系统     | OMS              | PAY               |
| 关注点       | “订单有没有付钱” | “支付单生命周期”  |
| 是否驱动业务 | ✅ 是             | ❌ 否              |
| 是否对接三方 | ❌ 不对接         | ✅ 对接微信/支付宝 |
| 是否幂等     | ❌ 不唯一         | ✅ 唯一            |
| 是否可退款   | ❌ 不负责         | ✅ 负责            |

👉 它们的关系是：

```
oms_order
   └── oms_order_pay（OMS 视角的支付结果）
           ↑
      pay_order（支付中台唯一真理）
```

✅ 不是重复
✅ 是“视角不同”

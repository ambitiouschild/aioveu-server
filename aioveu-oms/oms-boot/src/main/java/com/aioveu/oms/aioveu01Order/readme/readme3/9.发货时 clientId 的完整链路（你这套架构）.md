---

## ✅ 一句话终极定义（先给你）

> 微信发货里的 `clientId`，就是你系统里「微信小程序的唯一业务身份标识」

它不是：
• ❌ 微信的 `appid`

• ❌ 数据库的流水号

• ❌ 随便起的一个名字


它是：
• ✅ 你系统 → 微信生态的「桥梁主键」


---

## 二、它在微信发货里的「真实作用」

| 你在发货代码里做的事                | clientId 的作用          |
| ----------------------------------- | ------------------------ |
| `getAccessToken(clientId)`          | ✅ 找 `appid + secret`    |
| `uploadShippingInfo(clientId, ...)` | ✅ 决定用哪个小程序身份   |
| Redis token key                     | ✅ 隔离不同小程序的 token |
| 防并发锁                            | ✅ 防止同一小程序重复获取 |

👉 一句话：
> 没有 clientId，你根本不知道“用哪个微信身份去发货”

---

## 三、它和你那张关系表的对应关系（✅ 关键）

```
┌────────────────────────────┐
│ sys_oauth2_client_wx_app  │
│----------------------------│
│ client_id (PK)            │ ← ✅ 你用的 clientId
│ appid                     │ ← ✅ 微信要的
│ app_secret                │ ← ✅ 微信要的
│ enabled                   │ ← ✅ 业务开关
│ is_deleted                │ ← ✅ 逻辑删除
└────────────────────────────┘
              │
              ▼
┌────────────────────────────┐
│ oauth2_registered_client   │
│----------------------------│
│ client_id                  │ ← ✅ 和上面保持一致
│ client_settings            │ ← ✅ Spring OAuth2
└────────────────────────────┘
```

✅ clientId 是连接你系统和微信的“锚点”

---

## 四、发货时 clientId 的完整链路（你这套架构）

```
订单 orderSn
   ↓
OmsOrder.clientId
   ↓
sys_oauth2_client_wx_app
   ↓
appid + app_secret
   ↓
微信 access_token
   ↓
微信发货接口 ✅
```

✅ 订单只认 clientId
✅ 不关心 appid
✅ 不关心 tenantId

---

## 五、为什么不用 appid 直接当 clientId？

| 原因           | 说明                     |
| -------------- | ------------------------ |
| 微信控制 appid | ❌ 不可控                 |
| 多环境         | ❌ 测试 / 生产 appid 不同 |
| 多租户         | ❌ 不安全                 |
| 业务解耦       | ✅ clientId 是你自己的    |

👉 clientId 是你的“内部稳定标识”

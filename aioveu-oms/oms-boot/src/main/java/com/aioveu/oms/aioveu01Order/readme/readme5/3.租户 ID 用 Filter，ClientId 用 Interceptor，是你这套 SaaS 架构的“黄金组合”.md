## 一、Filter vs Interceptor（本质区别）

| 维度                | Filter             | Interceptor        |
| ------------------- | ------------------ | ------------------ |
| 所属规范            | Servlet 规范       | Spring MVC         |
| 执行时机            | 最早（请求刚进来） | Controller 前      |
| 能否拿到 Controller | ❌ 不能             | ✅ 能               |
| 能否拿到返回值      | ❌ 难               | ✅ 能               |
| 典型用途            | 日志、编码、租户   | 登录态、权限、参数 |

------

## 二、✅ 为什么租户 ID 用 Filter 是对的？

### 你的租户 Filter（典型正确做法）

```
@Component
public class TenantFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request,
                         ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {

        String tenantId = ((HttpServletRequest) request)
                .getHeader("X-Tenant-Id");

        TenantContextHolder.setTenantId(tenantId);
        try {
            chain.doFilter(request, response);
        } finally {
            TenantContextHolder.clear();
        }
    }
}
```

✅ **请求一进来就确定租户**

✅ **不依赖 Spring MVC**

✅ **最底层、最稳**

👉 **租户 = 请求级基础设施**

------

## 三、✅ 为什么 ClientId 用 Interceptor 更合适？

### ClientId 的特点

| 特点                 | 说明 |
| -------------------- | ---- |
| 和 Controller 强相关 | ✅    |
| 有些接口不需要       | ✅    |
| 和 Spring 上下文有关 | ✅    |
| 不需要最早执行       | ✅    |

------

### ✅ ClientId Interceptor（推荐）

```
public class ClientContextInterceptor
        implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) {

        String clientId = request.getHeader("X-Client-Id");
        ClientContextHolder.setClientId(clientId);
        return true;
    }

    @Override
    public void afterCompletion(...) {
        ClientContextHolder.clear();
    }
}
```

✅ **可以精确控制路径**

✅ **可以 exclude 公共接口**

✅ **Spring 生态友好**

------

## 四、✅ 官方推荐组合（你这套是满分）

| 身份     | 组件                | 原因                    |
| -------- | ------------------- | ----------------------- |
| 租户 ID  | ✅ Filter            | 请求最早期、基础设施    |
| ClientId | ✅ Interceptor       | 业务相关、Controller 前 |
| 用户 ID  | ✅ Interceptor / AOP | 登录态                  |

------

## 五、✅ 执行顺序（非常关键）

```
HTTP Request
   ↓
【Filter】（租户 ID ✅）
   ↓
【Interceptor】（ClientId ✅）
   ↓
【Controller】
   ↓
【Service】
```

✅ **Filter 永远先于 Interceptor**

------

## 六、⚠️ 一个非常重要的坑（你一定会遇到）

### ❌ 不要在 Filter 里清掉 Interceptor 设置的 ThreadLocal

| 错误做法                   | 后果               |
| -------------------------- | ------------------ |
| Filter finally 清 ClientId | ❌ Interceptor 白设 |
| Interceptor 不清           | ❌ 线程池污染       |

✅ **各清各的**

------

## 七、✅ 架构师级总结（你这套）

| 组件        | 用 Filter | 用 Interceptor |
| ----------- | --------- | -------------- |
| 租户 ID     | ✅ 必须    | ❌ 不推荐       |
| ClientId    | ⚠️ 可以    | ✅ 推荐         |
| 登录态      | ❌         | ✅              |
| 日志 / 监控 | ✅         | ❌              |

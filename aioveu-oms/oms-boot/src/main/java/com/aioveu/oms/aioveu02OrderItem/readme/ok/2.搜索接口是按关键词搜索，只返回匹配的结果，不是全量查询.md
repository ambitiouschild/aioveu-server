我的意思是这个搜索接口的查找条件是啥，难道不是全查吗

**不是全查！** 搜索接口是**按关键词搜索**，只返回匹配的结果，不是全量查询。

## 搜索接口的工作原理

### 搜索场景举例：

1. **用户输入："202310"**
   - 后端SQL：`WHERE order_sn LIKE '202310%'`
   - 结果：只返回以"202310"开头的订单
2. **用户输入："张三"**
   - 后端SQL：`WHERE customer_name LIKE '%张三%'`
   - 结果：只返回客户名包含"张三"的订单
3. **用户什么都没输入**
   - 结果：返回空数组，或者最近的一些订单

## 具体实现

### 1. 后端搜索接口

```
@GetMapping("/api/orders/search")
public List<OrderOptionVO> searchOrders(@RequestParam(required = false) String keyword) {
    if (StringUtils.isBlank(keyword)) {
        // 如果没输入关键词，可以返回空，或者返回最近的一些订单
        return Collections.emptyList();
        // 或者：return orderService.getRecentOrders(20);
    }
    
    // 只搜索匹配的订单
    return orderService.searchOrders(keyword, 20);  // 最多返回20条
}
```

### 2. Service层实现

```
@Service
public class OrderServiceImpl implements OrderService {
    
    public List<OrderOptionVO> searchOrders(String keyword, int limit) {
        // 构建查询条件
        LambdaQueryWrapper<Order> wrapper = new LambdaQueryWrapper<>();
        
        // 只查未删除的
        wrapper.eq(Order::getDeleted, 0);
        
        // 按订单号或客户名搜索
        wrapper.and(w -> w
            .like(Order::getOrderSn, keyword + "%")  // 订单号前缀匹配
            .or()
            .like(Order::getCustomerName, "%" + keyword + "%")  // 客户名模糊匹配
        );
        
        // 限制数量
        wrapper.last("LIMIT " + limit);
        
        // 执行查询
        List<Order> orders = orderMapper.selectList(wrapper);
        
        // 转换为VO
        return orders.stream()
            .map(this::convertToOptionVO)
            .collect(Collectors.toList());
    }
}
```

### 3. 生成的SQL

```
-- 用户输入"202310"时
SELECT id, order_sn, customer_name, total_amount 
FROM oms_order 
WHERE deleted = 0 
  AND (order_sn LIKE '202310%' OR customer_name LIKE '%202310%')
LIMIT 20;

-- 只返回最多20条匹配记录，不是全表！
```

## 为什么不是全查？

| 场景         | 数据量                            | 性能       |
| ------------ | --------------------------------- | ---------- |
| **全表查询** | 假设10万订单                      | 慢，占内存 |
| **搜索查询** | 用户输入"202310"，可能只匹配100个 | 快，内存小 |

## 搜索优化

### 1. 分页搜索

```
public List<OrderOptionVO> searchOrders(String keyword, int page, int size) {
    PageHelper.startPage(page, size);
    
    // ... 查询条件
    
    // PageHelper会自动加上LIMIT
    List<Order> orders = orderMapper.selectList(wrapper);
    return convertToVO(orders);
}
```

### 2. 字段索引优化

```
-- 为搜索字段创建索引
CREATE INDEX idx_order_sn ON oms_order(order_sn);
CREATE INDEX idx_customer_name ON oms_order(customer_name);
```

### 3. 搜索结果优化

```
// 可以按相关度排序
wrapper.orderByDesc(Order::getCreateTime);  // 最新的在前
// 或者：wrapper.orderByAsc(Order::getOrderSn);  // 按订单号排序
```

## 前端调用示例

```
// 用户输入"202310"时调用
const response = await fetch('/api/orders/search?keyword=202310');
const orders = await response.json();
// 返回: [{id: 123, orderSn: "202310250001", ...}, ...] 最多20条

// 用户输入"张三"时调用
const response = await fetch('/api/orders/search?keyword=张三');
const orders = await response.json();
// 返回: [{id: 124, orderSn: "202310250002", customerName: "张三"}, ...]
```

## 搜索流程图示

```
用户操作: 输入"202310"
         ↓
前端: 调用 /api/orders/search?keyword=202310
         ↓
后端: 执行SQL: WHERE order_sn LIKE '202310%' LIMIT 20
         ↓
数据库: 用索引快速查找，返回匹配的20条
         ↓
前端: 显示这20条供用户选择
         ↓
用户: 选择一条，得到ID: 123
```

## 总结

**搜索接口不是全查，是：**

1. **按关键词过滤**：只查匹配的记录
2. **数量限制**：通常限制20-50条
3. **使用索引**：数据库快速查询
4. **实时搜索**：用户输入时实时触发

**所以性能完全没问题**，用户输入"202310"，数据库用索引快速找到匹配的订单，最多返回几十条，不是全表扫描10万条数据。

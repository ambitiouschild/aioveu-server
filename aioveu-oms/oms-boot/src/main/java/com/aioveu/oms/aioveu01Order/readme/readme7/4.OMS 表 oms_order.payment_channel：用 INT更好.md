| 历史兼容性 | ✅ 适合老系统 | ✅ 适合新系统      |
| 多端协作   | ✅ 枚举统一   | ❌ 前端乱传        |
| 你这个场景 | ✅ 完美匹配   | ❌ 不匹配          |

## 三、为什么你这个场景**必须用 INT**

### ✅ 1️⃣ 你已经有“唯一真理源”

```java
PaymentChannelEnum
├── ALIPAY(1, "ALIPAY")
├── WECHAT(2, "WECHAT")
└── UNIONPAY(3, "UNIONPAY")
```

👉 Java 层是 Enum
👉 DB 层是 INT
👉 这是行业标准

---

### ✅ 2️⃣ OMS 是“历史系统 + 数字存量”

你前面说过：

> OMS 原来就是数字
> PAY 用字符串

👉 OMS 继续用 INT，是对历史的尊重
👉 PAY 用字符串，是对未来的负责

✅ 这是“异构存储”的典型正确解

---

### ✅ 3️⃣ 避免“字符串污染数据库”

如果存字符串：

```sql
payment_channel = 'WECHAT'
```

将来你一定会遇到：

• `WeChat`

• `wechat`

• `WX`

• `WXPAY`


👉 DB 直接变垃圾堆

## 四、✅ 正确建模方式（你这套系统的最优解）

### ✅ OMS 表结构

```sql
ALTER TABLE oms_order
ADD COLUMN payment_channel INT COMMENT '支付渠道：1-支付宝 2-微信 3-银联';
```

---

### ✅ OMS 实体

```java
@Data
@TableName("oms_order")
public class OmsOrder {

    /** 支付渠道（INT → Enum） */
    @TableField(typeHandler = PaymentChannelEnumCodeTypeHandler.class)
    private PaymentChannelEnum paymentChannel;
}
```

---

### ✅ TypeHandler（OMS 专用）

```java
@MappedTypes(PaymentChannelEnum.class)
public class PaymentChannelEnumCodeTypeHandler
        extends BaseTypeHandler<PaymentChannelEnum> {

    @Override
    public void setNonNullParameter(
            PreparedStatement ps,
            int i,
            PaymentChannelEnum parameter,
            JdbcType jdbcType) throws SQLException {
        ps.setInt(i, parameter.getCode());
    }

    @Override
    public PaymentChannelEnum getNullableResult(ResultSet rs, String columnName)
            throws SQLException {
        return PaymentChannelEnum.fromCode(rs.getInt(columnName));
    }

    @Override
    public PaymentChannelEnum getNullableResult(ResultSet rs, int columnIndex)
            throws SQLException {
        return PaymentChannelEnum.fromCode(rs.getInt(columnIndex));
    }

    @Override
    public PaymentChannelEnum getNullableResult(CallableStatement cs, int columnIndex)
            throws SQLException {
        return PaymentChannelEnum.fromCode(cs.getInt(columnIndex));
    }
}
```

## 五、为什么 PAY 用字符串，OMS 用 INT 是合理的？

| 系统 | 职责     | 存储                   |
| ---- | -------- | ---------------------- |
| OMS  | 订单中心 | INT（历史 + 性能）     |
| PAY  | 支付中台 | VARCHAR（语义 + 扩展） |

👉 它们通过 Enum 统一语义
👉 通过 TypeHandler 屏蔽差异

✅ 这是中台架构的标准解法

## 六、❌ 不推荐的方案（说清楚为什么）

### ❌ OMS 也用 VARCHAR

• 破坏历史数据

• 索引膨胀

• 迁移成本高

• 收益极小


### ❌ OMS 用 TINYINT + 不用 Enum

• Java 层失去约束

• 校验分散

• 后期维护地狱


---

## 七、一句话封死这个问题 ✅

> ✅ OMS：`INT + Enum + TypeHandler`
>
> ✅ PAY：`VARCHAR + Enum + @EnumValue`
>
> ✅ Java 层永远只认 Enum
>
> ❌ OMS 不要用字符串存渠道

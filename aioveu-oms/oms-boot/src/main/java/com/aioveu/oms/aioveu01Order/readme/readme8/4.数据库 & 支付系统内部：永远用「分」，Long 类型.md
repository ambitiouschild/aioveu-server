# ✅ 一句话结论（先记住这个）

> 数据库 & 支付系统内部：永远用「分」，Long 类型  
> BigDecimal：只在“元”的场景出现  
> 前端：永远传「元」，BigDecimal

---

## 一、BigDecimal 是元还是分？

### ✅ 正确答案：

> BigDecimal = 元（展示 & 交互）  
> Long = 分（计算 & 存储 & 比较）

| 层级          | 类型       | 单位 | 是否可信   |
| ------------- | ---------- | ---- | ---------- |
| 前端          | BigDecimal | 元   | ❌ 不可信   |
| 后端接口层    | BigDecimal | 元   | ❌ 只接收   |
| 业务层        | Long       | 分   | ✅ 唯一可信 |
| 数据库        | BIGINT     | 分   | ✅ 最终落地 |
| 微信 / 支付宝 | int        | 分   | ✅ 官方标准 |

---

## 二、前端应该传什么？（标准答案 ✅）

### ✅ 前端统一传：**元（BigDecimal）**

```json
{
  "orderNo": "ORD123",
  "amount": 99.99
}
```

❌ 前端永远不要传分

原因：
• 前端算分 100% 会出 Bug

• 用户眼里只有“元”

• 99.99 元 ≠ 9999 分（心理成本）

## 三、Pay 服务拿到后怎么处理？（关键）

### ✅ 正确流程（你现在这个系统就该这样）

```java
// 1️⃣ 前端 / OMS 传进来的是元
BigDecimal amountYuan = paymentForm.getAmount();

// 2️⃣ 立刻转成分（唯一入口）
Long amountFen = yuanToFen(amountYuan);

// 3️⃣ 后面所有逻辑只用 amountFen
if (amountFen <= 0) {
    throw new BizException("金额非法");
}
```

### ✅ 元 → 分（标准工具方法）

```java
private Long yuanToFen(BigDecimal yuan) {
    return yuan
            .setScale(2, RoundingMode.HALF_UP)
            .multiply(BigDecimal.valueOf(100))
            .longValueExact();
}
```

## 四、PayOrder 里 BigDecimal 是干嘛的？

### ✅ PayOrder 表里可以有 BigDecimal（元）

```sql
payment_amount DECIMAL(10,2) COMMENT '单位：元'
```

但——
> 它只是“展示字段”
>
> 真正用来比较、计算、防资损的，一定是分

### ✅ 正确设计（推荐）

```java
// 元（展示用）
private BigDecimal paymentAmountYuan;

// 分（资金用）
private Long paymentAmountFen;
```

或者更狠一点（✅ 我推荐）：

```java
// 只存分
private Long paymentAmount;
```

展示时再转：

```java
BigDecimal yuan = BigDecimal.valueOf(paymentAmount)
        .divide(BigDecimal.valueOf(100), 2, RoundingMode.HALF_UP);
```

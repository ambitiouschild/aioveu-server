## ✅ 一句话结论（先记死）

> ❌ **`transaction_id`不应该写在 `oms_order`里**
>
> ✅ **必须写在 `pay_order`里**

> ✅ **发货时：oms → 查 pay → 拿 transaction_id**

------

## 一、为什么不能写在 `oms_order`里（三个硬理由）

### 1️⃣ 职责边界（最重要）

| 表          | 职责                         |
| ----------- | ---------------------------- |
| `oms_order` | 货：买了什么、发不发、发给谁 |
| `pay_order` | 钱：付没付、付多少、微信单号 |

👉 **`transaction_id`是“钱”的属性**

👉 不是“货”的属性

如果你写在 `oms_order`里：

- 

  以后退款、分账、对账、补单

- 

  都要从 oms 拿

- 

  oms 会变成“支付中心”

------

### 2️⃣ 一笔订单可能有多笔支付（设计前瞻性）

| 场景     | 结果              |
| -------- | ----------------- |
| 部分退款 | 多 transaction_id |
| 组合支付 | 多 transaction_id |
| 补差价   | 新 transaction_id |

👉 `oms_order`只能存一个

👉 `pay_order`可以多条

------

### 3️⃣ 微信回调只关心“支付订单”

微信回调内容：

```
{
  "out_trade_no": "PAY20260627...",
  "transaction_id": "420000xxxx"
}
```

👉 这个 `out_trade_no`就是 **pay_order.payment_no**

👉 回调落 `pay_order`是最自然的

------

## 二、正确的数据流向（你这套架构的“标准答案”）

```
微信回调
 ↓
pay-service
 ↓
pay_order.transaction_id ✅
 ↓
更新 pay_order 状态
 ↓
通知 oms：订单已支付
```

### ✅ oms 里只需要知道：

```
order.setStatus(PAID);
order.setPaymentTime(now);
```

❌ **不需要知道 transaction_id**

------

## 三、发货时“正确且稳定的获取方式”

### ✅ 发货接口（oms 微服务内）

```
public void uploadShipping(Long orderId) {

    // 1. 查业务订单
    OmsOrder order = orderMapper.selectById(orderId);

    // 2. 查支付订单（✅ 正确来源）
    PayOrder payOrder = payOrderService.getByOrderNo(order.getOutTradeNo());

    // 3. 拿微信 transaction_id
    String transactionId = payOrder.getTransactionId();
    if (StringUtils.isBlank(transactionId)) {
        throw new BusinessException("微信支付单号未回写，无法发货");
    }

    // 4. 调用微信发货
    weChatApiClient.uploadShippingInfo(transactionId, body);
}
```

✅ **transaction_id 永远只从 pay 来**

✅ oms 不存、不维护、不计算

## 四、你担心的“多一次 RPC”是不是问题？

### ✅ 不是问题，而且是好事

| 担心            | 实际情况                     |
| --------------- | ---------------------------- |
| 多一次 RPC 慢   | 发货是低频操作，可忽略       |
| 依赖 pay 不稳定 | 支付是核心链路，本来就要稳定 |
| 以后不用 pay 了 | 不可能                       |

👉 **发货本来就是“钱 + 货”的强一致性操作**

👉 查一次 pay 是**正确耦合**，不是错误耦合

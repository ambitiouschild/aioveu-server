## ✅ 什么时候会“看起来像阻塞 / 卡死”？

### ❌ 1️⃣ 下游服务慢

| 服务   | RT    |
| ------ | ----- |
| Tenant | 50ms  |
| Member | 100ms |
| Pay    | 200ms |
| WeChat | 800ms |

👉 发货接口 RT ≈ 1.2s
👉 线程被占满 → 新请求进不来

---

### ❌ 2️⃣ 下游服务挂了（最危险）

```
Order → Tenant ❌ 超时
```

结果：
• Order 线程卡住

• Tomcat 线程耗尽

• 整个订单服务不可用


✅ 这就是“级联阻塞”

---

### ❌ 3️⃣ Seata + Feign + 线程池（你这套高危点）

你已经有：
• `@GlobalTransactional`

• `CompletableFuture`

• 线程池

• Feign


👉 任何一个下游慢，整条链路都会被拖死

## ✅ 哪些“阻塞是正常的”，哪些是“灾难”？

| 情况                   | 是否正常 |
| ---------------------- | -------- |
| 单次 Feign 调用几十 ms | ✅ 正常   |
| 发货接口几百 ms        | ✅ 正常   |
| 一个请求占用线程 2s    | ⚠️ 危险   |
| 下游挂了，上游全卡     | ❌ 灾难   |

## ✅ 你现在这套，怎么“保命”（终局级）

### ✅ 1️⃣ 发货 **必须异步化（最重要）**

❌ 错误：
```java
@PostMapping("/ship")
public Result ship() {
    weChatApiClient.uploadShipping(...); // 同步
}
```

✅ 正确：
```java
@PostMapping("/ship")
public Result ship() {
    mqSender.sendShippingMessage(orderId);
    return Result.success();
}
```

👉 发货 = 最终一致
👉 用户不需要等微信返回

---

### ✅ 2️⃣ 所有 Feign 调用必须加超时（你很可能没配全）

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 2000
        readTimeout: 3000
  httpclient:
    connection-timeout: 2000
```

✅ 不超时 = 必死

---

### ✅ 3️⃣ 熔断 / 降级（Hystrix / Sentinel）

```java
@SentinelResource(
    value = "getWechatConfig",
    fallback = "getWechatConfigFallback"
)
```

✅ 下游挂了，上游还能活

---

### ✅ 4️⃣ 发货接口不做强一致校验

| 校验              | 是否必须    |
| ----------------- | ----------- |
| clientId 是否存在 | ✅           |
| 订单是否已支付    | ✅           |
| 微信发货是否成功  | ❌（可重试） |

---

## ✅ 用一句永远不会错的话总结

> 微服务同步调用 = 一定会阻塞  
> 能不能撑住，取决于：超时 + 异步 + 熔断  
> 发货这种事，永远不要同步等微信

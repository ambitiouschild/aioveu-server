这个问题问得非常关键 👍

结论先给你：

> ✅ **消息只发送了一次**
>
> ✅ **但被消费（投递）了两次**
>
> ❌ **不是业务代码主动发了两遍**

下面用**严格区分“发送”和“消费”**的方式说明。

------

## ✅ 一句话结论

| 行为                   | 次数       | 说明                 |
| ---------------------- | ---------- | -------------------- |
| **Producer 发送消息**  | ✅ **1 次** | 支付成功后只发了一条 |
| **MQ Broker 投递消息** | ✅ **2 次** | broker 重投          |
| **Consumer 实际消费**  | ✅ **2 次** | 两次都被线程执行     |
| **订单业务处理**       | ✅ **1 次** | 第二次直接跳过       |

👉 **是「消费了两次」，不是「发送了两次」**

------

## 🔍 为什么能确定不是“发送了两次”？

### 1️⃣ message_id 完全一致

```
PAY_PAY1778317956207874129_PAY_20260529164134566_03_0000_153
```

- 

  同一个 messageId

- 

  MQ 层面 **不可能由 producer 生成两次**

- 

  这是 broker 重投的典型特征

------

### 2️⃣ 消费记录表里只有 1 条 insert

```
SELECT * FROM mq_consume_record
WHERE message_id = 'xxx'
```

结果：

```
id = 7
consume_status = 1
retry_count = 1
```

👉 说明：

- 

  **第一次消费：INSERT**

- 

  **第二次消费：UPDATE / ACK**

- 

  ✅ 没有第二条 INSERT

如果是 producer 发了两次：

- 

  一定会有 **两条消费记录**

------

### 3️⃣ 日志明确说是“同一条消息”

```
同一条消息发 2 次
第 2 次直接 ACK，不处理订单
```

这里的「发」是 **MQ 语义上的“投递”**，不是 Producer 发送。

------

## 🧠 正确理解 MQ 的“发”

| 说法                | 是否准确 |
| ------------------- | -------- |
| Producer 发了两次   | ❌ 不准确 |
| Broker 投递了两次   | ✅ 准确   |
| Consumer 消费了两次 | ✅ 准确   |
| 订单处理了两次      | ❌ 没有   |

------

## 🔁 真实发生的过程（时序）

```
① Producer → MQ：发送消息（1 次）
② MQ → Consumer：第一次投递
③ Consumer：处理订单 ✅
④ MQ 未收到 ACK / 超时 / rebalance
⑤ MQ → Consumer：第二次投递
⑥ Consumer：检测到已处理 ✅
⑦ Consumer：直接 ACK ✅
```

------

## ✅ 最终结论（重点）

> ✅ **不是发送了两次**
>
> ✅ **是 MQ 重投，导致消费了两次**
>
> ✅ **你们的幂等设计完全生效**

## ✅ 一句话结论（先给你）

> ✅ 订单创建时必须“选定一个 clientId”
>
> ✅ 选择方式 = 前端指定 OR 后端规则
>
> ❌ 不能靠“临时查 tenant 决定”

---

## 一、先明确：为什么必须“选”？

| 如果不选          | 会发生什么           |
| ----------------- | -------------------- |
| 发货时再查 tenant | ❌ 数据不一致         |
| 用默认 clientId   | ❌ 多小程序时错乱     |
| 用户投诉          | ❌ 微信支付单号不匹配 |

✅ 订单的微信身份，创建时就该确定

---

## 二、✅ 三种主流“选择方式”（你一定能对号入座）

| 方式                | 适合场景          | 推荐指数 |
| ------------------- | ----------------- | -------- |
| ✅ 前端指定 clientId | 多小程序 / 多品牌 | ⭐⭐⭐⭐⭐    |
| ✅ 后端按业务规则选  | 固定业务线        | ⭐⭐⭐⭐     |
| ❌ 下单时查默认      | 多 appid 会炸     | ⭐        |

---

## 三、✅ 方案一（强烈推荐）：前端指定 clientId

### 1️⃣ 前端请求

```json
{
  "productId": 123,
  "quantity": 1,
  "clientId": "mini_program_A"
}
```

### 2️⃣ 后端接收

```java
@PostMapping("/order/create")
public OrderVO createOrder(
        @RequestBody OrderCreateDTO dto
) {
    Long tenantId = TenantContext.getCurrentTenantId();

    OmsOrder order = new OmsOrder();
    order.setTenantId(tenantId);
    order.setClientId(dto.getClientId()); // ✅ 显式选择

    orderService.save(order);
}
```

✅ 最清晰
✅ 最安全
✅ 最易扩展

---

## 四、✅ 方案二：后端按“业务规则”选（推荐）

### 场景举例

| 业务 | 用哪个 client  |
| ---- | -------------- |
| 商城 | mall_client    |
| 预约 | booking_client |
| 外卖 | takeout_client |

### 后端实现

```java
String clientId = businessClientMapper
        .getClientIdByBusinessType(
                tenantId,
                dto.getBusinessType()
        );
```

✅ 前端无感知
✅ 业务驱动

---

## 五、✅ 数据库层面（必须支持）

### tenant_wx_app 表

```sql
CREATE TABLE tenant_wx_app (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT,
  client_id VARCHAR(64),
  appid VARCHAR(64),
  business_type VARCHAR(32),
  is_default TINYINT DEFAULT 0
);
```

✅ 一个租户多个 client
✅ 业务可区分

## 六、✅ 发货时永远只用 order.clientId

```java
weChatApiClient.uploadShippingInfo(
        order.getClientId(),
        body
);
```

✅ 不查 tenant
✅ 不猜
✅ 不翻车

## 七、✅ 架构师级总结（你现在的高度）

>

>

✅ 创建时选  
✅ 创建后不改  
✅ 发货只用订单里的

## ✅ 一句话结论

> 现在：没必要
>
> ✅ 一个 tenantId 对应一个 appid 是最优解
>
> ⚠️ 但未来 80% 的多租户 SaaS 都会走到「一个 tenant 多个 appid」

## 一、你现在的状态（✅ 非常健康）

```
tenantId
   ↓
clientId
   ↓
appid / secret
   ↓
微信发货 ✅
```

✅ 简单
✅ 稳定
✅ 不容易出错

👉 这是“最值得守住的阶段”

---

## 二、什么时候会「被迫」变成 1 tenant → N appid？

| 场景                | 说明               |
| ------------------- | ------------------ |
| ✅ 小程序 + H5       | 不同应用           |
| ✅ 多品牌            | 每个品牌独立小程序 |
| ✅ 国内 + 海外       | 不同微信主体       |
| ✅ 直连商户 + 服务商 | 不同支付主体       |
| ✅ 不同业务线        | 商城 / 预约 / 外卖 |

⚠️ 只要出现任意一个，你就必须支持多 appid

---

## 三、✅ 架构师视角：你现在该怎么做？

### ✅ 当前（强烈推荐）

```text
order
  └─ tenantId
       └─ clientId（唯一）
```

✅ 不要提前复杂化

---

### ✅ 未来（预留扩展点）

```text
order
  └─ clientId ✅（而不是 tenantId）
```

> 哪怕现在 tenant → 1 client
>
> order 里也存 clientId

## 四、✅ 给你一个“现在不后悔”的设计方案

| 层次      | 建议                           |
| --------- | ------------------------------ |
| 订单表    | ✅ 存 clientId（不是 tenantId） |
| 发货      | ✅ 直接用 order.clientId        |
| tenant 表 | ✅ 存「默认 clientId」          |
| 多 appid  | ✅ 未来加表即可                 |

## 五、✅ 你现在这套代码，最佳改法（最小成本）

### ✅ 下单时

```java
order.setClientId(
    tenantService.getDefaultClientId(tenantId)
);
```

### ✅ 发货时

```java
weChatApiClient.uploadShippingInfo(
    order.getClientId(),
    body
);
```

✅ tenantId 只是“帮你找默认 clientId”
✅ 发货永远只看 clientId

你说得很对！这是一个关键的设计问题。**如果你修改的是订单ID，但需要显示订单号，那么问题的核心就不是联查，而是数据结构设计的问题。**

## 问题分析

1. **后端联查能解决查询展示问题**，但新增/编辑时：
   - 前端需要选择订单（需要订单号和ID）
   - 后端保存只需要订单ID
   - 前端需要从订单号映射到订单ID
2. **根本矛盾**：前端界面上用户看到的是订单号，但数据库存储的是订单ID

## 解决方案

### 方案一：前后端都使用ID（推荐✨）

这是最清晰、最少出错的方案：

```
// 前端数据结构
{
  "orderId": 123,  // 直接使用ID
  "spuName": "商品A",
  "skuName": "规格A"
}

// 新增/编辑时
// 订单选择器返回：{id: 123, orderNo: "202310250001", customerName: "张三"}
// 保存时只传：{orderId: 123, ...}
```

**优势：**

- 前后端一致，都操作ID
- 避免映射错误
- 性能最优，无需转换

**缺点：** 用户界面看到的是ID而不是有意义的订单号

### 方案二：前端使用订单号，后端转换

```
// 前端传
{
  "orderNo": "202310250001",  // 传订单号
  "spuName": "商品A"
}

// 后端接收，查询订单表得到ID
String orderNo = dto.getOrderNo();
Long orderId = orderMapper.getOrderIdByOrderNo(orderNo);
// 保存到订单商品表
orderItem.setOrderId(orderId);
```

**劣势：**

- 多一次数据库查询
- 需要处理订单号不存在的情况
- 订单号可能重复（除非唯一约束）

### 方案三：混合方案（查询用订单号，存储用ID）

这是实际项目中最常用的方案：

#### 1. 查询时联查显示订单号

```
<select id="getOmsOrderItemPage" resultType="OmsOrderItemVO">
    SELECT
        oi.order_id,
        o.order_no,  <!-- 显示用 -->
        oi.spu_name
    FROM oms_order_item oi
    LEFT JOIN oms_order o ON oi.order_id = o.id
</select>
```

#### 2. 新增/编辑时

前端使用**选择器组件**，返回完整对象：

```
// 订单选择器组件
<OrderSelector 
  v-model="form.order"
  :show-fields="['orderNo', 'customerName']"
/>

// 选择后得到
form.order = {
  id: 123,           // 隐藏字段，用于保存
  orderNo: "202310250001",  // 显示字段
  customerName: "张三"
}

// 保存时
const saveData = {
  orderId: form.order.id,  // 只传ID
  spuName: form.spuName
  // ...
}
```

#### 3. 编辑回显时

```
// 先查询订单商品
OmsOrderItemVO item = orderItemMapper.getById(id);
// 再查询订单信息
OrderVO order = orderMapper.getById(item.getOrderId());
// 返回给前端
item.setOrderNo(order.getOrderNo());
item.setOrder(order);  // 整个订单对象，用于回显选择器
```

### 方案四：DTO分层处理

```
// 1. 前端提交DTO
public class OrderItemCreateDTO {
    private String orderNo;  // 前端传订单号
    // 其他字段...
}

// 2. 后端接收，转换成Entity
public OrderItem convertToEntity(OrderItemCreateDTO dto) {
    OrderItem item = new OrderItem();
    // 查询订单ID
    Long orderId = orderService.getOrderIdByOrderNo(dto.getOrderNo());
    item.setOrderId(orderId);
    return item;
}

// 3. 返回给前端的VO
public class OrderItemVO {
    private Long orderId;
    private String orderNo;  // 联查得到
    // 其他字段...
}
```

## 我的建议

**使用方案三（混合方案）**，因为：

1. **查询时**：联查显示订单号，用户体验好
2. **编辑时**：通过选择器组件，用户看到订单号，但实际保存的是ID
3. **数据一致性**：数据库始终存储ID，避免订单号变更带来的问题
4. **性能**：只有一次JOIN查询，没有N+1问题

## 具体实现步骤

1. **修改查询SQL**：JOIN联查订单号
2. **前端选择器**：实现订单选择组件，返回`{id, orderNo, ...}`
3. **保存逻辑**：前端只传`orderId`，后端直接保存
4. **回显逻辑**：后端查询时带上完整的订单信息

这样既解决了显示问题，又解决了编辑问题，性能也最优。

### 🔁 第二次消费是怎么来的？

关键在这一段：

```
catch (Exception e) {
    // ❌ 系统异常 → NACK 重试
    log.error("【Pay-Consumer】系统异常，等待重试, messageId={}", messageId, e);
    nackAndRequeue(channel, deliveryTag, messageId, orderNo);
}
```

#### 第二次消费一定是由 **NACK + requeue=true** 触发的：

```
channel.basicNack(deliveryTag, false, true); // 重试
```

也就是说：

> **某一次消费过程中：**
>
> - 
>
>   抛出了 `Exception`（非 BizException）
>
> - 
>
>   触发了 `basicNack(requeue = true)`
>
> - 
>
>   RabbitMQ 把消息重新入队
>
> - 
>
>   **再次投递 → 第二次消费**

✅ 这就是“消费两次”的唯一来源。

# ⚠️ 你现在这段代码里的 3 个隐患（重点）

## ❌ 隐患 1：日志误导 + 逻辑不一致

```
log.info("同一条消息发 2 次\n第 2 次直接 ACK，不处理订单");
```

但**代码并没有提前 return**。

👉 实际上：

- 

  第 2 次仍然进了 `handlePaymentSuccess()`

- 

  只是里面自己判断“订单已支付 → 跳过”

✅ 建议改成**真正的“直接 ACK”**：

```
if (mqConsumeRecordService.isConsumed(messageId)) {
    log.info("【Pay-Consumer】消息已消费，第 2 次直接 ACK, messageId={}", messageId);
    channel.basicAck(deliveryTag, false);
    return;
}
```

------

## ❌ 隐患 2：`basicNack + requeue=true`风险很高

你现在用的是：

```
channel.basicNack(deliveryTag, false, true);
```

⚠️ 问题：

- 

  如果 DB 抖动、锁等待、超时

- 

  **同一条消息会在短时间内反复消费**

- 

  日志会刷爆

✅ 更稳妥的做法：

| 场景              | 推荐                      |
| ----------------- | ------------------------- |
| 业务异常          | `basicReject(false)`→ DLQ |
| 幂等失败          | `basicAck()`              |
| 系统异常          | `basicReject(false)`→ DLQ |
| 禁止 requeue=true | ✅                         |

------

## ❌ 隐患 3：事务 + ACK 顺序耦合

```
@Transactional
public void onMessage(...) {
    handlePaymentSuccess();
    channel.basicAck();
}
```

⚠️ 风险：

- 

  事务提交前 ACK

- 

  或事务回滚但 ACK 已发出

✅ **生产级写法**应是：

```
handlePaymentSuccess(); // 事务方法
transactionTemplate.execute(status -> {
    // 确保 DB 已落库
});
channel.basicAck();
```

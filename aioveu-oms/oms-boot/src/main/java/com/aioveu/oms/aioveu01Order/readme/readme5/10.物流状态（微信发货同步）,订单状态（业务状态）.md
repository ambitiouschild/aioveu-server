## 二、✅ 标准职责划分（直接写进技术方案）

### 1️⃣ 物流状态（只给微信用）

```java
DeliverySyncStatusEnum
├── SYNCED(0)   // 已同步微信
├── PENDING(1)   // 待同步
└── UNKNOWN(3)   // 异常兜底
```

✅ 规则：

| 行为           | delivery_status |
| -------------- | --------------- |
| 下单           | `PENDING`       |
| 发货成功       | `SYNCED`        |
| 幂等跳过       | `SYNCED`        |
| ❌ 不表示已收货 | ——              |

---

### 2️⃣ 订单状态（你系统核心）

```java
OrderStatusEnum
├── UNPAID
├── PAID
├── SHIPPED
├── COMPLETED
└── CANCELLED
```

✅ “已收货” = `COMPLETED`

## 三、✅ 一个极其清晰的对照表（防坑）

| 场景         | delivery_status | order.status |
| ------------ | --------------- | ------------ |
| 下单         | `PENDING`       | `UNPAID`     |
| 支付成功     | 不变            | `PAID`       |
| 发货成功     | `SYNCED`        | `SHIPPED`    |
| 用户确认收货 | 不变            | `COMPLETED`  |
| 超时自动收货 | 不变            | `COMPLETED`  |

✅ 物流状态永远不关心“有没有人收货”

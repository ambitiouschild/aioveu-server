这是 **Git Commit 消息的规范化分类标签**，遵循类似 **Angular Commit Message Conventions** 或 **Conventional Commits** 的规范。每个 emoji + 标签 表示不同类型的代码变更。

### 各标签含义详解：

| 标签/Emoji     | 英文全称      | 中文含义     | 适用场景                                                     |
| -------------- | ------------- | ------------ | ------------------------------------------------------------ |
| **🍉 build**    | Build System  | **构建系统** | 影响构建系统或外部依赖的更改 • 更新 npm/yarn 包版本 • 修改 webpack/gulp 配置 • Dockerfile 或 CI/CD 脚本变更 |
| **🍏 feat**     | Feature       | **新功能**   | 新增功能（用户可见的） • 添加新 API 接口 • 新增页面/组件 • 实现新业务逻辑 |
| **🍎 fix**      | Fix           | **修复 Bug** | 修复代码中的错误 • 修复程序崩溃问题 • 修正逻辑错误 • 解决安全漏洞 |
| **🍑 docs**     | Documentation | **文档更新** | 仅文档相关变更 • 更新 README.md • 修改代码注释 • 添加 API 文档 |
| **🍇 refactor** | Refactor      | **代码重构** | 既不修复 bug 也不增加功能的代码更改 • 代码结构优化 • 重命名变量/函数 • 性能优化（不影响外部行为） |
| **🍌 style**    | Style         | **代码风格** | 不影响代码逻辑的样式/格式变更 • 调整代码缩进、空格 • 修正分号、引号格式 • 遵循 ESLint/Prettier 规则 |

### 其他常见标签：

| 标签      | 含义     | 示例                     |
| --------- | -------- | ------------------------ |
| **test**  | 测试相关 | 添加/修改单元测试        |
| **chore** | 日常维护 | 工具配置、生成文件等杂项 |

### 使用示例（标准的 commit message 格式）：

**实际例子：**

```
🍏 feat(用户模块): 添加用户登录功能
- 实现 JWT 令牌认证
- 添加登录页面 UI
- 完善表单验证逻辑

Closes #123
```

```
🍎 fix(订单API): 修复下单时库存扣减异常
- 解决并发下单导致的库存负数问题
- 添加数据库事务锁

Fixes #45
```

### 为什么要这样分类？

1. **自动生成 CHANGELOG**：工具可以根据标签自动归类发布日志
2. **语义化版本控制**：
   - `feat`→ 次版本号增加 (v1.2.0 → v1.3.0)
   - `fix`→ 修订号增加 (v1.2.0 → v1.2.1)
3. **代码审查**：快速识别提交目的
4. **项目维护**：便于追溯变更历史

### 在 Git 中的使用场景：

```
# 提交一个新功能
git commit -m "🍏 feat(购物车): 添加商品批量删除功能"

# 修复一个bug
git commit -m "🍎 fix(支付): 解决微信支付回调失败问题"

# 更新文档
git commit -m "🍑 docs: 更新 API 接口文档"
```

这些 emoji 是可选的，但标签分类是业界推荐的最佳实践。很多大厂（如 Vue、Angular、React 等开源项目）都在使用这种规范。

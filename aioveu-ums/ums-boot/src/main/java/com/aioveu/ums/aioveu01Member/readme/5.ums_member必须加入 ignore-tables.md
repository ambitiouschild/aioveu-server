## 一、先给你一个“定论版”结论

> ✅ `ums_member` 必须加入 `ignore-tables`
>
> ✅ 理由：登录 / 注册阶段是“系统态”，不能依赖自动租户过滤
>
> ✅ 你的配置本身就是为这种场景准备的
>
> ❌ 不要指望 MP 插件“聪明到能区分登录态”

## 二、为什么 `ums_member` 一定要 ignore？

### ✅ 你现在遇到的“铁三角问题”

1. 登录接口
   • 无 JWT

   • 无 `SecurityContext`

   • `TenantContext` 为空


2. Feign 调用 UMS
   • `ums_member` 查询

   • 参数里有 `tenantId`

   • 但插件不认参数


3. MP 租户插件
   • `getTenantId()` → `null`

   • 仍然拼接 `tenant_id = NULL`

   • 查询结果永远为空


👉 这不是 bug
👉 是“系统态查询”和“自动租户策略”的根本冲突

## 三、✅ 正确加进去的方式（你这个配置完美）

```yaml
tenant:

✅ 不再是：

```

## 四、⚠️ 一个“非常重要”的补丁（不然你会踩新坑）

### ❗ 你日志里有一句很关键的话：

> `sys_user #无租户id登录忽略，有租户id不忽略`

👉 `ums_member` 和 `sys_user` 是同一类表
👉 必须保持行为一致

### ✅ 推荐做法：在 `MyTenantLineHandler` 里加“二次确认”

```java
@Override
public boolean ignoreTable(String tableName) {
    // 1. 配置文件里的表直接忽略
    if (ignoreTables.contains(tableName)) {
        return true;
    }

    // 2. 登录期特殊逻辑（防止配置遗漏）
    ServletRequestAttributes attrs =
        (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
    if (attrs != null) {
        String uri = attrs.getRequest().getRequestURI();
        if (uri.contains("/openIdAndTenantId")
            || uri.contains("/register")) {
            if ("ums_member".equals(tableName)
                || "sys_user".equals(tableName)) {
                return true;
            }
        }
    }

    return false;
}
```

✅ 配置 + 代码双保险
✅ 防止以后有人把 `ums_member` 从 ignore 里删了
✅ 登录期行为可审计

## 五、那“有租户 ID 时不 ignore”怎么办？

你可能会担心：

> “以后我用 JWT 查 `ums_member`，是不是也会被 ignore？”

✅ 不会，只要你控制好“登录期”这个条件

### ✅ 用户态查询 `ums_member`（正常业务）

```text
/aioveu/api/v8/app/ums/members/profile
```

• ✅ 有 JWT

• ✅ 有 `tenantId`

• ✅ URI 不在登录期白名单

• ✅ `ignoreTable()` → `false`

• ✅ MP 插件自动拼 `tenant_id`


✅ 系统态 ≠ 用户态
✅ ignore 只在登录期生效

👉 不是“业务租户表”

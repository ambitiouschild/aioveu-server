# ✅ 核心原则（你这句话的正式版）

> OMS 可以主动调用 Pay（同步）  
> Pay 只能被动通知 OMS（异步）  
> Pay 绝不直接依赖 OMS

---

## 一、标准依赖 & 调用关系（✅ 最终形态）

```text
OMS
 ├─ 调用 Pay Feign（同步）
 │    ├─ 创建支付单
 │    ├─ 发起支付
 │    └─ 查询支付结果
 │
 └─ 监听 Pay 消息（异步）
      ├─ 支付成功
      ├─ 退款成功
      └─ 支付关闭
```

```text
Pay
 ├─ 不依赖 OMS
 ├─ 不发 Feign 给 OMS
 └─ 只发 MQ / Event
```

---

## 二、为什么 Pay 绝不能调用 OMS？

### ❌ 1️⃣ 防止循环依赖（致命）

如果 Pay → OMS，OMS → Pay  
👉 循环依赖成立，系统早晚炸

---

### ❌ 2️⃣ 支付系统必须“业务无关”

> Pay 不关心你卖的是机票、课程、还是咖啡

Pay 只知道：
• 订单号

• 金额

• 支付方式


✅ Pay 是基础设施
❌ 不是业务系统

---

### ❌ 3️⃣ 回调链路必须可重放

微信 / 支付宝回调：
• 会重试

• 会乱序

• 会重复


👉 同步调用 OMS = 回调不可重入 = 资损风险

---

## 三、✅ Pay → OMS 的唯一正确方式：事件 / MQ

### ✅ 正确示例

```java
// Pay 回调完成后
PaySuccessEvent event = new PaySuccessEvent();
event.setOrderSn(orderSn);
event.setPayAmount(amount);
event.setPayTime(LocalDateTime.now());

mqTemplate.send(PAY_SUCCESS_TOPIC, event);
```

```java
// OMS 消费
@RabbitListener(queues = PAY_SUCCESS_QUEUE)
public void handlePaySuccess(PaySuccessEvent event) {
    omsOrderService.onPaid(event.getOrderSn());
}
```

✅ 幂等  
✅ 可重试  
✅ 可追踪  
✅ 可降级  

---

## 四、OMS → Pay 的调用，也要“克制”

### ✅ OMS 只能调 Pay 的这几种接口

| 场景         | 是否允许  |
| ------------ | --------- |
| 创建支付单   | ✅         |
| 发起支付     | ✅         |
| 查询支付结果 | ✅（兜底） |
| 申请退款     | ✅         |
| 关闭支付单   | ✅         |

❌ OMS 不应：
• 修改支付单金额

• 修改支付状态

• 干涉支付实现细节

## 五、你这套架构的“三条铁律”（建议贴墙上）

> ✅ OMS 依赖 Pay，Pay 不依赖 OMS  
> ✅ Pay 回调只发消息，不调业务接口  
> ✅ 所有跨模块通信：Feign 只读，MQ 只写

## 六、唯一“可以破戒”的例外（⚠️ 谨慎）

### ✅ 例外：Pay 查 OMS 只读数据（不推荐，但可接受）

比如：
• 查订单是否存在

• 查用户 ID


✅ 前提：
• 只读

• 非事务

• 可降级

• 不阻塞支付核心链路


👉 更优方案：在创建支付单时就把这些数据带过来

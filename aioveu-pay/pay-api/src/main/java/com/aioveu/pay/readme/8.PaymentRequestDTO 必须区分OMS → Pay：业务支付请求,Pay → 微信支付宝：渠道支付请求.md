# ✅ 结论（先拍板）

> **PaymentRequestDTO 必须区分：**
>
> - 
>
>   ✅ **OMS → Pay：业务支付请求**
>
> - 
>
>   ✅ **Pay → 微信/支付宝：渠道支付请求**
>
> ❌ **不能共用同一个 DTO**

------

## 一、为什么一定要拆？（否则必踩坑）

### ❌ 共用 DTO 的后果

```
PaymentRequestDTO
├── orderSn        // OMS 用
├── userId        // OMS 用
├── openId        // 微信用
├── appId         // 微信用
├── spbillCreateIp // 微信用
├── notifyUrl     // 微信用
├── payer         // 微信用
└── amount        // 两边都有，但语义完全不同
```

🚨 问题来了：

| 问题         | 后果     |
| ------------ | -------- |
| 字段越来越多 | DTO 爆炸 |
| OMS 要改字段 | 影响微信 |
| 微信字段必填 | OMS 崩溃 |
| 校验规则不同 | 逻辑混乱 |
| 安全字段泄露 | 资损风险 |

👉 **这是“支付系统腐化的起点”**

------

## 二、✅ 正确的 DTO 拆分方式（标准答案）

### ✅ 1️⃣ OMS → Pay（业务语义）

```
public class UnifiedPayDTO {

    private String orderSn;

    private PaymentMethodEnum paymentMethod;
    private PaymentChannelEnum paymentChannel;
    private Long userId;

```

✅ 特点：

- 

  **业务视角的支付**

- 

  不关心微信 / 支付宝

- 

  不关心 OpenId / AppId

### ✅ 2️⃣ Pay → 微信（渠道语义）

✅ 特点：

  不关心 OMS

  可复用官方 SDK

## 三、✅ Pay 模块内部的“转换器”（关键）

```
@Component
public class WechatPayConverter {

    public WechatPayUnifiedOrderDTO toWechatDto(UnifiedPayDTO dto, String openId) {
        WechatPayUnifiedOrderDTO wx = new WechatPayUnifiedOrderDTO();
        wx.setOutTradeNo(dto.getOrderSn());
        wx.setTotalFeeFen(dto.getPaymentAmountFen());
        wx.setOpenId(openId);
        return wx;
    }
}
```

✅ **Pay 是唯一知道“怎么调微信”的模块**

# ✅ 多租户下的“唯一正确”方案

## ✅ 三个 DTO，各司其职（铁三角）

---

## ① **UnifiedPayDTO（OMS → Pay）**
> ✅ 业务意图，完全租户无关

```java
public class UnifiedPayDTO {

    private String orderSn;
    private Long tenantId;          // ✅ 只在这里出现
    private Long userId;
    private Long paymentAmountFen;
    private PaymentChannelEnum channel;
    private PaymentMethodEnum method;
}
```

特点
• 不关心微信 / 支付宝

• 不关心 AppId / MchId

• 只告诉 Pay：谁（tenant）要付多少钱


---

## ② **ChannelParamDTO（Pay 内部）**
> ✅ 租户 × 渠道配置（核心）

```java
public class WechatChannelParam {

    private String appId;
    private String appSecret;
    private String mchId;
    private String subMchId;
    private String apiV3Key;
    private String certPath;
    private String serialNo;
}
```

从哪里来？
• `tenant_id + channel` → DB / 配置中心

• Pay 启动时缓存

• 支付时按 `tenantId` 取


✅ 这是“多租户支付系统”的灵魂

---

## ③ **ChannelRequestDTO（Pay → 微信）**
> ✅ 100% 第三方协议

```java
public class WechatUnifiedOrderDTO {

    private String appId;
    private String mchId;
    private String subMchId;
    private String outTradeNo;
    private Long totalFeeFen;
    private String openId;
    private String notifyUrl;
}
```

✅ 不关心 tenant  
✅ 不关心 orderSn 是谁的  
✅ 只负责“调通微信”

---

## ✅ 多租户标准调用链（非常重要）

```text
OMS
 └── UnifiedPayDTO(tenantId, orderSn, amount)
        ↓
PAY
 ├── 根据 tenantId + channel
 │     ↓
 │   ChannelParamDTO(appId, mchId, key...)
 │
 ├── 组装 ChannelRequestDTO
 │
 └── 调微信
```

---

## 四、❌ 多租户下“绝对不能做的事”

### ❌ 把 appId / mchId 放进 UnifiedPayDTO

```java
// 🚨 灾难级设计
UnifiedPayDTO {
    String appId;
    String mchId;
}
```

后果：
• 前端能改商户号

• 租户 A 能付到租户 B 账户

• 资损 + 合规事故


---

### ❌ OMS 知道微信参数

OMS 甚至不应该知道“微信”两个字

---

## 五、✅ 多租户 PaymentRequestDTO 设计口诀（贴墙上）

> OMS 只说“谁要付”  
> Pay 决定“用哪个商户号”  
> 微信只认“参数对不对”  
> tenantId 永远不出 Pay

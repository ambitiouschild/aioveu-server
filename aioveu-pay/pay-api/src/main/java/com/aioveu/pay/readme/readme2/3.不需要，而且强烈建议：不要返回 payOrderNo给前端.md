✅ 结论非常明确（直接拍死）：

> 不需要，而且强烈建议：不要返回 `payOrderNo` 给前端。

这是支付中台级别的“安全 + 架构红线”，不是个人偏好问题。

我给你 4 个你这套系统必须这么做的硬理由，你就再也不会纠结了。

---

## 一、最核心原因：前端“根本不需要”

### 前端真正需要什么？

✅ 知道这是哪笔业务订单 → `orderSn`  
✅ 拿到调起微信 / 支付宝的参数 → `payParams`  
❌ 不需要知道 Pay 内部有几笔支付单  

👉 `payOrderNo` 是“系统内部技术键”，不是业务键

就像你不会把：
• 数据库主键 `id`

• 雪花算法 `id`

• 分布式锁 key`


返回给前端一样。

---

## 二、安全红线（你这套多租户 + 支付系统非常重要）

### ❌ 返回 `payOrderNo` 的风险

| 风险                 | 后果            |
| -------------------- | --------------- |
| 前端知道支付单号规则 | 可枚举          |
| 被爬取支付单号       | 刷单 / 撞库     |
| 被恶意构造请求       | 伪造支付 / 重放 |
| 内部实现细节泄露     | 攻击面扩大      |

✅ payOrderNo 只存在于：

```
Pay DB
Pay Service
微信 out_trade_no
```

👉 绝不出 Pay 边界

---

## 三、架构边界（你前面所有设计都在为这一步服务）

你之前已经定好的铁律：

| 层           | 订单号                        |
| ------------ | ----------------------------- |
| OMS → Pay    | `orderSn` ✅                   |
| Pay → 第三方 | `payOrderNo` ✅                |
| 第三方 → Pay | `out_trade_no = payOrderNo` ✅ |
| Pay → OMS    | `orderSn` ✅                   |
| Pay → 前端   | ❌ 不返回 payOrderNo           |

👉 现在这个 VO，就是“Pay → 前端”的边界

一旦返回，你前面所有分层设计全部被击穿。

## 五、✅ 你这个 VO 的「最终正确形态（给前端）」

```java
@Getter
@Setter
@Builder
@Schema(description = "支付参数对象（前端调起支付用）")
public class PaymentParamsVO implements Serializable {

    @Schema(description = "业务订单号", example = "ORD202401010001", requiredMode = REQUIRED)
    private String orderSn;

    @Schema(description = "支付渠道：WECHAT / ALIPAY")
    private String channel;

    @Schema(description = "支付类型：JSAPI / APP / H5 / NATIVE")
    private String payType;

    @Schema(description = "前端调起支付所需参数")
    private Map<String, Object> payParams;

    @Schema(description = "支付过期时间（毫秒）")
    private Long expireTime;
}
```

✅ 没有 `payOrderNo`
✅ 没有 `paymentNo`
✅ 没有 `out_trade_no`

## ✅ 一句话结论（先钉死）

> ❌ `WxResult` 不推荐

> ✅ 推荐：`WechatPayQueryResult` / `PaymentStatusDTO`
>
> ✅ 命名要体现：来源 + 行为 + 层级
>
> ❌ 不要用人名缩写、也不要用“泛化的 Result”

## 二、为什么 `WxResult` 不好？

### 1️⃣ `Wx` 是“黑话”

| 问题       | 说明                       |
| ---------- | -------------------------- |
| 新人看不懂 | 是微信？企业微信？小程序？ |
| 国际不友好 | WeChat 才是通用词          |
| 搜索困难   | `Wx` 不是完整单词          |

👉 代码是写给人看的

---

### 2️⃣ `Result` 太泛

`Result` 在 Java 里已经被污染了：

```java
Result<T>
CommonResult
ApiResult
RpcResult
```

👉 你这个是：
• 微信

• 支付

• 查询

• DTO


✅ 信息密度太低

---

### 3️⃣ 无法区分“行为”

你现在是：

```
微信 → 查询支付结果
```

以后一定会有：

```
微信 → 下单结果
微信 → 退款结果
微信 → 关单结果
```

如果都叫 `WxResult`：

```java
WxResult r1 = create();
WxResult r2 = query();
WxResult r3 = refund();
```

👉 灾难

---

## 三、推荐的命名（按优先级）

### ✅✅✅ 首选（最推荐）

```java
WechatPayQueryResult
```

✅ 优点：
• 一眼知道是 微信

• 一眼知道是 支付

• 一眼知道是 查询

• 和 `create` / `refund` 不冲突


---

### ✅✅ 次选（偏 DTO 风格）

```java
PaymentStatusDTO
```

✅ 优点：
• 更抽象

• 可复用（微信 / 支付宝 / 余额）

• 适合内部模型


⚠️ 缺点：
• 看不出是微信

• 需要包名辅助


---

### ✅ 可接受（但不优雅）

```java
WechatPaymentStatus
WechatPayStatusResult
```

---

### ❌ 不推荐

| 命名           | 原因          |
| -------------- | ------------- |
| `WxResult`     | 黑话          |
| `WXPayResult`  | 全大写 + 缩写 |
| `WeChatResult` | Result 太泛   |
| `WxQueryResp`  | Resp 混用     |
| `WxDTO`        | 无意义        |

## 四、一个非常重要的“命名原则”（支付系统专用）

> ✅ 类名 = 业务语义 + 技术角色
>
> ❌ 类名 ≠ 技术泛型

### ✅ 好名字的结构

```
[渠道][业务][行为][类型]
```

| 例子                   | 解读                      |
| ---------------------- | ------------------------- |
| `WechatPayQueryResult` | 微信 + 支付 + 查询 + 结果 |
| `AlipayRefundResponse` | 支付宝 + 退款 + 响应      |
| `PaymentStatusDTO`     | 支付 + 状态 + DTO         |
| `PayOrderEntity`       | 支付 + 订单 + 实体        |

## ✅ 一句话结论（先给你）

> ✅ **支付微服务负责“生成支付订单”**
>
> ✅ **MQ 消息只发“支付订单的关键标识”**
>
> ✅ **订单微服务消费时，通过 Feign / RPC 去支付微服务查一次 PayOrder**

**而不是：**

❌ 把 `PayOrder`实体直接塞进 MQ

❌ 订单服务直接连支付库

------

## ✅ 你现在的正确架构应该是这样

```
┌──────────────┐

       ▼

       ▼
```

## ✅ 一、支付微服务（Producer）怎么做？✅

### ✅ 1️⃣ 微信回调 → 保存 PayOrder

```
@Transactional
public void handleWechatNotify(WechatPayNotify notify) {

    PayOrder payOrder = buildPayOrder(notify);
    payOrderMapper.insert(payOrder);

    PaymentSuccessMessage msg = new PaymentSuccessMessage();
    msg.setMessageId(UUID.randomUUID().toString());
    msg.setPayOrderNo(payOrder.getPayOrderNo());
    msg.setOrderNo(payOrder.getOrderNo());
    msg.setTenantId(payOrder.getTenantId());
    msg.setPaidAmount(payOrder.getAmount());
    msg.setPaymentTime(OffsetDateTime.now());

    rocketMQTemplate.convertAndSend("payment_success_topic", msg);
}
```

✅ **不查**

✅ **不发整个 PayOrder**

✅ **只发“凭证”**

## ✅ 二、订单微服务（Consumer）怎么做？✅

### ✅ 1️⃣ 消费 MQ

```
@Transactional
public void handlePaymentSuccess(PaymentSuccessMessage message) {

    // 1️⃣ 幂等
    if (consumeRecordMapper.existsByMessageId(message.getMessageId())) {
        return;
    }

    // 2️⃣ 调用支付服务（Feign）
    PayOrderDTO payOrder = payClient.getByPayOrderNo(
            message.getPayOrderNo()
    );

    if (payOrder == null) {
        throw new RetryableException("支付订单不存在");
    }

    // 3️⃣ 更新订单
    Order order = orderMapper.selectByOrderNo(message.getOrderNo());
    order.setStatus(OrderStatusEnum.PAID);
    order.setPaymentTime(
            message.getPaymentTime()
                   .atZoneSameInstant(ZoneId.systemDefault())
                   .toLocalDateTime()
    );

    orderMapper.updateById(order);
}
```

## ✅ 三、支付微服务暴露 Feign 接口（✅ 必须）

```
@GetMapping("/api/v1/pay-order/{payOrderNo}")
public PayOrderDTO getByPayOrderNo(@PathVariable String payOrderNo) {
    PayOrder po = payOrderMapper.selectByPayOrderNo(payOrderNo);
    return PayOrderConvertor.toDTO(po);
}
```

## ✅ 四、为什么不能“订单服务直接读支付库”？

| 原因            | 说明               |
| --------------- | ------------------ |
| 服务边界被破坏  | 订单服务依赖支付表 |
| 表结构一改全炸  | 强耦合             |
| 多租户难做      | 数据隔离复杂       |
| 无法灰度 / 拆分 | 技术债             |

✅ **微服务 = 服务自治**

------

## ✅ 五、为什么不能发整个 PayOrder 到 MQ？

| 风险       | 后果               |
| ---------- | ------------------ |
| 实体变更   | 消费者反序列化失败 |
| 懒加载     | JSON 异常          |
| 数据不一致 | MQ 比 DB 旧        |
| 安全风险   | 敏感字段泄露       |

✅ **MQ 是契约，不是数据库镜像**

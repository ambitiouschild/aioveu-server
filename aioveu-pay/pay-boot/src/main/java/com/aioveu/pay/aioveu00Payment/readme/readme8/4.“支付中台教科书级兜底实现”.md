## 二、❌ 唯一一个“语义级错误”（必须改）

### 🚨 问题在这里

```java
if (updated) {
    paymentEventPublisher.publishPaymentSuccess(payOrder);

}
```

### ❌ 问题本质

> 你在这里“无条件发支付成功 MQ”

但此时：
• 微信返回的状态可能是 FAIL / CLOSED / REFUND

• 你却仍然：

  ```java
  paymentEventPublisher.publishPaymentSuccess(payOrder);
  ```

👉 MQ 名字叫 `PaymentSuccess`，但状态不一定是 SUCCESS

---

### ✅ 正确做法（关键）

只有微信确认“支付成功”时，才发 MQ + 触发业务

```java
if (updated) {
    if (PaymentStatusEnum.PAID == wx.getPaymentStatus()) {
        paymentEventPublisher.publishPaymentSuccess(payOrder);
        triggerBusinessProcess(paymentNo);
    } else {
        // 失败 / 关闭：只更新状态，不推进业务
        log.info("兜底查单确认非成功终态, paymentNo={}, status={}",
                paymentNo, wx.getPaymentStatus());
    }
}
```

✅ 这是“支付兜底”的铁律

## 三、⚠️ 一个“并发边缘风险”（强烈建议你加）

### 问题：`recover()` 可能被并发调用

• Job 正在扫

• 回调同时到达

• 人工补单接口触发


虽然你有：
```java
updateStatusIfNonTerminal(...)
```

但 MQ / Processor 仍然可能被并发触发两次：

• 线程 A：update 返回 true

• 线程 B：update 返回 false

• 但 A 和 B 都“查到微信成功”


### ✅ 解决方案 1（推荐）：乐观锁 + 状态双判

你已经有了 `version`，确保 `updateStatusIfNonTerminal` 使用了乐观锁：

```sql
UPDATE pay_order
SET payment_status = ?,
    third_transaction_no = ?,
    payment_time = ?,
    version = version + 1
WHERE id = ?
  AND payment_status NOT IN ('PAID', 'FAILED', 'CLOSED')
  AND version = ?
```

✅ 你现在的 `updateLocalStatus()` 必须长这样
✅ 如果不是，立刻改

---

### ✅ 解决方案 2（更保险）：业务幂等（可选）

在 `triggerBusinessProcess` 内部：

```java
if (alreadyProcessed(paymentNo)) {
    return;
}
```

👉 Processor 层兜底

### 步骤2：修改支付服务

#### 2.1 创建MQ生产者服务

```
// PaymentService中新增
@Service
@Slf4j
public class PaymentMQProducer {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    @Value("${rocketmq.topic.payment-success:payment_success_topic}")
    private String paymentSuccessTopic;
    
    /**
     * 发送支付成功消息
     */
    public boolean sendPaymentSuccess(PayOrder payOrder, Map<String, String> wechatParams) {
        try {
            // 构建消息
            PaymentSuccessMessage message = PaymentSuccessMessage.builder()
                .paymentNo(payOrder.getPaymentNo())
                .orderNo(payOrder.getOrderNo())
                .transactionId(wechatParams.get("transaction_id"))
                .amount(payOrder.getAmount())
                .channel(payOrder.getChannel())
                .memberId(payOrder.getMemberId())
                .paymentTime(LocalDateTime.now())
                .build();
            
            // 发送顺序消息
            SendResult result = rocketMQTemplate.syncSendOrderly(
                paymentSuccessTopic + ":wechat_pay",
                MessageBuilder.withPayload(message).build(),
                payOrder.getOrderNo(),  // 使用订单号保证顺序
                3000
            );
            
            return result.getSendStatus() == SendStatus.SEND_OK;
            
        } catch (Exception e) {
            log.error("发送支付成功消息失败: paymentNo={}", payOrder.getPaymentNo(), e);
            return false;
        }
    }
}
```

### 步骤3：修改订单服务

#### 3.1 创建MQ消费者

```
@Service
@Slf4j
@RocketMQMessageListener(
    topic = "${rocketmq.topic.payment-success:payment_success_topic}",
    consumerGroup = "${spring.application.name}-payment-consumer",
    selectorExpression = "wechat_pay || alipay"
)
public class OrderPaymentConsumer implements RocketMQListener<PaymentSuccessMessage> {
    
    @Autowired
    private OrderService orderService;
    
    @Override
    @Transactional(rollbackFor = Exception.class)
    public void onMessage(PaymentSuccessMessage message) {
        log.info("收到支付成功消息: orderNo={}", message.getOrderNo());
        
        try {
            // 调用现有订单服务
            orderService.handlePaymentSuccess(
                message.getOrderNo(),
                message.getTransactionId(),
                message.getAmount(),
                message.getPaymentTime()
            );
            
        } catch (Exception e) {
            log.error("处理支付消息异常: orderNo={}", message.getOrderNo(), e);
            throw e; // 抛出异常让RocketMQ重试
        }
    }
}
```

## 💡 关键考虑点

### 1. **不要新建服务的理由**

- ✅ 支付逻辑已经在支付服务，不应该拆分
- ✅ 订单逻辑已经在订单服务，不应该拆分
- ✅ 新增服务会增加运维复杂度
- ✅ 现有服务职责清晰，符合单一职责原则

### 2. **需要新增的代码量**

```
支付服务：
├── MQ生产者类：1个
├── 消息DTO：1-2个
├── 配置类：1个
└── 修改回调服务：约20行代码

订单服务：
├── MQ消费者类：1个
├── 配置类：1个
└── 修改订单服务：新增1个方法
```

### 3. **部署影响**

- 零停机部署：可以先部署消费者，再部署生产者
- 向后兼容：MQ发送失败可降级到同步调用
- 灰度发布：可以先用少量流量测试

### 4. **回滚方案**

如果MQ引入有问题，可以快速回滚：

```
// 在支付回调中增加开关
@Value("${payment.mq.enabled:true}")
private boolean mqEnabled;

public String handleWechatCallback(String xmlData) {
    // ... 处理逻辑
    
    if (mqEnabled) {
        // 发送MQ
        mqProducer.sendPaymentSuccess(...);
    } else {
        // 同步调用（原有逻辑）
        orderFeignClient.updateOrderStatus(...);
    }
}
```

## 🎯 总结建议

**不要新建微服务**，直接在现有服务中添加MQ能力：

1. **支付服务**：添加MQ生产者，在回调成功后发送消息
2. **订单服务**：添加MQ消费者，消费消息并更新订单状态
3. **逐步迁移**：可以先MQ和同步调用并存，稳定后切到MQ
4. **监控告警**：添加MQ的监控和告警

这样改动最小，风险最低，能快速上线验证。

## 一、一句话结论（先钉死）

> ✅ 查询支付状态：用「支付订单号（paymentNo）」
>
> ❌ 不要用「业务订单号（orderSn）」
>
> ✅ paymentNo 才是支付系统的主键
>
> ✅ orderSn 只是业务系统的关联键

👉 你现在代码里 `startPollingPaymentStatus(paymentData.paymentNo)` 是 ✅ 完全正确的

## 二、为什么是 `paymentNo`，而不是 `orderSn`？

### ✅ 1️⃣ 一张图看懂两个号的区别

| 字段        | 谁生成   | 作用                   | 是否唯一               |
| ----------- | -------- | ---------------------- | ---------------------- |
| `orderSn`   | 订单系统 | 业务订单号（用户可见） | ❌ 一个订单可能多次支付 |
| `paymentNo` | 支付系统 | 支付流水号（支付唯一） | ✅ 一次支付一个         |

### ✅ 2️⃣ 现实业务场景（非常关键）

#### ✅ 正常流程
```
订单创建 → orderSn=ORD20260101
         → 发起支付 → paymentNo=PAY20260101A
         → 支付成功
```

#### ✅ 取消再付 / 失败重试
```
订单创建 → orderSn=ORD20260101
         → 发起支付 → paymentNo=PAY20260101A ❌ 失败
         → 重新发起支付 → paymentNo=PAY20260101B ✅
```

👉 orderSn 不变
👉 paymentNo 变了

✅ 你查支付状态，必须查最新的 paymentNo

## 三、为什么用 `orderSn` 查会出大问题？

### ❌ 问题 1：查到旧的支付记录

```sql
SELECT * FROM pay_order WHERE order_sn = 'ORD20260101'
```

可能返回：
• 第一条：FAILED

• 第二条：PAID


👉 你查到 FAILED，但用户其实已经付过钱了

---

### ❌ 问题 2：第三方回调对不上

• 微信 / 支付宝回调：只认 `out_trade_no = paymentNo`

• 你用 `orderSn` 查

• 回调和查询不是同一条数据


---

### ❌ 问题 3：重复支付无法区分

• 用户付了两次

• 后台只能靠 `paymentNo` 区分

• `orderSn` 会糊成一团


---

## 四、✅ 标准支付系统的“三单关系”

```
订单系统（OMS）
 └── orderSn（业务单号）

支付系统（PMS）
 ├── paymentNo（支付流水号，主键）
 └── orderSn（外键，关联订单）

第三方支付（微信 / 支付宝）
 └── out_trade_no = paymentNo
```

### ✅ 查询规则（背下来）

| 场景             | 用哪个      |
| ---------------- | ----------- |
| 前端轮询支付状态 | ✅ paymentNo |
| 后端接收回调     | ✅ paymentNo |
| 对账             | ✅ paymentNo |
| 用户查看订单     | orderSn     |
| 退款             | paymentNo   |
| 关单             | paymentNo   |

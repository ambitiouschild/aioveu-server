## 一、为什么“数据库是数字，Entity 是 Enum”不矛盾？

### 因为中间有**转换器**

```text
DB(INT) → MyBatis → Entity(Enum) → Service → VO(INT)
```

### 对应关系

| 层      | 类型      | 例子                     |
| ------- | --------- | ------------------------ |
| 数据库  | `INT`     | `2`                      |
| MyBatis | 自动映射  | `PaymentStatusEnum.PAID` |
| Entity  | `Enum`    | `PAID`                   |
| VO      | `Integer` | `2`                      |
| 前端    | `number`  | `2`                      |

👉 数字只在“存储”和“传输”时出现  
👉 Enum 只在“Java 内存”里出现

## 三、那我为什么说“数据库用数字”？

我是说👇

> 数据库字段类型是数字，不是 varchar，不是 enum

✅ 正确：

```sql
payment_status INT COMMENT '2-支付成功'
```

❌ 错误：

```sql
payment_status VARCHAR(20) COMMENT 'PAID'
```

❌ 错误：

```sql
payment_status ENUM('PAID','UNPAID')
```

我说的是“字段类型”，不是“Entity 类型”

## 四、MyBatis-Plus 是怎么做到的？

你之前已经写对了 👇

```java
@TableField("payment_status")
private PaymentStatusEnum paymentStatus;
```

配合枚举：

```java
public enum PaymentStatusEnum {

    UNPAID(0, "待支付"),
    PAID(2, "支付成功");

    @EnumValue
    private final Integer code;
}
```

MyBatis 在干的事：

```text
DB 2 → PaymentStatusEnum.PAID
PaymentStatusEnum.PAID → DB 2
```

👉 你完全感知不到数字的存在

## 五、一个“血泪教训”级别的总结

> ❌ Entity 用 Integer = 给自己埋雷  
> ✅ Entity 用 Enum = 给未来省命

### 对比一下你就懂了

#### ❌ Integer 写法

```java
if (order.getPaymentStatus() == 2) {
    ...
}
```

半年后：
• 2 是啥？

• 有没有 20？

• 有没有 -1？

• 谁能改？


#### ✅ Enum 写法

```java
if (order.getPaymentStatus() == PaymentStatusEnum.PAID) {
    ...
}
```

• 编译期发现错误

• IDE 自动提示

• 枚举里集中管理

• 不怕别人乱传值

## 六、什么时候 Entity 才能用 Integer？

几乎只有这两种情况：

1. 遗留系统，改不动
2. 纯粹的血缘 DTO（没有业务行为）

除此之外：

> ✅ 只要是业务 Entity，状态 / 类型 / 渠道 / 方法 → 全部用 Enum

## ✅ 一句话结论（先给你）

> ✅ VO / Form / DTO 里，也必须用枚举  
> ❌ 不要为了“方便前端”而退回到 String / int

数据库、实体、VO、Form、DTO —— 状态字段必须一致。

---

## ✅ 二、为什么 VO / Form 也要用枚举？

### ❌ 错误做法（90% 项目都这么死）

```java
public class PayFlowVO {
    private String flowType;   // PAYMENT / REFUND
    private Integer flowStatus;// 1 / 2 / 3
}
```

后果：

| 问题       | 说明                              |
| ---------- | --------------------------------- |
| 前端乱传   | `"payment"` / `"PAY"`             |
| 后端判断乱 | `if ("PAYMENT".equals(flowType))` |
| 扩展灾难   | 加状态要改 N 个地方               |

---

## ✅ 三、正确做法（✅ 推荐）

### ✅ VO（给前端看）

```java
public class PayFlowVO {

    private String flowNo;

    private PayFlowTypeEnum flowType;
    private PayFlowStatusEnum flowStatus;

    private BigDecimal amount;
}
```

✅ 前端看到的是：

```json
{
  "flowType": "PAYMENT",
  "flowStatus": "SUCCESS"
}
```

✅ 不是数字
✅ 不是魔法字符串

---

### ✅ Form（前端提交）

```java
public class PayFlowQueryForm {

    private PayFlowTypeEnum flowType;
    private PayFlowStatusEnum flowStatus;

    private LocalDate startDate;
    private LocalDate endDate;
}
```

✅ 前端传：

```json
{
  "flowType": "REFUND",
  "flowStatus": "SUCCESS"
}
```

✅ Spring / Jackson 自动映射

---

## ✅ 四、那前端怎么办？（这是你最担心的）

### ✅ 前端从来不需要知道“数字”

#### 下拉框数据（字典接口）

```json
GET /dict/flowType

[
  { "code": "PAYMENT", "desc": "支付" },
  { "code": "REFUND", "desc": "退款" }
]
```

✅ 前端只认 code / desc
✅ 后端只认 enum

---

## ✅ 五、什么时候“允许”用 String / int？

✅ 只允许这一种情况：

| 场景       | 用法             |
| ---------- | ---------------- |
| 前端下拉框 | ✅ code（String） |
| 前端展示   | ✅ desc           |
| 后端所有层 | ✅ enum           |

❌ 不允许：

• Controller 里判断 `if (status == 1)`

• Form 里用 `Integer status`

• VO 里用 `String flowType`

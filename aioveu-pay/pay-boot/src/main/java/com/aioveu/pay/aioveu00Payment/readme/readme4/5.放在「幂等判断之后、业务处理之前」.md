## ✅ 一、它在整个回调流程里的“正确位置”

### ✅ 标准微信回调流程（✅ 你现在的系统应该用这个）

```
微信回调
   ↓
1️⃣ 解析参数
   ↓
2️⃣ 幂等判断（transaction_id）
   ↓
3️⃣ 超过最大回调次数？（✅ 就是这里）
   ↓
4️⃣ 已处理 → incrNotifyCount → 返回 SUCCESS
5️⃣ 未处理 → markConsumed → 执行业务
```

## ✅ 二、具体代码放在哪里（✅ 精确到行）

### ✅ 放在 `handleWechatCallback` 里（✅ 正确）

```java
// 1️⃣ 幂等判断（必须用 transaction_id）
if (payCallbackRecordService.isConsumed(transactionId)) {

    PayCallbackRecord record =
            payCallbackRecordService.getByTransactionId(transactionId);

    if (record != null && record.getNotifyCount() >= 10) {
        log.warn("【微信回调】超过最大回调次数，忽略通知, transactionId={}", transactionId);
        return generateWechatSuccessResponse();
    }

    payCallbackRecordService.incrNotifyCount(transactionId);
    return generateWechatSuccessResponse();
}
```

✅ 这是唯一正确位置

---

## ✅ 三、为什么不能放在别的地方？

### ❌ 1️⃣ 不能放在 `markConsumed` 里

```java
markConsumed(...) {
    if (notifyCount >= 10) { ... } // ❌ 错
}
```

原因：
• `markConsumed` 只负责 第一次落库

• 第一次 `notifyCount = 1`，永远不可能 ≥ 10


---

### ❌ 2️⃣ 不能放在 `incrNotifyCount` 里

```java
incrNotifyCount(...) {
    if (notifyCount >= 10) { ... } // ❌ 错
}
```

原因：
• 那是“更新方法”

• 不应该决定业务是否继续


---

### ❌ 3️⃣ 不能放在支付成功逻辑里

原因：
• 回调风暴发生在 支付之前

• 不能等业务处理完才拦


---

## ✅ 四、为什么要返回 SUCCESS 而不是 FAIL？

### ✅ 非常关键的一点（微信规则）

| 情况     | 返回      |
| -------- | --------- |
| 已处理   | ✅ SUCCESS |
| 超过次数 | ✅ SUCCESS |
| 业务失败 | ❌ FAIL    |

👉 否则微信会一直重试

## 一、为什么支付成功 / 失败一定要分开？

### 1️⃣ **消费方完全不一样（核心原因）**

| 消息类型 | 谁关心                     | 行为                               |
| -------- | -------------------------- | ---------------------------------- |
| 支付成功 | 订单、库存、会员、风控     | 发货 / 开通 / 记账                 |
| 支付失败 | 对账、清算、用户通知、风控 | 释放库存 / 提醒用户 / 记录失败原因 |

👉 成功是“业务推进”，失败是“异常处理”
👉 Topic / RoutingKey / 消费逻辑完全不同

---

### 2️⃣ **MQ 语义必须清晰**

如果你合并成一条：

```json
{
  "paymentNo": "...",
  "success": false,
  "errorCode": "BALANCE_NOT_ENOUGH"
}
```

那消费者第一句永远是：

```java
if (msg.isSuccess()) {
    ...
} else {
    ...
}
```

❌ 这是“RPC 思维”，不是“事件驱动”
❌ MQ 是“发生了什么事”，不是“状态字段”

---

### 3️⃣ **重试策略完全不同**

| 类型     | 是否重试           |
| -------- | ------------------ |
| 支付成功 | ✅ 可重试（幂等）   |
| 支付失败 | ❌ 多数失败不应重试 |

👉 失败消息往往需要：
• 死信队列

• 人工干预

• 风控分析


👉 成功消息需要：
• 高可靠

• 多次投递

• 最终一致


---

### 4️⃣ **监控 / 告警 / 大盘天然分开**

• `pay.payment.success.count`

• `pay.payment.failed.count`

• `pay.payment.failed.by_channel`

• `pay.payment.failed.by_reason`


👉 合在一起，监控几乎没法做

---

## 二、那你现在的代码，是不是“重复”了？

你现在的代码：

```java
sendPaymentSuccessMessage(...)
sendPaymentFailedMessage(...)
```

✅ 这是对的
✅ 这不是重复，是“职责分离”

但你现在的实现方式里，确实有可以“收敛”的地方 👇

### ✅ 第一层：对外接口分开（必须）

👉 接口层面坚决分开

## 四、RabbitMQ 层面的正确姿势

### ✅ Exchange 共用，RoutingKey 分开

```text
pay.payment.exchange
 ├── pay.payment.success  → 订单服务 / 会员服务
 └── pay.payment.failed   → 对账服务 / 风控服务
```

✅ 一个交换机
✅ 两个队列
✅ 互不干扰

## 五、一个非常高级的“认知升级”

> 支付成功消息 = “业务事件”
>
> 支付失败消息 = “系统事件 / 异常事件”

它们：
• 生命周期不同

• 重要性不同

• 处理方式不同

• 保留策略不同


👉 把它们当“一类消息”，本身就是设计失误

---

## 六、什么时候“可以”合并？（现实妥协）

只有在以下全部成立时，才可以合并：

• 内部系统

• 消费方只有 1 个

• 不需要精细化监控

• 失败不需要单独告警

• 未来不会拆分服务


👉 支付系统，几乎不可能满足这些条件

## ✅ 一句话结论（先给你答案）

> ✅ **`biz_type`、`topic`这类“消息语义字段”
>
> 👉 必须在 `PaymentSuccessMessage`构建时就确定**
>
> ❌ **不应该在 `RabbitSendRequest`里才补**

**理由只有一个：**

> **`PaymentSuccessMessage`是“业务语义”
>
> `RabbitSendRequest`只是“传输载体”**

## 二、为什么必须在 `PaymentSuccessMessage`构建？

### 1️⃣ 分层职责要清楚（非常重要）

| 对象                    | 职责                   |
| ----------------------- | ---------------------- |
| `PaymentSuccessMessage` | ✅ 表达“这是一件什么事” |
| `RabbitSendRequest`     | ✅ 表达“怎么发、发给谁” |
| `MqSendRecord`          | ✅ 表达“发的结果是什么” |

👉 **“这是什么业务消息” = 业务层决定**

------

### 2️⃣ 你现在的问题本质

你现在的痛点是：

- 

  `biz_type`不知道在哪给

- 

  `topic`不知道在哪给

- 

  结果到处补、补不全、补出 NPE

✅ **根因：业务语义被推迟到了技术层**

------

## 三、推荐的「标准做法」（强烈建议你照这个来）

------

### ✅ 1️⃣ `PaymentSuccessMessage`—— 业务语义（必须有）

```
@Data
@Builder
public class PaymentSuccessMessage {

    private String messageId;
    private String paymentNo;
    private String orderNo;
    private BigDecimal amount;
    private Long tenantId;

    /** ✅ 业务类型 */
    @Builder.Default
    private String bizType = "PAYMENT";

    /** ✅ 消息主题 */
    @Builder.Default
    private String topic = "payment.success";

    /** ✅ 消息类型 */
    @Builder.Default
    private String messageType = "PAYMENT_SUCCESS";
}
```

✅ **到这里，消息已经是“完整的业务事实”**

------

✅ **RabbitSendRequest 不做任何业务决策**

### ✅ 3️⃣ `saveMqSendRecord()`—— 只负责落库

## 四、如果你反过来会怎样？（反例）

❌ **在 RabbitSendRequest 才给 biz_type / topic**

```
RabbitSendRequest.builder()
    .bizType("PAYMENT") // ❌ 这里定业务语义
```

后果：

- 

  业务改一个类型 → 要改发送层

- 

  多个 Consumer → 语义混乱

- 

  新同事根本不知道“什么叫 PAYMENT”

👉 **这是架构腐化的开始**

## 六、最终结论（请你记住）

✅ **`PaymentSuccessMessage`：定义“发生了什么”**

✅ **`RabbitSendRequest`：定义“怎么发出去”**

✅ **默认值在业务对象里给，不在技术对象里补**

> **你这次所有 NPE、DB 报错、补偿失败，本质上都是：
>
> 业务语义被推迟 + 技术层被迫兜底**

这个问题问到了这套设计最精髓的地方 👍  
我一步一步把 `businessProcessorComposite.onPaid(paymentNo)` 是怎么“自动找到不同实现类” 给你拆开，看完你会发现：Spring 已经帮你把最难的事干了。

## 一、先给结论（记住这句）

> ✅ 不是 `Composite` 在“查场景”
>
> ✅ 是 Spring 在启动时，把所有 `BusinessProcessor` 实现类塞进了 `List<BusinessProcessor>`
>
> ✅ `Composite` 只是按顺序遍历，问每个 Processor：“你支持这个场景吗？”
>
> ✅ 谁说 `supports(scene) == true`，谁就干活

## 二、完整调用链（从你这行代码出发）

```java
businessProcessorComposite.onPaid(paymentNo);
```

### 1️⃣ `BusinessProcessorComposite.onPaid()`

你之前贴过的代码（我精简一下）：

```java
@Override
public void onPaid(String paymentNo) {

    PayOrder order = payOrderService.selectByPaymentNo(paymentNo);
    PaymentSceneEnum scene = order.getPaymentScene();

    for (BusinessProcessor processor : processors) {
        try {
            if (processor.supports(scene)) {
                processor.onPaid(paymentNo);
            }
        } catch (Exception e) {
            log.error("业务处理器执行失败, class={}, paymentNo={}",
                    processor.getClass().getSimpleName(), paymentNo, e);
        }
    }
}
```

👉 关键点：

• `processors` = Spring 注入的 `List<BusinessProcessor>`

• 不 if-else

• 不 switch

• 不问 paymentNo

• 只问：`supports(scene)`

## 三、`processors` 里到底有谁？（Spring 自动干的）

### ✅ Spring 启动时自动收集

```java
@Resource

```

Spring 会扫描到：

```text
orderBusinessProcessorImpl
membershipBusinessProcessorImpl
accountBusinessProcessorImpl
businessProcessorComposite（⚠️ 你自己也在里面）
```

👉 顺序默认不保证
👉 你可以用 `@Order` 控制

## 六、为什么 `Composite` 自己也在这个 List 里？（你现在的小坑）

你现在大概率是这样：

```java
@Component
public class BusinessProcessorComposite implements BusinessProcessor {
    @Resource
    private List<BusinessProcessor> processors;
}
```

⚠️ 这会导致：

• `processors` 里包含 `Composite` 自己

• 虽然你 `supports()` 返回 `true`

• 但不会无限递归（因为你没再调 composite）

• 不过设计上不优雅


### ✅ 推荐做法（更干净）

```java
@Component
public class BusinessProcessorComposite implements BusinessProcessor {

    private final List<BusinessProcessor> processors;

    public BusinessProcessorComposite(List<BusinessProcessor> processors) {
        // ✅ 把自己排除掉
        this.processors = processors.stream()
                .filter(p -> !(p instanceof BusinessProcessorComposite))
                .toList();
    }

    @Override
    public boolean supports(PaymentSceneEnum scene) {
        return true; // 我全都要
    }

    @Override
    public void onPaid(String paymentNo) {
        ...
    }
}
```

## 七、一句话把“怎么找到不同类”说死

> Spring 负责“找到所有人”
>
> `Composite` 负责“挨个问”
>
> `Processor` 负责“回答支不支持”
>
> `supports()` 是唯一决策点

## 九、一个非常高级的“进化形态”（可选）

如果你想更优雅：

```java
@Component
public class PaymentSceneDispatcher {

    private final Map<PaymentSceneEnum, BusinessProcessor> processorMap;

    public PaymentSceneDispatcher(List<BusinessProcessor> processors) {
        this.processorMap = processors.stream()
                .filter(p -> !(p instanceof BusinessProcessorComposite))
                .collect(Collectors.toMap(
                        BusinessProcessor::supports,
                        Function.identity()
                ));
    }

    public void dispatch(PaymentSceneEnum scene, String paymentNo) {
        BusinessProcessor processor = processorMap.get(scene);
        if (processor != null) {
            processor.onPaid(paymentNo);
        }
    }
}
```

👉 但你现在这套 `Composite` 已经完全够用，甚至更符合“失败隔离”

## 十、终极总结（你可以拿去讲给别人听）

> ✅ `businessProcessorComposite.onPaid(paymentNo)` 并不关心“是谁”
>
> ✅ 它只遍历 Spring 给它的所有 `BusinessProcessor`
>
> ✅ 每个 Processor 通过 `supports(PaymentSceneEnum)` 声明自己负责什么场景
>
> ✅ 支付场景 = 数据字段，不是 if-else
>
> ✅ 这是支付中台“可插拔业务处理”的标准写法

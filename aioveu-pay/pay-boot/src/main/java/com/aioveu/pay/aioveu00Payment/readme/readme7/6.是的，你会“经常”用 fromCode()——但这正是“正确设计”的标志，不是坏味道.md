## ✅ 一句话结论（先给你定心丸）

> 是的，你会“经常”用 `fromCode()`——但这正是“正确设计”的标志，不是坏味道。

如果你很少用 `fromCode()`，反而说明你的架构有问题。

---

## 一、为什么“经常用 `fromCode()`”是好事？

### ✅ 1️⃣ 说明你在**严格遵守边界**

```
HTTP / Feign  → Integer
Service       → Enum
DB            → Integer
MQ            → Integer
```

每一次 `fromCode()`，都是在说：

> “外部数据不可信，我现在把它变成可信的业务语义。”

✅ 这是安全边界
✅ 这是资损防火墙
✅ 这是系统成熟度的体现

---

### ✅ 2️⃣ `fromCode()` 是“防腐层”的核心

DDD 里有个词：Anti-Corruption Layer（ACL）

```java
PaymentChannelEnum channel =
    PaymentChannelEnum.fromCode(dto.getPaymentChannel());
```

这行代码：
• 挡住了非法值

• 挡住了魔法数字

• 挡住了未来枚举变更


👉 你每写一次 `fromCode()`，系统就更稳一点

---

### ✅ 3️⃣ 对比一下“不用 `fromCode()`”的世界

#### ❌ 世界 A（你现在拒绝的）

```java
if (dto.getPaymentChannel() == 1) {
    ...
}
```

• 没人知道 1 是什么

• 改枚举 = 全局搜数字

• 新人不理解

• 线上出事故


#### ✅ 世界 B（你现在在的）

```java
PaymentChannelEnum channel = PaymentChannelEnum.fromCode(code);
switch (channel) {
    case WECHAT:
    case ALIPAY:
}
```

✅ 编译器帮你
✅ IDE 帮你
✅ Code Review 一眼懂

---

## 二、那“经常写”怎么办？能不能少写点？

### ✅ 方案 1（✅ 最推荐）：**Converter / Assembler**

把“烦人的重复”收拢到一个地方。

```java
@Component
public class PaymentRequestAssembler {

    public PayOrder toEntity(PaymentRequestPayToTPPDTO dto) {
        return PayOrder.builder()
                .paymentNo(dto.getPayOrderNo())
                .orderNo(dto.getOrderSn())
                .bizType(dto.getBizType())
                .paymentChannel(dto.getPaymentChannel())
                .paymentMethod(dto.getPaymentMethod())
                .amount(dto.getPaymentAmount())
                .build();
    }

    public PaymentSuccessMessage toMessage(PayOrder order) {
        return PaymentSuccessMessage.builder()
                .paymentNo(order.getPaymentNo())
                .paymentChannel(order.getPaymentChannel())
                .paymentScene(order.getPaymentScene())
                .build();
    }
}
```

✅ Controller / Service 不再直接碰 DTO
✅ `fromCode()` 只在 Assembler 里出现
✅ 其他地方全是 Enum

---

### ✅ 方案 2（✅ 非常推荐）：**MapStruct（终极解）**

```java
@Mapper(componentModel = "spring")
public interface PaymentRequestMapper {

    PaymentRequestMapper INSTANCE = Mappers.getMapper(PaymentRequestMapper.class);

    @Mapping(
        target = "paymentChannelEnum",
        expression = "java(com.aioveu.common.enums.pay.PaymentChannelEnum.fromCode(dto.getPaymentChannel()))"
    )
    PayOrder toEntity(PaymentRequestPayToTPPDTO dto);
}
```

✅ 编译期生成代码
✅ 零手写 `fromCode()`
✅ 性能 = 手写

---

### ✅ 方案 3（❌ 不推荐但可用）：DTO 里加“只读视图”

```java
public PaymentChannelEnum getPaymentChannelEnum() {
    return PaymentChannelEnum.fromCode(this.paymentChannel);
}
```

⚠️ 缺点：
• DTO 带逻辑

• 容易被误用

• 不纯粹

## 三、一个“职业级”的心态转变（很重要）

> ❌ “怎么又要写 `fromCode()`，好烦”
>
> ✅ “这里需要 `fromCode()`，说明边界是对的”

支付系统里，`fromCode()` 出现的频率，和系统的稳定性是正相关的。

---

## 四、你这套系统里，`fromCode()` 的“正确分布”

| 位置                  | `fromCode()` 是否合理         |
| --------------------- | ----------------------------- |
| Controller            | ✅（少量，建议交给 Assembler） |
| Assembler / Converter | ✅✅✅（主战场）                 |
| Service               | ✅（业务逻辑用 Enum）          |
| StrategyFactory       | ✅（入口）                     |
| MQ Consumer           | ✅（唯一入口）                 |
| Entity                | ❌                             |
| Enum 内部             | ❌                             |

---

## 五、一句话帮你把这件事“想通”

> `fromCode()` 不是“重复代码”，而是“语义防火墙”。
>
> 你不是在写胶水代码，你是在定义“什么是合法的支付事实”。

---

## 六、最终建议（给你一个行动指南）

1️⃣ 接受现实：支付系统里，`fromCode()` 会很多  
2️⃣ 收拢位置：用 Assembler / MapStruct  
3️⃣ 禁止扩散：Entity / Domain 里不许出现 `fromCode()`  
4️⃣ 心态升级：每次写 `fromCode()`，心里默念“我又防了一个资损”

---

## 七、一句话总结（你可以记在团队 Wiki 里）

> 支付系统中，`fromCode()` 是外部数据进入领域模型的必经之路，其高频出现并非设计缺陷，而是强类型、防资损、可演化的必然结果；应通过 Assembler 或 MapStruct 将其集中管理，而非消除。

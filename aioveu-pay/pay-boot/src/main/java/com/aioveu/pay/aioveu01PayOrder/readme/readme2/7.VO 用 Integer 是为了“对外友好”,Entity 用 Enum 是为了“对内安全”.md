## 一、先给终极结论（死死记住）

> ✅ Entity（实体类）= `PaymentStatusEnum`
>
> ✅ VO / DTO（接口对象）= `Integer`
>
> ❌ Entity 里不要用 Integer
> ❌ VO 里不要用 Enum

这是支付系统级别的铁律。

## 二、一句话解释“为什么 Entity 必须用 Enum”

> Entity 是“业务真相”
>
> Enum 是“业务语义”
>
> Integer 只是“传输编码”

👉 业务真相，不能用“编码”来表达

## 四、你的代码里，谁是谁？（对号入座）

### ✅ Entity（数据库映射 / 业务核心）

```java
@TableName("pay_order")
public class PayOrder {

    /**
     * 支付状态
     */
    private PaymentStatusEnum paymentStatus;
}
```

✅ 这里必须用 Enum
✅ MyBatis 用 `@EnumValue`
✅ 业务逻辑天天用

```java
if (order.getPaymentStatus() == PaymentStatusEnum.PAID) {
    ...
}
```

---

### ✅ VO（接口返回 / HTTP 协议）

```java
public class PaymentStatusVO {

    /**
     * 支付状态（数字编码）
     */
    private Integer paymentStatus;
}
```

✅ 这里必须用 Integer
✅ Jackson 返回 `3`
✅ 前端直接用

```ts
if (res.paymentStatus === 3)
```

## 五、为什么 Entity 用 Integer 是“灾难”？

### ❌ 错误写法

```java
private Integer paymentStatus;
```

#### 后果 1：魔法数字满天飞

```java
if (order.getPaymentStatus() == 3) {
    ...
}
```

👉 3 是啥？
👉 新人直接懵

---

#### 后果 2：状态判断不安全

```java
if (order.getPaymentStatus() == 2 || order.getPaymentStatus() == 3) {
    ...
}
```

👉 少写一个等于号都发现不了

---

#### 后果 3：重构直接爆炸

```java
PAID(3 → 4)
```

👉 所有 `== 3` 全部变成 bug

## 六、为什么 VO 用 Enum 也是错的？

👉 前端炸

## 七、一个“黄金对照表”（你贴墙上）

| 层级       | 类型             | 原因      |
| ---------- | ---------------- | --------- |
| Database   | `int`            | 存储高效  |
| Entity     | `Enum`           | 业务语义  |
| Service    | `Enum`           | 逻辑判断  |
| Converter  | `Enum ↔ Integer` | 边界转换  |
| VO / DTO   | `Integer`        | 网络协议  |
| Controller | `VO`             | HTTP 返回 |
| 前端       | `number enum`    | UI 判断   |
| 日志       | `Enum.name()`    | 可读性    |

---

## 八、你现在这个报错，本质是“边界穿越”

```java
update.setPaymentStatus(wxResult.getPaymentStatus()); 
// Integer → Enum ❌
```

✅ 正确穿越方式

```java
update.setPaymentStatus(
    PaymentStatusEnum.fromCode(wxResult.getPaymentStatus())
);
```

👉 Enum 不能“裸接” Integer
👉 必须经过 `fromCode()`

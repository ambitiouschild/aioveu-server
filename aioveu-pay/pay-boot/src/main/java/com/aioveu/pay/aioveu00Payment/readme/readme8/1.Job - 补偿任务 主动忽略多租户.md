## 二、为什么 `tenant_id = NULL` 是灾难？

### ✅ SQL 语义复习

| 条件                | 结果         |
| ------------------- | ------------ |
| `tenant_id = 1`     | ✅ 正常       |
| `tenant_id IS NULL` | ✅ 能查到     |
| `tenant_id = NULL`  | ❌ 永远 FALSE |

因为：

```sql
NULL = NULL  -- 结果是 NULL，不是 true
```

👉 你的 Job / 补偿任务，一条数据都扫不到是正常的

### ✅✅ 方案 2：返回“忽略条件”而不是 `null`

修改你的 `TenantLineHandler`：

```java
@Override
public Expression getTenantId() {
    Long tenantId = TenantContext.getTenantId();

    if (tenantId == null) {
        // ✅ 返回 TRUE 表达式，等价于“1=1”
        return new BooleanLiteralIsTrue();
    }

    return new LongValue(tenantId);
}
```

⚠️ 风险：
• 所有 SQL 都会变成“无条件”

• 容易被误用

• 不推荐作为全局方案


👉 只适合你非常清楚自己在干什么

## 六、✅ 你这个 Job 的“正确姿势”（支付中台级）

### ✅ 支付兜底 Job = **必须无视租户**

因为：

• 支付单是全局事实

• 回调可能迟到

• MQ 可能丢失

• 必须全表扫描


👉 Job 的语义是“系统级修复”，不是“租户级业务”

### ✅ 正确写法

```java
@InterceptorIgnore(tenantLine = "true")
@Scheduled(cron = "0 */5 * * * ?")
public void reconcile() {
    log.info("支付兜底Job开始执行");

    // 全表扫描，不区分租户
    List<PayOrder> orders = payOrderMapper.selectPendingOrdersForQuery(...);

    orders.forEach(o -> recoveryService.recover(o.getPaymentNo()));
}
```

✅ `mq_send_record` 同理

```java
@InterceptorIgnore(tenantLine = "true")
List<MqSendRecord> selectRetryRecords(...);
```

## 七、一个非常重要的“多租户设计原则”

| 场景        | 是否忽略租户 |
| ----------- | ------------ |
| 普通 CRUD   | ❌ 不忽略     |
| 管理后台    | ❌ 不忽略     |
| 登录 / 注册 | ✅ 忽略       |
| 定时任务    | ✅ 忽略       |
| 补偿任务    | ✅ 忽略       |
| 对账任务    | ✅ 忽略       |
| 数据修复    | ✅ 忽略       |

👉 “系统级行为” ≠ “租户级行为”

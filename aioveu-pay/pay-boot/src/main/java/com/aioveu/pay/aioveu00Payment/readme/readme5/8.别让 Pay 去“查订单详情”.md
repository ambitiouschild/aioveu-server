## 一、你描述的调用链是哪一种？

你这句话：

> **“我从 oms 调用的 pay-api，再从 pay 调用 oms-api”**

等价于下面两种模型之一👇

------

### 🅐 模型一：直接循环调用（最危险 ❌）

```
OMS
 └── feign → pay-api
               └── feign → oms-api
                              └── （回到 OMS）
```

✅ **这是典型的微服务循环依赖**

🚨 风险包括：

| 风险           | 后果                   |
| -------------- | ---------------------- |
| 无限递归       | 栈溢出 / 线程打满      |
| Seata 死锁     | 全局事务无法提交       |
| 租户上下文污染 | 租户 ID 错乱           |
| 雪崩           | 一个慢接口拖死两个服务 |
| 排查地狱       | 你现在已经体验到了     |

------

### 🅑 模型二：通过 API 模块解耦（稍好一点，但仍 ⚠️）

```
oms-service
 └── 依赖 pay-api（Feign Client 接口模块）
       └── pay-service
             └── 依赖 oms-api
                   └── oms-service
```

✅ **代码层面解耦了**

❌ **运行时仍然是循环调用**

> **API 模块 ≠ 调用安全**
>
> 接口拆得再干净，**运行起来还是一个环**

------

## 二、你现在的异常，和这个有没有关系？

### ✅ 有直接关系（大概率）

你之前的现象：

- 

  OMS 报 `No static resource app-api/v1/orders`

- 

  Pay 调 OMS 404

- 

  Seata 回滚

- 

  租户上下文反复清理

这在**循环调用 + Feign + Seata + 租户过滤器**的组合下非常常见：

| 环节                | 问题                  |
| ------------------- | --------------------- |
| 第一次 OMS 调用 Pay | 正常                  |
| Pay 回调 OMS        | 路径写错 / 接口不存在 |
| OMS 再次进入过滤器  | 租户上下文混乱        |
| Seata 感知异常      | 全局回滚              |
| 异常被层层包装      | 你看到的 B0001        |

👉 **这不是巧合**

## 四、这个架构的正确姿势是什么？（重点）

### ✅ 原则一：**微服务之间禁止循环依赖（铁律）**

> ❌ OMS → Pay → OMS
>
> ❌ Pay → OMS → Pay

------

### ✅ 原则二：谁拥有数据，谁提供接口

| 数据 | 接口归属 |
| ---- | -------- |
| 订单 | OMS      |
| 支付 | Pay      |
| 用户 | User     |
| 商户 | Merchant |

**别让 Pay 去“查订单详情”**

**别让 OMS 去“算支付金额”**

## 五、你现在这个场景，正确拆分方式 ✅

### ❌ 错误设计（你现在可能是这样）

```
Pay 需要订单信息
→ 调 OMS 查订单
→ OMS 又依赖 Pay 计算结果
```

------

### ✅ 正确设计 ①：下沉公共能力（推荐）

#### 抽一个 **OrderCore / TradeCommon 模块**

```
trade-common
 ├── OrderDTO
 ├── OrderQueryService（非 Feign，纯 Java）
OMS 实现 OrderQueryService
Pay 只依赖 trade-common
```

✅ 不走 HTTP

✅ 不循环

✅ 不踩 Seata

------

### ✅ 正确设计 ②：事件驱动（异步解耦）

如果 Pay 只需要“订单状态”：

```
OMS 发事件 → MQ → Pay 消费
```

✅ 彻底断开反向调用

✅ 天然支持重试

✅ 非常适合支付场景

------

### ✅ 正确设计 ③：单向依赖（底线方案）

如果一定要 Feign：

```
OMS → Pay ✅
Pay → OMS ❌
```

Pay 需要的订单信息：

- 

  由 **OMS 在调用 Pay 时一次性传全**

- 

  或

- 

  由 **Pay 只依赖订单号，不直接查订单**

------

## ✅ 一句话结论（先拍板）

> **Form（入参）：用 `code`（Integer / String）**
>
> **VO（出参）：用 `code + desc`**
>
> ❌ **Form / VO 里都不应该直接暴露 Java Enum**

**Enum 是 Java 的私有语言，Form/VO 是对外契约。**

------

## 一、先分清楚：Form ≠ VO

| 对象                 | 本质           | 方向        |
| -------------------- | -------------- | ----------- |
| Form / DTO / Request | **输入契约**   | 前端 → 后端 |
| VO / Response        | **输出契约**   | 后端 → 前端 |
| Entity               | **持久化模型** | DB          |
| Domain               | **业务模型**   | Service     |

------

## 二、Form（入参）里为什么**不能用 Enum**？

### ❌ 错误写法

```
public class PaymentCreateForm {

    @NotNull
    private PaymentChannelEnum paymentChannel; // ❌
}
```

### 问题：

1. 

   **前端要传什么？**

   ```
   { "paymentChannel": "WECHAT" }
   ```

   - 

     拼写错了怎么办？

   - 

     大小写错了怎么办？

   - 

     前后端强耦合 Java 枚举名

2. 

   **版本演化**

   - 

     Enum 改名 → 接口不兼容

   - 

     删除 Enum → 前端直接炸

3. 

   **非 Java 客户端**

   - 

     H5 / 小程序 / App / 第三方

   - 

     他们根本没有 Java Enum

✅ **Form 的唯一职责：接收“数据”，不是“语义”**

------

### ✅ Form 正确写法

```
public class PaymentCreateForm {

    @NotNull(message = "支付渠道不能为空")
    private Integer paymentChannel; // ✅ code
}
```

------

## 三、VO（出参）里为什么**不建议直接暴露 Enum**？

### ❌ 不推荐

```
public class PaymentResultVO {

    private PaymentChannelEnum paymentChannel; // ❌
}
```

返回：

```
{
  "paymentChannel": "WECHAT"
}
```

### 问题：

- 

  前端只能用字符串比较

- 

  无法国际化

- 

  无法扩展

- 

  改 Enum 名 = 接口变更

------

### ✅ VO 标准写法（推荐）

```
public class PaymentResultVO {

    /** 支付渠道编码 */
    private Integer paymentChannelCode;

    /** 支付渠道描述 */
    private String paymentChannelDesc;
}
```

返回：

```
{
  "paymentChannelCode": 1,
  "paymentChannelDesc": "微信支付"
}
```

✅ 前端：

- 

  `switch(code)`

- 

  直接展示 `desc`

- 

  **零心智负担**

------

## 四、Form / VO 的“黄金规则”（重点）

### ✅ Form（入参）

| 字段        | 类型      |
| ----------- | --------- |
| 业务状态    | `Integer` |
| 业务类型    | `Integer` |
| 渠道 / 场景 | `Integer` |
| 枚举名      | ❌ 禁止    |
| Enum        | ❌ 禁止    |

------

### ✅ VO（出参）

| 字段          | 类型      |
| ------------- | --------- |
| 业务状态 code | `Integer` |
| 业务状态 desc | `String`  |
| 渠道 code     | `Integer` |
| 渠道 desc     | `String`  |
| Enum          | ❌ 禁止    |

------

## 五、那 Enum 到底在哪用？（再次强调）

```
Form
 ↓ code
Assembler / Converter
 ↓ fromCode()
Service / Domain ← ✅ Enum 主战场
 ↓ toCode()
VO
 ↓ code + desc
```

✅ **Enum 只在 JVM 内部活着**

✅ **Form / VO 只是“搬运工”**

## 六、你这套支付系统的“明确规范”

### ✅ Controller 层

```
@PostMapping("/pay")
public PaymentResultVO pay(@Valid PaymentCreateForm form) {

    PayOrder order = paymentAssembler.toEntity(form);

    paymentService.pay(order);

    return paymentAssembler.toVO(order);
}
```

### ✅ Assembler（关键）

```
@Component
public class PaymentAssembler {

    public PayOrder toEntity(PaymentCreateForm form) {
        return PayOrder.builder()
                .paymentChannel(form.getPaymentChannel())
                .paymentScene(form.getPaymentScene())
                .build();
    }

    public PaymentResultVO toVO(PayOrder order) {
        PaymentResultVO vo = new PaymentResultVO();
        vo.setPaymentChannelCode(order.getPaymentChannel());
        vo.setPaymentChannelDesc(
            PaymentChannelEnum.fromCode(order.getPaymentChannel()).getDesc()
        );
        return vo;
    }
}
```

## 七、一个“职业级”的接口设计口诀（背下来）

> **Form 收 code，VO 返 code + desc**
>
> **Enum 不进 HTTP，Enum 不出 HTTP**
>
> **前端要数字，用户要文案**
>
> **Java 管语义，JSON 管兼容**

> **它是 Form / Command，不是 VO**

✅ 所以你现在用 `Integer`**是完全正确的**

✅ 不需要改成 Enum

✅ 也不需要“后悔经常写 `fromCode()`”

## 九、一句话总结（你可以直接写在团队 Wiki）

> 系统对外 HTTP 接口中，Form（入参）仅接收业务编码（Integer / String），VO（出参）返回业务编码与业务描述；Java 枚举仅用于内部领域模型，严禁直接暴露在 Form / VO 中，以保障接口的跨端兼容性、版本稳定性与演进能力。

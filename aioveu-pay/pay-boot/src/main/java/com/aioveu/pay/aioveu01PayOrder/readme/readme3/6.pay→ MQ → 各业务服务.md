# ✅ 一句话结论（先记住）

> **BusinessProcessor 放在 `pay` 微服务里是对的，但它本身不写业务，也不直接 feign 调业务。  
> 正确姿势是：`pay` → MQ → 各业务服务。**

Feign 调用不是首选，MQ 才是支付中台的标准解法。

---

# 一、三种主流方案对比（先给结论）

| 方案                            | 是否推荐        | 原因               |
| ------------------------------- | --------------- | ------------------ |
| `pay` → Feign → 订单 / 会员     | ❌ 不推荐        | 强耦合、慢、易雪崩 |
| `pay` → MQ → 业务服务           | ✅✅✅ 强烈推荐    | 解耦、削峰、可重试 |
| `pay` → Domain Event → 本地处理 | ✅ 单体 / 小系统 | 演进到 MQ 的过渡   |

# 二、为什么**不直接 Feign 调用**（重点）

### ❌ 如果你这样写：

```java
@Async
public void onPaid(String paymentNo) {
    orderClient.deliver(paymentNo);
    memberClient.openVip(paymentNo);
    accountClient.increase(paymentNo);
}
```

### ❌ 会立刻引入这些问题：

| 问题         | 后果               |
| ------------ | ------------------ |
| 网络抖动     | 支付成功，发货失败 |
| 业务服务发布 | 回调阻塞           |
| 超时重试     | 重复发货           |
| 新增业务     | 改 pay 服务        |
| 雪崩         | 支付不可用         |

👉 支付是“核心生命线”，绝不能依赖业务服务的稳定性

# 三、企业级标准结构（强烈推荐）

```
┌──────────────┐
│   pay 服务   │
│──────────────│
│ 回调 / Job   │
│ 状态机更新   │ ✅ 强一致
│ Composite    │
│ Processor    │
│ 发 MQ 消息   │ ✅ 最终一致
└──────┬───────┘
       │ MQ
       ▼
┌──────────────┐
│ 订单服务     │ → 发货
├──────────────┤
│ 会员服务     │ → 开卡
├──────────────┤
│ 账户服务     │ → 记账
├──────────────┤
│ 营销服务     │ → 发券
└──────────────┘
```

✅ pay 服务只负责“钱”
✅ 业务服务只负责“事”
✅ MQ 负责“可靠通知”

# 四、BusinessProcessor 的正确职责（重新定义）

> BusinessProcessor = 支付成功后的“消息发射器”

### ✅ pay 服务里的 Processor

```java
@Component
@Slf4j
public class OrderBusinessProcessor implements BusinessProcessor {

    @Resource
    private RocketMQTemplate rocketMQTemplate;

    @Override
    public boolean supports(PaymentScene scene) {
        return scene == PaymentScene.ORDER;
    }

    @Async("bizExecutor")
    @Override
    public void onPaid(String paymentNo) {

        if (alreadySent(paymentNo)) {
            return;
        }

        PayMessage msg = new PayMessage(paymentNo, PaymentScene.ORDER);
        rocketMQTemplate.send("TOPIC_PAY_SUCCESS", msg);

        markSent(paymentNo);
    }
}
```

✅ pay 服务不关心谁消费
✅ 不关心消费是否成功
✅ 只保证消息发出

# 五、业务服务怎么消费（示例）

```java
@Service
@RocketMQMessageListener(
    topic = "TOPIC_PAY_SUCCESS",
    consumerGroup = "order-pay-consumer"
)
public class OrderPayConsumer implements RocketMQListener<PayMessage> {

    @Override
    public void onMessage(PayMessage msg) {

        if (!supports(msg.getScene())) {
            return;
        }

        deliverOrder(msg.getPaymentNo());
    }

    private boolean supports(PaymentScene scene) {
        return scene == PaymentScene.ORDER;
    }
}
```

✅ 业务服务自己决定是否关心这个支付
✅ 失败自动重试
✅ 天然幂等

# 六、那 Feign 什么时候能用？（有，但有限）

### ✅ 唯一推荐场景：**支付前 / 支付中**

| 场景         | 是否 Feign   |
| ------------ | ------------ |
| 创建支付单   | ✅ Feign      |
| 校验金额     | ✅ Feign      |
| 校验订单状态 | ✅ Feign      |
| 支付成功     | ❌ 不用 Feign |
| 退款成功     | ❌ 不用 Feign |

👉 支付前是“业务驱动支付”
👉 支付后是“支付通知业务”

# 七、如果你现在还不想上 MQ（过渡方案）

### ✅ 可接受方案：本地消息表

```text
pay_order
biz_event_send_log
```

1. 支付成功
2. 插入一条 `biz_event_send_log`
3. 定时任务扫描 → Feign 调用
4. 成功 → 标记 sent
5. 失败 → 重试

✅ 这是阿里早期架构
✅ 后期无缝迁 MQ

# 八、你现在这套架构的“最终形态总结”

| 模块              | 职责                 |
| ----------------- | -------------------- |
| pay 服务          | 管钱、管状态、发消息 |
| BusinessProcessor | 支付 → 消息的转换    |
| MQ                | 解耦、削峰、重试     |
| 业务服务          | 处理自己的业务       |
| Feign             | 只用于支付前         |

> pay 服务不知道订单、会员、账户的存在

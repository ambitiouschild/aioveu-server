# ✅ 一句话结论（先记住）

> ❌ 不是“都用数字”
>
> ✅ 而是：数据库用数字，Java 用枚举，中间自己转换
>
> ✅ 这是支付 / 账务 / 电商系统的行业标准

---

# 一、为什么“只用数字”是灾难？

```java
if (status == 2) {
    // 2 是啥？成功？失败？已发货？
}
```

❌ 问题：
• 可读性为零

• 新人不敢改

• 枚举值一变，逻辑全炸

• 对账 / BI / 运维全是魔法数字


👉 支付系统是“资损高发区”，魔法数字是事故源头

---

# 二、为什么“数据库存枚举”也不对？

### ❌ MySQL 存 ENUM

```sql
status ENUM('PENDING','SUCCESS','FAILED')
```

问题：
• 改枚举要 ALTER TABLE

• 跨语言不友好

• 数据分析工具支持差

• 历史数据难迁移


👉 数据库是“存储系统”，不是“业务语义系统”

---

# 三、✅ 行业标准方案（你现在的系统就应该这样）

## ✅ 三层模型（非常重要）

| 层级      | 用什么               | 原因               |
| --------- | -------------------- | ------------------ |
| 数据库    | `INT`                | 稳定、高效、跨语言 |
| Java 内部 | `Enum`               | 类型安全、可读性强 |
| 对外接口  | `Integer` / `String` | 兼容前端 / 第三方  |

---

## ✅ 数据库（只认数字）

```sql
CREATE TABLE mq_send_record (
    send_status TINYINT NOT NULL COMMENT '0-未发送 1-已发送 2-失败'
);
```

---

## ✅ Java 实体（Integer）

```java
@Data
public class MqSendRecord {

    /** 发送状态 */
    private Integer sendStatus;
}
```

---

## ✅ Java 业务代码（Enum）

```java
SendStatusEnum status = SendStatusEnum.fromValue(record.getSendStatus());

if (status == SendStatusEnum.SUCCESS) {
    // 清晰、安全
}
```

---

## ✅ 对外接口（转成数字）

```java
public Integer getSendStatus() {
    return this.sendStatus;
}
```

---

# 四、为什么支付系统**强烈推荐**这套？

### ✅ 1️⃣ 对账友好

财务系统、BI、报表、脚本：

```sql
SELECT send_status, COUNT(*)
FROM mq_send_record;
```

✅ 全是数字，无需 JOIN

---

### ✅ 2️⃣ 演化安全

• 加状态：只改 Java Enum

• 不改表结构

• 不迁数据


---

### ✅ 3️⃣ 跨语言 / 跨团队

• PHP / Go / Python / 大数据

• 都认识 `int`

• 不一定认识 Java Enum


---

### ✅ 4️⃣ 故障排查

日志里：

```text
sendStatus=2
```

✅ 任何人都能查字典
✅ 不需要反序列化 Java 对象

---

# 五、那什么时候“直接用 Enum”？

### ✅ 可以用 Enum 的场景

| 场景         | 是否推荐           |
| ------------ | ------------------ |
| 内存状态机   | ✅                  |
| Switch 判断  | ✅                  |
| 业务规则引擎 | ✅                  |
| 数据库字段   | ❌                  |
| RPC 返回值   | ⚠️（需谨慎）        |
| MQ 消息体    | ⚠️（推荐转成 code） |

> 数据库不管业务含义
>

# 八、你现在的枚举，应该怎么“收敛”？

RabbitMQ 真实只需要 3 个核心状态：

```java
PENDING(0, "未发送")
SENT(1, "已发送")
FAILED(2, "发送失败")
```

其余状态：
• ✅ `TIMEOUT` → 业务推导

• ✅ `DEAD` → DLQ 独立字段

• ✅ `CONFIRM_NACK` → `FAILED`


👉 Enum 越少，系统越稳

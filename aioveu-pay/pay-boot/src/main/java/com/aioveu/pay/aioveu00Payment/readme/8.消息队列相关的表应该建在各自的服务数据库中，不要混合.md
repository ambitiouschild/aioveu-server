消息队列相关的表应该**建在各自的服务数据库中**，不要混合。

## 📊 建议的表分布

### 方案1：分库分表（推荐）

```
支付服务数据库 (payment_db)         订单服务数据库 (order_db)
┌──────────────────────┐          ┌──────────────────────┐
│ pay_order            │          │ oms_order            │
│ mq_send_record       │◀──MQ───▶│ mq_consume_record    │
│ mq_compensation      │          │ mq_dead_letter       │
└──────────────────────┘          └──────────────────────┘
```

### 方案2：统一消息服务数据库

```
消息服务数据库 (mq_db) - 如果你有统一的消息服务
┌──────────────────────┐
│ mq_send_record       │
│ mq_consume_record    │
│ mq_topic_config      │
│ mq_monitor_log       │
└──────────────────────┘
```

## 🔧 具体建表方案

### 1. **支付服务数据库** - 建发送相关表

```
-- 支付服务数据库 payment_db
-- 1. 消息发送记录表
CREATE TABLE `mq_send_record` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `message_id` varchar(64) NOT NULL COMMENT '消息ID',
  `biz_id` varchar(64) NOT NULL COMMENT '业务ID(支付单号)',
  `biz_type` varchar(50) NOT NULL COMMENT '业务类型:payment_success',
  `topic` varchar(100) NOT NULL COMMENT 'Topic',
  `tag` varchar(50) DEFAULT NULL COMMENT 'Tag',
  `sharding_key` varchar(100) DEFAULT NULL COMMENT '分片Key',
  `message_body` json NOT NULL COMMENT '消息体(JSON格式)',
  `send_status` tinyint(4) NOT NULL DEFAULT '0' COMMENT '发送状态:0-未发送,1-发送中,2-发送成功,3-发送失败',
  `retry_count` int(11) DEFAULT '0' COMMENT '重试次数',
  `max_retry` int(11) DEFAULT '5' COMMENT '最大重试次数',
  `next_retry_time` datetime DEFAULT NULL COMMENT '下次重试时间',
  `send_time` datetime DEFAULT NULL COMMENT '发送时间',
  `confirm_time` datetime DEFAULT NULL COMMENT '确认时间',
  `error_msg` varchar(1000) DEFAULT NULL COMMENT '错误信息',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_message_id` (`message_id`),
  KEY `idx_biz_id` (`biz_id`),
  KEY `idx_topic_status` (`topic`,`send_status`),
  KEY `idx_retry_time` (`next_retry_time`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='MQ消息发送记录表';

-- 2. 补偿任务表
CREATE TABLE `mq_compensation_task` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `task_type` varchar(50) NOT NULL COMMENT '任务类型:send_retry',
  `biz_id` varchar(64) NOT NULL COMMENT '业务ID',
  `biz_data` json COMMENT '业务数据',
  `status` tinyint(4) NOT NULL DEFAULT '0' COMMENT '状态:0-待处理,1-处理中,2-成功,3-失败',
  `retry_count` int(11) DEFAULT '0' COMMENT '重试次数',
  `next_execute_time` datetime DEFAULT NULL COMMENT '下次执行时间',
  `execute_result` varchar(1000) DEFAULT NULL COMMENT '执行结果',
  `error_msg` text COMMENT '错误信息',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_biz_id` (`biz_id`),
  KEY `idx_status_time` (`status`,`next_execute_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='MQ补偿任务表';

-- 3. 在pay_order表添加MQ相关字段
ALTER TABLE `pay_order` 
ADD COLUMN `mq_status` tinyint DEFAULT 0 COMMENT 'MQ状态:0-未发送,1-发送成功,2-发送失败',
ADD COLUMN `mq_message_id` varchar(64) COMMENT 'MQ消息ID',
ADD COLUMN `mq_send_time` datetime COMMENT 'MQ发送时间',
ADD INDEX `idx_mq_status` (`mq_status`);
```

### 2. **订单服务数据库** - 建消费相关表

```
-- 订单服务数据库 order_db
-- 1. 消息消费记录表
CREATE TABLE `mq_consume_record` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `message_id` varchar(64) NOT NULL COMMENT '消息ID',
  `topic` varchar(100) NOT NULL COMMENT 'Topic',
  `tag` varchar(50) DEFAULT NULL COMMENT 'Tag',
  `consumer_group` varchar(100) NOT NULL COMMENT '消费者组',
  `biz_id` varchar(64) NOT NULL COMMENT '业务ID(订单号)',
  `consume_status` tinyint(4) NOT NULL DEFAULT '0' COMMENT '消费状态:0-未消费,1-消费中,2-消费成功,3-消费失败,4-进入死信',
  `retry_count` int(11) DEFAULT '0' COMMENT '重试次数',
  `max_retry` int(11) DEFAULT '3' COMMENT '最大重试次数',
  `next_retry_time` datetime DEFAULT NULL COMMENT '下次重试时间',
  `consume_time` datetime DEFAULT NULL COMMENT '消费时间',
  `finish_time` datetime DEFAULT NULL COMMENT '完成时间',
  `error_msg` text COMMENT '错误信息',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_message_consumer` (`message_id`,`consumer_group`),
  KEY `idx_biz_id` (`biz_id`),
  KEY `idx_consumer_status` (`consumer_group`,`consume_status`),
  KEY `idx_retry_time` (`next_retry_time`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='MQ消息消费记录表';

-- 2. 死信队列表
CREATE TABLE `mq_dead_letter` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `message_id` varchar(64) NOT NULL COMMENT '消息ID',

  `biz_id` varchar(64) NOT NULL COMMENT '业务ID',
  `message_body` json NOT NULL COMMENT '消息体',
  `consume_times` int(11) DEFAULT '0' COMMENT '消费次数',
  `error_msg` text COMMENT '错误信息',
  `dead_reason` varchar(500) COMMENT '死信原因',
  `handle_status` tinyint(4) DEFAULT '0' COMMENT '处理状态:0-未处理,1-已处理',
  `handle_time` datetime DEFAULT NULL COMMENT '处理时间',
  `handle_result` varchar(1000) DEFAULT NULL COMMENT '处理结果',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_message_consumer` (`message_id`,`consumer_group`),
  KEY `idx_biz_id` (`biz_id`),
  KEY `idx_handle_status` (`handle_status`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='MQ死信队列表';

-- 3. 消费幂等性表
CREATE TABLE `mq_consume_idempotent` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `biz_unique_key` varchar(128) NOT NULL COMMENT '业务唯一键',
  `biz_type` varchar(50) NOT NULL COMMENT '业务类型',
  `message_id` varchar(64) NOT NULL COMMENT '消息ID',

  `consume_time` datetime NOT NULL COMMENT '消费时间',

  KEY `idx_message_id` (`message_id`),
```

## 🔧 服务配置

```
# payment-service application.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/payment_db
    username: root
    password: 123456
  
mq:
  record:
    send-table-enabled: true    # 是否记录发送表
    consume-table-enabled: false # 支付服务不需要消费表
```

### 订单服务配置

```
# order-service application.yml
spring:
  datasource:

    username: root
    password: 123456
  
mq:
  record:
    send-table-enabled: false   # 订单服务不需要发送表
    consume-table-enabled: true  # 是否记录消费表
```

## 💡 为什么这样设计？

### 优点：

1. **数据隔离**：各服务只关心自己的数据
2. **性能优化**：各自数据库压力分散
3. **独立维护**：表结构可以独立变更
4. **容灾**：一个服务数据库挂了不影响另一个

### 查询场景：

```
-- 支付服务：查看MQ发送情况
SELECT * FROM mq_send_record 
WHERE biz_id = 'PAY123' AND topic = 'payment_success_topic';

-- 订单服务：查看消息消费情况
SELECT * FROM mq_consume_record 
WHERE biz_id = 'ORDER456' AND consumer_group = 'order-service';

-- 运营：需要跨服务查询时
-- 方案1：通过业务ID关联查询
-- 方案2：导出到数据仓库
-- 方案3：通过日志系统
```

## 🎯 总结建议

**建表策略**：

1. **发送相关表** → 建在**支付服务**数据库
2. **消费相关表** → 建在**订单服务**数据库
3. **各自独立** → 不要混在一个数据库

**如果后续有多个消费者**：

```
订单服务数据库：记录订单服务的消费情况
积分服务数据库：记录积分服务的消费情况
通知服务数据库：记录通知服务的消费情况
```

这样设计既符合微服务的数据自治原则，又能满足业务需求。

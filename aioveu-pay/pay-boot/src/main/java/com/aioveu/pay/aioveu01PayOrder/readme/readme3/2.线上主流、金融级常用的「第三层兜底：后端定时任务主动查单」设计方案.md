下面给你一套**线上主流、金融级常用**的「第三层兜底：后端定时任务主动查单」设计方案。

特点：**安全、幂等、不踩微信风控、可水平扩展**。

------

## 一、设计目标（先对齐认知）

这个 Job **只干一件事**：

> **在“微信回调可能丢了”的情况下，把悬空订单捞回来。**

它不负责：

- 

  用户展示

- 

  实时响应

- 

  复杂业务编排

只负责：**“非终态 → 终态”的最终一致性**。

------

## 二、整体设计原则（主流共识）

| 原则         | 说明                       |
| ------------ | -------------------------- |
| ✅ 只查非终态 | 已 SUCCESS / CLOSED 不再查 |
| ✅ 查单有间隔 | 防止疯狂打微信             |
| ✅ 游标分页   | 不用 offset，避免深分页    |
| ✅ 分批处理   | 防止一次性扫全表           |
| ✅ 状态机约束 | 防止回调 & Job 互相覆盖    |
| ✅ 可重入     | 宕机 / 重启无副作用        |
| ✅ 限流       | 控制微信 QPS               |

------

## 三、扫描条件（非常关键）

### ✅ 要查的订单必须满足：

```
payment_status IN (PENDING, PROCESSING)
AND created_at >= now - 30分钟   -- 太老的走对账
AND (
    last_query_time IS NULL
    OR last_query_time <= now - 5分钟
)
```

### ❌ 不查的订单：

- 

  已支付 / 已关闭

- 

  刚创建 < 30 秒（等回调）

- 

  最近 5 分钟内已查过

- 

  超过 30 分钟的（交给 T+1 对账）

### 2️⃣ 批量拉取待查订单（游标分页）

```
int batchSize = 200;
LocalDateTime maxCreatedAt = LocalDateTime.now().minusMinutes(30);
LocalDateTime lastIdCreatedAt = LocalDateTime.now();
Long lastId = Long.MAX_VALUE;

while (true) {
    List<PayOrder> orders = payOrderMapper.selectPendingOrdersForQuery(
            maxCreatedAt,
            lastIdCreatedAt,
            lastId,
            batchSize
    );

    if (CollectionUtils.isEmpty(orders)) {
        break;
    }

    for (PayOrder order : orders) {
        processOrder(order);
    }

    PayOrder last = orders.get(orders.size() - 1);
    lastId = last.getId();
    lastIdCreatedAt = last.getCreatedAt();
}
```

#### Mapper 示例

```
SELECT *
FROM pay_order
WHERE payment_status IN ('PENDING', 'PROCESSING')
  AND created_at >= #{maxCreatedAt}
  AND created_at <= #{lastIdCreatedAt}
  AND id < #{lastId}
ORDER BY created_at DESC, id DESC
LIMIT #{limit}
```

✅ **不用 offset，性能稳定**

### 3️⃣ 单笔订单处理（核心逻辑）

```
@Transactional(rollbackFor = Exception.class)
public void processOrder(PayOrder order) {

    // 再次校验，防止并发
    PayOrder fresh = payOrderMapper.selectById(order.getId());
    if (fresh == null
        || fresh.getPaymentStatus().isTerminal()) {
        return;
    }

    // 5 分钟内刚查过，跳过
    if (fresh.getLastQueryTime() != null
        && fresh.getLastQueryTime().isAfter(LocalDateTime.now().minusMinutes(5))) {
        return;
    }

    try {
        PaymentStatusVO wxResult =
                wechatPayService.queryPayment(fresh.getPaymentNo());

        if (wxResult == null) {
            return;
        }

        // 非终态，只更新查询时间
        if (!wxResult.getPaymentStatus().isTerminal()) {
            payOrderMapper.updateLastQueryTime(
                    fresh.getId(),
                    LocalDateTime.now()
            );
            return;
        }

        // ✅ 状态机更新：只允许非终态 → 终态
        int rows = payOrderMapper.updateStatusIfNonTerminal(
                fresh.getId(),
                wxResult.getPaymentStatus(),
                wxResult.getThirdPaymentNo(),
                wxResult.getPaymentTime()
        );

        if (rows == 1) {
            log.info("Job主动查单更新成功, paymentNo={}, status={}",
                    fresh.getPaymentNo(), wxResult.getPaymentStatus());

            // 触发后续业务（发货 / 记账）
            businessProcessor.onPaid(fresh.getPaymentNo());
        }

    } catch (Exception e) {
        log.error("微信查单异常, paymentNo={}", order.getPaymentNo(), e);
    }
}
```

### 4️⃣ UPDATE SQL（状态机核心）

```
UPDATE pay_order
SET
    payment_status   = #{status},
    third_transaction_no = #{thirdNo},
    payment_time     = #{paymentTime},
    last_query_time  = NOW(),
    update_time      = NOW()
WHERE id = #{id}
AND payment_status IN ('PENDING', 'PROCESSING');
```

✅ **这是兜底安全的灵魂**

- 

  回调先到 → Job 自动失效

- 

  Job 先到 → 回调自动失效

- 

  绝不会互相覆盖

## 五、并发 & 多实例安全设计（非常重要）

### ✅ 方案 1：数据库行锁（简单、主流）


### ✅ 方案 2：分布式锁（推荐）

```
String lockKey = "pay:query:" + order.getId();
if (distributedLock.tryLock(lockKey, 10, TimeUnit.SECONDS)) {
    try {
        processOrder(order);
    } finally {
        distributedLock.unlock(lockKey);
    }
}
```

## 六、微信 QPS & 容错策略

| 场景       | 处理方式                    |
| ---------- | --------------------------- |
| 微信超时   | 记录日志，下次再查          |
| 微信限流   | 指数退避（5m → 10m → 20m）  |
| 订单不存在 | 标记为 CLOSED               |
| 系统重启   | 从上次 last_query_time 继续 |

## 二、✅ 锁里【应该写什么】（标准支付中台写法）

### ✅ 只做 3 件事

```java
try {
    // 1️⃣ 再次确认 PayOrder 状态（防止并发）
    PayOrderVO current = payOrderService.getPayOrderByOrderNo(orderSn);
    if (current == null) {
        throw new BizException("支付订单不存在");
    }

    // 2️⃣ 只有“未支付”才允许继续
    if (current.getPaymentStatus() != PaymentStatusEnum.UNPAID) {
        throw new BizException("订单已在支付中或已完成");
    }

    // 3️⃣ （可选）标记支付中状态，防止重复进入
    payOrderService.updateStatusToPaying(current.getPaymentNo());

} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

✅ 特点
• 无 RPC

• 无 IO

• 只操作 DB

• 毫秒级完成

## 三、❌ 锁里【绝对不能写什么】

| 行为         | 后果                 |
| ------------ | -------------------- |
| 调 OMS Feign | Redis 锁持有时间过长 |
| 调微信       | 线程池打爆           |
| 发 MQ        | MQ 超时 → 锁不释放   |
| 金额比较     | 前面已经做过         |
| 构建支付请求 | 属于“业务处理”       |

👉 一句话：锁里只干“并发安全”这一件事

## 四、✅ 你这个方法的“完整正确结构”

```java
// ✅ 1. 前置校验（金额 / 状态 / openId）
// ✅ 2. 查 OMS / PayOrder
// ✅ 3. 金额比较（OMS 分 vs PayOrder 元→分）

RLock lock = redissonClient.getLock(ORDER_LOCK_PREFIX + orderSn);
lock.lock();
try {
    // ✅ 4. 并发安全校验（幂等核心）
    PayOrderVO current = payOrderService.getPayOrderByOrderNo(orderSn);
    if (current == null) {
        throw new BizException("支付订单不存在");
    }
    if (current.getPaymentStatus() != PaymentStatusEnum.UNPAID) {
        throw new BizException("订单已在支付中或已完成");
    }
    payOrderService.updateStatusToPaying(current.getPaymentNo());

} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}

// ✅ 5. 锁外：构建支付请求
// ✅ 6. 锁外：调微信 / 支付微服务
return processRealPayment(...);
```

✅ 这是“支付中台标准并发模型”

## 五、✅ 为什么要“先锁 → 再改状态 → 再解锁 → 再调微信”？

### 时序图（文字版）

```
线程A                线程B
 |---获取锁------------>|
 |---更新状态=PENDING-->|
 |---释放锁------------>|
 |---调用微信---------->|
                       |---获取锁------------>|
                       |---发现状态≠UNPAID---|
                       |---直接抛异常---------|
```

✅ 线程B 根本不会去调微信
✅ 微信只会被调一次
✅ 幂等 100% 成立

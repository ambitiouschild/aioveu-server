微信支付的**后端异步回调（Callback）**与**前端查询（Polling/查单）**是两套并行且互补的机制，**主流设计绝不以前端回调为最终依据，而是以“后端异步回调为主，前端主动查单为辅（兜底）”**。

以下是详细的时序逻辑与主流设计方案：

## 核心认知：前端回调 ≠ 支付成功

- 

  **wx.requestPayment / JSAPI 的 success 回调**：仅代表**用户在前端完成了支付操作（调起了收银台并点击完成）**，并不代表微信服务器已扣款成功并将结果通知到你的后端。网络抖动可能导致前端返回 success 而后端实际未收到钱。

- 

  **唯一真相源**：**微信支付服务器下发的异步回调（notify_url）** 或 **调用微信查询订单接口（API）返回的结果**。

------

## 标准时序流程（主流设计）

主流方案通常采用 **“前端轮询查单 + 后端接收回调 + 后端主动查单兜底”** 的三重保障模式：

### 阶段一：支付发起与返回

1. 

   前端调用 `wx.requestPayment`拉起微信收银台。

2. 

   用户支付完成，点击“完成”回到商户页面，**前端触发 `success`回调**。

3. 

   **关键点**：前端此时**立即**调用自家后端的「查询订单状态接口」，启动轮询（Polling），而不是直接展示支付成功页。

### 阶段二：后端异步回调（理想情况）

1. 

   微信服务器确认收款成功，向商户后端配置的 `notify_url`发送 **POST 异步通知**。

2. 

   商户后端接收通知，进行**验签、校验金额、幂等处理**（防止重复回调）。

3. 

   校验通过后，更新数据库订单状态为 `PAID`，并返回 `SUCCESS`告知微信停止重试。

4. 

   此时如果前端正在轮询，后端返回状态已变为“支付成功”，前端停止轮询并跳转成功页。

### 阶段三：异常情况与兜底（查单）

由于网络隔离、HTTPS超时、服务重启等原因，**后端可能收不到回调**，或者**回调比前端返回慢**。

1. 

   **前端兜底**：前端轮询自家后端接口（如每 2s/5s/10s 一次，建议最多 8-10 次或 45s 内）。若后端因未收到回调返回“未支付”，前端继续转圈等待；若超时仍未成功，提示用户“支付结果确认中，请勿重复支付”。

2. 

   **后端兜底（主动查单）**：商户后端启动定时任务（如订单创建 1 分钟后仍未支付），主动调用 **微信【查询订单API】** 核实状态。若查到已支付，则修正本地状态；若未支付，继续监控或在超时后关单。

## 时序对比表

| 维度         | 后端异步回调 (Callback) | 前端查询/轮询 (Polling)            | 后端主动查单 (Query API)   |
| ------------ | ----------------------- | ---------------------------------- | -------------------------- |
| **触发方**   | 微信服务器主动推        | 前端支付返回后主动拉               | 商户后端定时任务主动拉     |
| **可靠性**   | **高**（含重试机制）    | 中（依赖前端存活，切后台会停）     | **高**（最终兜底手段）     |
| **时效**     | 快（通常毫秒~秒级）     | 快（但可能有短暂状态滞后）         | 慢（用于补单，非实时展示） |
| **作用**     | **核心状态更新依据**    | **用户体验优化，实时展示结果**     | **解决单边账、回调丢失**   |
| **能否信？** | **能**（需验签）        | **不能**（仅作展示触发，不信状态） | **能**（官方API返回）      |

------

## 主流设计的最佳实践总结

1. 

   **前端不要信任 `success`回调**：拿到 `success`后必须**查单**，查到后端确认成功再跳转，否则提示加载中或稍后查看。

2. 

   **后端必须处理幂等**：回调可能重发，同一订单号处理逻辑需加锁或判重，避免发货/记账重复。

3. 

   **双重兜底**：

   - 

     **短期**：前端轮询（几秒到几十秒），解决回调延迟问题。

   - 

     **长期**：后端定时任务查单（分钟级），解决回调彻底丢失的单边账问题。

4. 

   **T+1 对账**：每日下载微信交易账单与本地订单比对，处理极端异常。

简单记忆：**前端负责“问”（轮询展示），后端负责“定”（接收回调+查单落库），微信回调是“准绳”。**

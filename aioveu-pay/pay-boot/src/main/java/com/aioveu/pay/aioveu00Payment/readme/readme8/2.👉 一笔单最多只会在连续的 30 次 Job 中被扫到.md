**结论先给你：是的，但——不是“重复扫同一批单”，也不是“每分钟从头扫”。**

## 一、先给一句话死答案

> ✅ **Job 是每分钟执行一次**
>
> ✅ **但每次执行，扫的 `create_time`区间是“新的”**
>
> ✅ **加上 `id < lastId`游标，保证不会重复扫**
>
> ❌ **不是每分钟把最近 30 分钟全量扫一遍**

## 二、你这个 Job 的“每分钟真相”

### 你的注解：

```
@Scheduled(cron = "0 */1 * * * ?")
```

👉 **每分钟的第 0 秒触发一次**

------

### 每次执行时，时间窗口是这样滑动的：

| Job 执行时间 | 扫描的 create_time 区间 |
| ------------ | ----------------------- |
| 10:00:00     | `[09:30:00, 10:00:00]`  |
| 10:01:00     | `[09:31:00, 10:01:00]`  |
| 10:02:00     | `[09:32:00, 10:02:00]`  |

✅ **窗口长度永远是 30 分钟**

✅ **窗口每分钟向右滑动 1 分钟**

✅ **新创建的支付单会被新窗口覆盖到**

✅ **老的支付单会逐渐“滑出窗口”**

------

## 三、那“会不会重复扫同一笔单”？

### ✅ 不会，原因有两个：

### 1️⃣ `create_time`是固定的（最重要）

```
支付单创建时间 = 09:35:00
```

那么：

| Job 时间 | 是否在窗口内 |
| -------- | ------------ |
| 10:00    | ✅ 是         |
| 10:01    | ✅ 是         |
| 10:05    | ❌ 已滑出     |

👉 **一笔单最多只会在连续的 30 次 Job 中被扫到**

👉 **不会无限重复**

------

### 2️⃣ `id < lastId`游标分页（兜底）

即使：

- 

  同一分钟内创建了多笔单

- 

  时间完全相同

游标保证：

```
AND id < lastId
ORDER BY create_time DESC, id DESC
```

👉 **同一批 Job 内不会重复**

👉 **跨批次也不会回头扫**

------

## 四、那“每分钟执行”的意义是什么？

### ✅ 核心目的：**缩短“发现异常”的延迟**

如果不每分钟，而是：

| 频率    | 后果                        |
| ------- | --------------------------- |
| 5 分钟  | 回调丢了，最多 5 分钟才发现 |
| 10 分钟 | 用户体验差                  |
| 1 分钟  | ✅ 兜底响应快                |
| 每秒    | ❌ 打爆 DB / 第三方          |

👉 **1 分钟是“支付兜底”的黄金频率**

## 五、一个非常关键的“认知升级”

### ❌ 直觉理解

> “每分钟执行 = 每分钟扫最近 30 分钟”

### ✅ 正确理解

> “每分钟执行 = **每分钟向前滑动 1 分钟的窗口**，对‘刚进入可疑期的单’做检查”

👉 **重点不是“扫多久”，而是“什么时候开始怀疑”**

## 七、生产环境的一个小优化（强烈推荐）

### ✅ 把“每分钟”改成“每 2~5 分钟”

```
@Scheduled(cron = "0 */5 * * * ?")
```

理由：

| 维度       | 1 分钟     | 5 分钟 |
| ---------- | ---------- | ------ |
| 发现延迟   | ✅ 更快     | ⚠️ 稍慢 |
| DB 压力    | ❌ 更高     | ✅ 更低 |
| 第三方 QPS | ❌ 更高     | ✅ 更低 |
| 实际收益   | ⚠️ 边际递减 | ✅ 足够 |

👉 **支付兜底 ≠ 实时回调**

👉 **5 分钟是业界常见值**

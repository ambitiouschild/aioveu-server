这个问题问得非常专业，说明你已经意识到：  
> `@Async` 放错位置，比不放还危险。

我直接给你一个企业级“标准答案”，不讲废话。

---

# ✅ 一句话结论（先记住）

> `@Async` 放在「BusinessProcessor 的具体实现类」里  
> 绝对不要放在 Composite 里，也不要放在支付核心流程里

---

# 一、正确位置（标准放法）

## ✅ 放在：具体业务 Processor 的实现方法中

```java
@Component
@Slf4j
public class OrderBusinessProcessor implements BusinessProcessor {

    @Override
    public boolean supports(PaymentScene scene) {
        return scene == PaymentScene.ORDER;
    }

    @Async("bizExecutor")
    @Override
    public void onPaid(String paymentNo) {
        log.info("异步处理订单支付, paymentNo={}", paymentNo);
        // 发货、减库存
    }
}
```

### ✅ 同理，其他 Processor 也一样

```java
@Async("bizExecutor")
@Override
public void onPaid(String paymentNo) { ... }
```

---

# 二、为什么不能放 Composite 里（非常关键）

### ❌ 错误示例（很多人会踩）

```java
@Component
public class BusinessProcessorComposite implements BusinessProcessor {

    @Async("bizExecutor")
    @Override
    public void onPaid(String paymentNo) {
        ...
    }
}
```

### ❌ 问题有 3 个：

| 问题            | 后果                                       |
| --------------- | ------------------------------------------ |
| 事务边界混乱    | Composite 可能在事务中，Async 后事务已提交 |
| 异常处理困难    | 一个失败，整个异步链丢失                   |
| 日志 & 追踪断裂 | TraceId 丢失，排查地狱                     |

✅ Composite 的职责是“编排”，不是“执行”

---

# 三、为什么不能放支付核心（Job / 回调）里

```java
// ❌ 不推荐
@Async
public void handlePaidOrder(...) { ... }
```

原因：
• 支付核心必须是 同步 + 强一致

• 支付状态变更 不能异步

• 否则会出现：

  • 钱扣了

  • 状态还是 PENDING

  • 用户投诉 / 资损


✅ 支付核心：同步
✅ 业务处理：异步

---

# 四、推荐的“分层异步模型”（企业级）

```
┌──────────────┐
│ 微信回调 / Job │  ← 同步（必须）
└──────┬───────┘
       ↓
┌────────────────────┐
│ BusinessProcessor  │  ← 同步调用 Composite
│    Composite       │
└──────┬───────┘
       ↓ 逐个调用
┌──────────────┐
│ Processor A  │ ← @Async ✅
│ Processor B  │ ← @Async ✅
│ Processor C  │ ← @Async ✅
└──────────────┘
```

✅ 支付状态：同步、强一致
✅ 业务动作：异步、最终一致

# 五、线程池 bizExecutor 怎么配（重要）

```java
@Configuration
@EnableAsync
public class ExecutorConfig {

    @Bean("bizExecutor")
    public ThreadPoolTaskExecutor bizExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200);
        executor.setThreadNamePrefix("biz-async-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
        executor.initialize();
        return executor;
    }
}
```

✅ `CallerRunsPolicy`：防止支付成功但业务丢失

# 六、异步下的“铁律”（非常重要）

## ✅ 1️⃣ 支付状态必须先落库（你已经做到了）

```java
updateStatusIfNonTerminal(...)
```

✅ 这一步 不能是异步

---

## ✅ 2️⃣ 异步方法必须幂等

```java
if (alreadyProcessed(paymentNo)) {
    return;
}
```

用：
• 支付单状态

• 或 `biz_process_log`


---

## ✅ 3️⃣ 异步失败必须可恢复

✅ 方案：
• 本地失败记录

• 定时任务重试

• 或人工干预

# 七、一个“企业级标准模板”（直接可用）

```java
@Async("bizExecutor")
@Override
public void onPaid(String paymentNo) {

    String traceId = MDC.get("traceId");
    MDC.put("traceId", traceId);

    try {
        if (processed(paymentNo)) {
            return;
        }

        doBusiness(paymentNo);

        markProcessed(paymentNo);

    } catch (Exception e) {
        log.error("业务处理失败, paymentNo={}", paymentNo, e);
        saveRetryRecord(paymentNo);
    } finally {
        MDC.clear();
    }
}
```

> ✅ 微信回调 / Job：同步、强一致

> ✅ 支付模块：完全不懂业务

这就是阿里 / 京东 / 美团支付中台的真实结构。

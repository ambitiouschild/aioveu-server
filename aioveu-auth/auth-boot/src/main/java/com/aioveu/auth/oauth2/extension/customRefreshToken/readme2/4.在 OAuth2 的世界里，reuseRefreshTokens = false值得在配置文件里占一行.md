简短结论先给你：

> 不是“语法必须”，但这是「工程上必须」的写法。

我分 “能不能不写 / 写在哪 / 写错会怎样 / 最佳实践” 四个层次给你讲清楚，你听完就知道为什么我强烈建议你写。

## 一、它“语法上”必须吗？❌

不必须。

因为你在代码里已经写了兜底值：

```java
@Value("${custom.oauth2.reuse-refresh-tokens:false}")
private boolean reuseRefreshTokens;
```

✅ Spring 的规则是：
• 配置里没写 → 用 `:false` 这个默认值

• 程序能正常启动

• 行为 = `false`


👉 所以从“能不能跑”的角度：可以不写

---

## 二、那为什么我每次都说「必须写」？✅（重点）

因为这是 生产级工程的铁律：

### ✅ 1️⃣ 消除“环境不确定性”

如果不写：

| 环境 | 实际值                    |
| ---- | ------------------------- |
| 本地 | false                     |
| 测试 | false                     |
| 预发 | false                     |
| 生产 | ❓（谁敢保证没人改代码？） |

👉 配置不可见 = 隐形炸弹

---

### ✅ 2️⃣ 防止“手滑改代码”

你现在是：

```java
@Value("${custom.oauth2.reuse-refresh-tokens:false}")
```

哪天有人重构，随手改成：

```java
@Value("${custom.oauth2.reuse-refresh-tokens:true}")
```

或者删掉默认值：

```java
@Value("${custom.oauth2.reuse-refresh-tokens}")
```

👉 生产直接爆炸

---

### ✅ 3️⃣ OAuth2 安全策略必须“显性声明”

`reuseRefreshTokens` 不是普通开关，它是：

> ✅ 安全策略  
> ✅ 合规要求  
> ✅ 审计项  

安全策略不允许“隐式默认”

---

### ✅ 4️⃣ 新人 / 运维 / 审计能看懂

配置文件里写了：

```yaml
custom:
  oauth2:
    reuse-refresh-tokens: false
```

任何人一看就知道：
> “哦，我们在用 Refresh Token Rotation”

否则别人只能去翻代码、猜逻辑。

---

## 三、写在哪？✅ 正确位置

### ✅ 推荐位置（和你现有结构一致）

```yaml
custom:
  oauth2:
    reuse-refresh-tokens: false
```

❌ 不要写成：

```yaml
security:
  oauth2:
    reuse-refresh-tokens: false # ❌ 不属于 security
```

❌ 不要写成：

```yaml
spring:
  security:
    oauth2:
      authorization-server:
        reuse-refresh-tokens: false # ❌ SAS 不认这个
```

👉 这是你自己定义的配置项，不是 Spring 的

---

## 四、如果你“真的不想写”，最低底线是什么？

### ✅ 最低可接受方案（不推荐）

1. 代码里必须有默认值
2. 注释必须写清楚

```java
/**
 * 是否复用刷新令牌
 * false = 每次刷新生成新 refresh_token（✅ 推荐，安全）
 * true  = 复用旧 refresh_token（❌ 不安全）
 */
@Value("${custom.oauth2.reuse-refresh-tokens:false}")
private boolean reuseRefreshTokens;
```

⚠️ 但即便如此，我仍然 不推荐

| Auth0                           | 强制 Rotation     |

👉 `false` 不是“可选项”，而是“事实标准”

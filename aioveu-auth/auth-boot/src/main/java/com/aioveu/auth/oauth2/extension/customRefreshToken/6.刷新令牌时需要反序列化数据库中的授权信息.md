刷新令牌时需要反序列化**数据库中的授权信息**。让我详细解释整个过程：

## 🔍 刷新令牌流程

```
刷新令牌请求
    ↓
验证 refresh_token
    ↓
从数据库查找对应的 OAuth2Authorization
    ↓
从 authorization.getAttribute(Principal.class.getName()) 获取 Authentication
    ↓
反序列化 Authentication 对象 ← 这里出错！
    ↓
使用 Authentication 生成新的 access_token
```

## 🎯 具体出错位置

在你的错误堆栈中：

```
at org.springframework.security.oauth2.server.authorization.JdbcOAuth2AuthorizationService$OAuth2AuthorizationRowMapper.parseMap
at org.springframework.security.oauth2.server.authorization.JdbcOAuth2AuthorizationService$OAuth2AuthorizationRowMapper.mapRow
at org.springframework.security.oauth2.server.authorization.JdbcOAuth2AuthorizationService.findByToken
at org.springframework.security.oauth2.server.authorization.authentication.OAuth2RefreshTokenAuthenticationProvider.authenticate
```

**关键**：`JdbcOAuth2AuthorizationService.findByToken()`从数据库读取授权记录，然后尝试反序列化为 Java 对象。

在 `attributes`字段中，存储了序列化的：

```
{

    "id": 6,

    "mobile": null,

  },
  "credentials": "{noop}",

```

## 🎯 为什么需要 Mixin？

Spring Security 6+ 的安全策略：

- 默认只允许反序列化**白名单中的类**
- `MemberDetails`不在白名单中
- 需要显式声明允许反序列化

## 📋 验证方法

你可以查看数据库中的实际数据：

```
-- 查看存储的 attributes
SELECT 
    id,
    principal_name,
    SUBSTRING(attributes, 1, 500) as attributes_preview
FROM oauth2_authorization
WHERE principal_name IS NOT NULL
LIMIT 5;
```

## 🔧 解决方案总结

| 方案                 | 优点                   | 缺点                         | 推荐     |
| -------------------- | ---------------------- | ---------------------------- | -------- |
| **添加 Mixin**       | 保持完整信息，标准做法 | 需要创建 Mixin 类            | ✅ 推荐   |
| **不存储完整对象**   | 无需 Mixin             | 信息不完整，刷新时需重新加载 | ⚠️ 次选   |
| **修改反序列化策略** | 简单粗暴               | 安全风险，不推荐生产         | ❌ 不推荐 |

## 🎯 你的选择

既然你需要完整的用户信息，**必须添加 `MemberDetails`的 Mixin**。这是标准做法，也是 Spring Security 推荐的方案。

**快速修复步骤：**

1. 创建 `MemberDetailsMixin.java`
2. 在配置中添加 `objectMapper.addMixIn(MemberDetails.class, MemberDetailsMixin.class)`
3. 重启应用
4. 重新登录生成新令牌（旧的令牌还是有问题）
5. 测试刷新

## 📊 注意

**旧的授权记录**在数据库中还包含没有 Mixin 的 `MemberDetails`，刷新时仍会出错。你有两个选择：

1. **清理旧数据**：

   

   ```
   DELETE FROM oauth2_authorization;
   ```

2. **让用户重新登录**生成新令牌

建议在测试环境先清理数据，确保所有新令牌都能正常刷新。

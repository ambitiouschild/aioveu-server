## ✅ 一句话死规则（请截图）

> **clientId 被禁用 / 删除时：  
> ✅ Auth 必须同步把该 clientId 从 Redis Set 中 `remove`**  
> ❌ 不能靠 TTL 自然过期（太慢、不安全）**

---

## ✅ 正确做法（终局版）

### ✅ 1️⃣ Auth 禁用 / 删除 clientId 时

```java
public void disableClient(String clientId) {
    // 1. 改 DB
    oauth2RegisteredClientBizMapper.disable(clientId);

    // 2. ✅ 立刻从 Redis 白名单移除
    redisTemplate.opsForSet()
            .remove("CLIENT_WHITELIST", clientId);

    log.info("【Auth】禁用 clientId，已从 Redis 白名单移除: {}", clientId);
}
```

✅ 立即生效  
✅ Gateway 下一请求就会拦截  
✅ 不需要重启  
✅ 不依赖 TTL  

---

### ✅ 2️⃣ 启用 / 注册时（你已有，确认即可）

```java
redisTemplate.opsForSet().add("CLIENT_WHITELIST", clientId);
redisTemplate.expire("CLIENT_WHITELIST", 30, TimeUnit.MINUTES);
```

---

## ✅ Gateway 里不用改任何东西

你现在的 Gateway 校验：

```java
Boolean cached = redisTemplate.opsForSet()
        .isMember("CLIENT_WHITELIST", clientId);

if (Boolean.TRUE.equals(cached)) {
    return true;
}
return false;
```

✅ 只要 Redis 里没了 → 拦截  
✅ 不用感知“什么时候删的”


## ✅ 常见坑提醒（你可能会遇到）

### ❌ 坑 1：Key 不一致
| Auth               | Gateway              |
| ------------------ | -------------------- |
| `CLIENT_WHITELIST` | `client:whitelist` ❌ |

👉 必须完全一致

✅ 建议抽成常量：

```java
public static final String CLIENT_WHITELIST_KEY = "auth:client:whitelist";
```

---

### ❌ 坑 2：事务里删 Redis
如果是 同一事务：

```java
@Transactional
public void disableClient(String clientId) {
    ...
    redisTemplate.opsForSet().remove(...); // ✅ OK
}
```

✅ Spring 事务提交后 Redis 操作没问题  
⚠️ 如果 Redis 失败，记录日志 + 告警（人工补偿）

---

## ✅ 用一句永远不会变的话总结

> 白名单“加”在注册/启用  
> 白名单“减”在禁用/删除  
> 加减都在 Auth，Gateway 只读

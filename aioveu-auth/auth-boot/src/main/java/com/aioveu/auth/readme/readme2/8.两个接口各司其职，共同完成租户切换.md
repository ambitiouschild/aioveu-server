**不是切换两次，而是一个完整的切换流程包含两个步骤**。

从您提供的代码来看，切换租户时确实**调用了两个后端接口**，但这不是重复切换，而是**一个完整的业务流程**：

## 完整切换流程解析

### 1. **AuthAPI.switchTenant(tenantId)** - 认证切换

```
// 第一个接口：刷新Token
const token = await AuthAPI.switchTenant(tenantId);
```

**目的**：获取包含新租户ID的新Token

**作用**：更新用户的认证信息，使后续请求携带正确的租户上下文

### 2. **TenantAPI.switchTenant(tenantId)** - 租户信息切换

```
// 第二个接口：更新租户信息
const tenantInfo = await TenantAPI.switchTenant(tenantId);
```

**目的**：获取租户的详细信息

**作用**：更新前端展示的租户名称、配置等信息

## 为什么需要两个接口？

这是**标准的权限与信息分离设计**：

| 步骤 | 接口                   | 目的              | 返回数据 | 存储位置                    |
| ---- | ---------------------- | ----------------- | -------- | --------------------------- |
| 1    | AuthAPI.switchTenant   | 更新认证（Token） | 新Token  | localStorage/sessionStorage |
| 2    | TenantAPI.switchTenant | 更新租户信息      | 租户详情 | Vuex/Pinia store            |

## 实际执行流程

```
async function switchTenant(tenantId: number) {
  // 步骤1：更新Token（如果需要）
  try {
    const token = await AuthAPI.switchTenant(tenantId);
    if (token) {
      AuthStorage.setTokens(token.accessToken, token.refreshToken);
    }
  } catch {
    // 如果接口不支持，跳过Token更新
  }
  
  // 步骤2：更新租户信息
  const tenantInfo = await TenantAPI.switchTenant(tenantId);
  if (tenantInfo) {
    setCurrentTenant(tenantInfo);  // 更新store和localStorage
  }
}
```

## 优化建议

如果觉得调用两个接口冗余，可以考虑：

### 方案1：合并接口

让后端提供一个接口同时返回Token和租户信息：

```
// 前端调用
const result = await AuthAPI.switchTenantWithInfo(tenantId);
// 返回: { accessToken, refreshToken, tenantInfo }
```

### 方案2：简化流程

如果您的系统不需要新Token，可以只调用租户信息接口：

```
// 修改 store 中的逻辑
async function switchTenant(tenantId: number) {
  // 只更新租户信息，不更新Token
  const tenantInfo = await TenantAPI.switchTenant(tenantId);
  if (tenantInfo) {
    setCurrentTenant(tenantInfo);
  }
}
```

## 总结

**这不是"切换了两次"**，而是：

1. **第一次切换**：认证层面的切换（更新Token）
2. **第二次切换**：业务层面的切换（更新租户信息）

这是一个**完整的多租户切换业务流程**，两个接口各司其职，共同完成租户切换。

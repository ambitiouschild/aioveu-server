在标准的 Spring Authorization Server 授权码模式（微信登录属于典型的授权码模式）中，`PasswordAuthenticationProvider`（密码模式）和 `WechatAuthenticationProvider`（微信授权模式）都需要处理 `token_version`。

因为它们本质上是两个不同的认证入口，只要用户通过其中任何一个入口成功登录，都需要生成一个新的版本号来“作废”之前的旧 Token。

下面是为你整理的这两个 Provider 的具体实现对照表，你可以直接照着改：

### 1. PasswordAuthenticationProvider（密码登录）
位置：在 `PasswordAuthenticationProvider` 类中，生成 Token 之前。
作用：用户账号密码登录时，更新版本号。

```java
// 在 PasswordAuthenticationProvider 的认证逻辑中（通常在生成 AccessToken 之前）
Long userId = ((SysUserDetails) usernamePasswordAuthentication.getPrincipal()).getUserId();

// 1. 写入 Redis（每次登录 +1）
String versionKey = "auth:user:token:version:" + userId;
Long tokenVersion = redisTemplate.opsForValue().increment(versionKey);
if (tokenVersion == null) {
    tokenVersion = 1L;
}

// 2. 把这个 version 放入 claims，让 JWT 携带出去
// （注意：这里是放入 claims，不是直接写 Redis）
claims.put("token_version", tokenVersion);
```

---

### 2. WechatAuthenticationProvider（微信登录）
位置：在 `WechatAuthenticationProvider` 类中，处理微信回调并生成 Token 的地方。
作用：用户微信扫码/静默登录时，同样需要更新版本号。

```java
// 在 WechatAuthenticationProvider 的认证逻辑中
// 假设你已经从微信用户信息中解析出了用户ID (userId)
Long userId = wechatUser.getId(); // 或者从 MemberDetails 中获取

// 1. 写入 Redis（同样每次登录 +1）
String versionKey = "auth:user:token:version:" + userId;
Long tokenVersion = redisTemplate.opsForValue().increment(versionKey);
if (tokenVersion == null) {
    tokenVersion = 1L;
}

// 2. 放入 claims
claims.put("token_version", tokenVersion);
```

---

### ⚠️ 关键提醒：关于 `MemberDetails` 和 `SysUserDetails`
你截图里有 `wechat` 包，里面应该用了 `MemberDetails`。
•   SysUserDetails（系统用户）：通常有 `userId`，直接用上面的代码。

•   MemberDetails（会员用户）：微信登录可能返回的是 `MemberDetails`，你需要确认 `MemberDetails` 里有没有 `getUserId()` 或者类似的唯一标识。如果有，就用那个 ID 作为 Redis Key；如果没有，可能需要用 `getOpenid()` 或者 `getId()` 作为 Key。


总结：
只要是登录入口，就必须写 `token_version`。所以 Password 和 Wechat 这两个 Provider 都要加，逻辑一模一样，只是获取的 `userId` 来源不同而已。

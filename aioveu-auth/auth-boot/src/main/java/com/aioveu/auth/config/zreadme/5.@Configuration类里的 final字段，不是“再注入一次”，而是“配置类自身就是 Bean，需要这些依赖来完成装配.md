## 二、为什么 `@Configuration` 类里还要“再写一遍”？

### ✅ 因为 `@Configuration` 本质也是一个 Bean

Spring 启动时会：

1. 实例化 `AuthorizationServerConfig`
2. 通过 构造器 注入这些依赖
3. 再调用 `@Bean` 方法

所以：

```java
@RequiredArgsConstructor
@Configuration
public class AuthorizationServerConfig {

    private final MemberDetailsService memberDetailsService;
    private final RedisTemplate<String, Object> redisTemplate;
    ...
}
```

👉 等价于：

```java
@Component
public class AuthorizationServerConfig {
    ...
}
```

只不过它是 配置型 Bean。

## 三、那为什么 `@Bean` 方法里还要再写参数？

这是 两个完全不同的注入点：

| 位置                             | 注入的是什么             | 用途             |
| -------------------------------- | ------------------------ | ---------------- |
| `AuthorizationServerConfig` 字段 | 配置类自身需要的依赖     | 装配其它 Bean    |
| `@Bean` 方法参数                 | 创建该 Bean 时需要的依赖 | 传给 Bean 构造器 |

### ✅ 举个例子（你代码里的真实情况）

#### 1️⃣ 配置类自身需要 `MemberDetailsService`

```java
private final MemberDetailsService memberDetailsService;
```

#### 2️⃣ 创建 `CustomRefreshTokenAuthenticationProvider` 时需要它

```java
new CustomRefreshTokenAuthenticationProvider(
    authorizationService,
    tokenGenerator,
    memberDetailsService,
    redisTemplate,
    reuseRefreshTokens
)
```

👉 不是重复注入
👉 是“配置类持有 → 传给 Provider”

## 五、哪些是“可以移走”的？（可选优化）

### ✅ 可以只留在方法参数里的

比如：

```java
@Bean
public SecurityFilterChain authorizationServerSecurityFilterChain(
    ...,
    SysUserDetailsService sysUserDetailsService
)
```

👉 `SysUserDetailsService` 只在方法里用
👉 可以不移成类字段

但这只是 整洁度优化，不是对错问题。

## 八、唯一一个“可以改进”的点（很小）

### ✅ `reuseRefreshTokens` 建议改成 `@Value`

```java
@Value("${custom.oauth2.reuse-refresh-tokens:false}")
private final boolean reuseRefreshTokens;
```

否则：
• 配置不可见

• 不同环境行为不一致

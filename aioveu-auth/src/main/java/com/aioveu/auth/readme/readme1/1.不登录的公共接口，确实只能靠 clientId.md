## 一、你现在的场景本质

| 场景       | 特点                      |
| ---------- | ------------------------- |
| 商品列表   | ✅ 公共接口                |
| 秒杀商品   | ✅ 公共接口                |
| 商品详情   | ✅ 公共接口                |
| ❌ 无 token | ❌ 无法从登录态拿 tenantId |

👉 你只能用 clientId 判断“这是哪个小程序 / H5 的数据”

---

## 二、✅ 为什么“只能传 clientId”？

| 如果不传       | 会怎样           |
| -------------- | ---------------- |
| 后端不知道是谁 | ❌ 查不到数据     |
| 用 tenantId    | ❌ 没登录，没有   |
| 用域名         | ❌ 不稳定、不安全 |

✅ clientId 是唯一“弱身份标识”

---

## 三、⚠️ 但你现在这样是「不安全的」

```js
const clientId = getClientId() || CLIENT_CONFIG.CLIENT_ID;
```

| 风险            | 说明         |
| --------------- | ------------ |
| 抓包改 clientId | ✅ 看别人数据 |
| 伪造 clientId   | ✅ 越权访问   |
| 爬虫批量扫      | ✅ 数据泄漏   |

---

## 四、✅ 正确、安全的公共接口设计（标准方案）

### ✅ 原则一句话

> 前端可以“声明”clientId，但后端必须“校验 + 白名单 + 限流”

---

## 五、✅ 推荐方案（你现在就该这么做）

### ✅ 前端（允许传，但不可信）

```js
export function listSpuPages(params) {
  const clientId = getClientId(); // 允许为空
  return request({
    url: `${AUTHSPU_BASE_URL}/spuLists`,
    method: "GET",
    params: {
      ...params,
      clientId
    }
  });
}
```

---

### ✅ 后端（✅ 核心校验）

```java
@GetMapping("/spuLists")
public PageResult<SpuVO> listSpuPages(
        @RequestParam(required = false) String clientId
) {
    if (!clientWhitelistService.isValid(clientId)) {
        throw new RuntimeException("非法 clientId");
    }

    return spuService.page(clientId);
}
```

---

### ✅ clientId 白名单校验（✅ 必须）

```java
public boolean isValid(String clientId) {
    return validClientIds.contains(clientId);
}
```

✅ 只允许已知 clientId
✅ 不允许任意字符串

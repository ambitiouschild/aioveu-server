## 三、✅ 可以不写 `tenant_id` 的条件（非常重要）

### ✅ 条件 1：ClientWhitelist **只关心**

```text
clientId 是否存在
clientId 是否被禁用
clientId 是否被删除
```

✅ 不涉及租户
✅ 不按租户过滤
✅ 不按租户做权限

👉 那 `tenant_id` 完全不需要

---

### ✅ 条件 2：映射关系只用于「管理端」

| 场景             | 是否需要 tenant_id |
| ---------------- | ------------------ |
| 管理后台展示     | ✅                  |
| 运营配置         | ✅                  |
| 运行时白名单校验 | ❌                  |

👉 校验 ≠ 管理

## 四、❌ 什么时候「必须写 tenant_id」？

| 场景                       | 说明   |
| -------------------------- | ------ |
| 一个 clientId 属于多个租户 | ❌ 必须 |
| 不同租户有同名 clientId    | ❌ 必须 |
| 白名单要按租户隔离         | ❌ 必须 |
| 将来要做租户级限流         | ❌ 必须 |

👉 只要有一条命中，就必须写

## 五、✅ 推荐你的最终表结构（✅ 可省略 tenant_id）

```sql
CREATE TABLE `oauth2_registered_client_biz` (
  `id` bigint NOT NULL AUTO_INCREMENT,

  `client_id` varchar(64) NOT NULL COMMENT 'OAuth2 客户端ID',

  `enabled` tinyint(1) NOT NULL DEFAULT 1 COMMENT '是否启用',
  `deleted` tinyint(1) NOT NULL DEFAULT 0 COMMENT '逻辑删除',

  `remark` varchar(255) DEFAULT NULL COMMENT '备注',

  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_client_id` (`client_id`),
  KEY `idx_enabled_deleted` (`enabled`, `deleted`)
) ENGINE=InnoDB
DEFAULT CHARSET=utf8mb4
COMMENT='OAuth2 客户端业务状态表（auth 本地校验用）';
```

✅ 最简
✅ 最快
✅ 最稳

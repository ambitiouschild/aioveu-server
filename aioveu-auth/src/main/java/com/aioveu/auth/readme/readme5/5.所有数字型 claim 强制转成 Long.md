
## 二、根因定位（精确到行）

### 异常栈里这一行是关键：

```
at com.aioveu.auth.config.JwtTokenCustomizerConfig.lambda$jwtTokenCustomizer$0(JwtTokenCustomizerConfig.java:101)
```

结合你日志里这句话：

```
【JwtTokenCustomizer】用户详情字段值:
userId=1, username=root, ..., tenantId=0
```

✅ **问题就是：**

| 字段           | Java 类型     | JWT 要求 |
| -------------- | ------------- | -------- |
| `tenantId = 0` | `Integer`     | ❌        |
| `userId = 1`   | `Integer`     | ❌        |
| JWT claim      | 必须是 `Long` | ✅        |

------

## 三、为什么 Spring 会在这里炸

Spring Security OAuth2 在 **生成 JWT 时**：

```
claims.claim("tenant_id", tenantId);
```

内部会做类型校验 / 转换，最终要求：

- 

  `Number`→ **必须是 `Long`**

- 

  `Integer`→ **直接 ClassCastException**

⚠️ 注意：

> **这不是 Jackson 的问题，是 JWT 标准 + Spring 的限制**

## 五、✅ 正确改法（三种，推荐第一种）

------

### ✅ 方案一（✅ 强烈推荐，最安全）

**所有数字型 claim 强制转成 `Long`**

```

        ? sysUserDetails.getTenantId().longValue()
        : null);

        ? sysUserDetails.getUserId().longValue()
        : null);
```

✅ 优点：

- 

  100% 不会再炸

- 

  不依赖 JVM 自动装箱

- 

  JWT 标准友好

## 六、你这个 JwtTokenCustomizer 的「正确模板」（直接用）

```
if (principal instanceof SysUserDetails sysUserDetails) {

    claims.claim("user_id", sysUserDetails.getUserId() != null
            ? sysUserDetails.getUserId().longValue()
            : null);

    claims.claim("username", sysUserDetails.getUsername());

    claims.claim("tenant_id", sysUserDetails.getTenantId() != null
            ? sysUserDetails.getTenantId().longValue()
            : null);

    claims.claim("data_scope", sysUserDetails.getDataScope());

    claims.claim("can_switch_tenant", sysUserDetails.getCanSwitchTenant());
}
```

✅ **这是 Authorization Server 官方推荐姿势**

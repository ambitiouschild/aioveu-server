## 二、为什么“之前什么都不写就能调”

### 1️⃣ 你之前的接口长这样（✅ 关键）

```java
@PostMapping("/aioveu/api/v8/admin/auth/auth/logout")
public Result<?> logout() {
    ...
}
```

### 2️⃣ 你之前的 Security 是这种（✅ 典型）

```java
.authorizeHttpRequests(auth -> auth
    .anyRequest().authenticated()
)
```

👉 这个规则的意思是：

> “只要你带了 JWT，我就让你过”

而你的请求：

```http
DELETE /aioveu/api/v8/admin/auth/auth/logout
Authorization: Bearer xxx
```

✅ JWT 合法  
✅ 路径匹配  
✅ 没被额外拦截  
✅ 自然就能访问

---

## 三、那为什么现在“看起来像必须写”

### ✅ 真正的变化只有一个：

> 你现在开始“显式地控制白名单”了

以前是：

```text
所有 /aioveu/api/v8/** 都要 JWT
```

现在是：

```text
/aioveu/api/v8/** 里，只有白名单 permitAll
其余 authenticated
```

👉 这个变化，是你自己引入的
👉 不是 Spring 升级
👉 不是 SAS 限制

---

## 四、为什么 `/logout` 现在“必须写进白名单”

因为你现在的规则是：

```java
.requestMatchers(matchers.toArray(new RequestMatcher[0]))
.permitAll()
.anyRequest()
.authenticated()
```

这意味着：

| 接口       | 结果            |
| ---------- | --------------- |
| 在白名单   | ✅ permitAll     |
| 不在白名单 | ❌ authenticated |
| 即使是 JWT | ❌ 403           |

👉 不是 token 不行
👉 是“授权规则”变了

## 五、那 `adminApiSecurityFilterChain` 为什么“之前也不需要”

### 以前：

### 现在：

• 你开始 精细化控制路径

• 开始 拆 FilterChain

• 开始 显式声明谁可以访问什么


👉 不是必须
👉 是你“主动收紧”了规则

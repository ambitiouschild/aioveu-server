然而，根据问题描述，我们有一个微服务架构，包括auth-service和system-service等。在微服务之间使用OAuth2.0进行认证，通常使用客户端凭证模式（client_credentials）来获取令牌，然后使用公钥来验证令牌。

在当前的配置中，我们已经生成了密钥对，并且auth-service负责签发令牌。其他微服务（如system-service）需要验证JWT令牌，因此它们需要公钥。

目前，公钥是通过JWK Set端点暴露的，其他服务可以通过访问auth-service的/.well-known/jwks.json端点来获取公钥。但是，在微服务架构中，我们可能希望每个服务本地都有公钥，或者从一个统一的地方获取公钥，而不是每次验证时都请求端点。

我们可以考虑以下方案：

1. 

   将公钥配置在每个微服务的配置文件中，这样它们就可以本地验证JWT，而不需要连接auth-service。但这样密钥轮换会麻烦。

2. 

   使用一个配置服务器来分发公钥，每个服务启动时从配置服务器获取公钥。

3. 

   使用服务发现机制，让微服务在启动时从auth-service获取公钥，并缓存起来。

但是，根据我们的代码，我们目前是将公钥存储在Redis中，并且auth-service在启动时会从Redis加载JWK Set，如果不存在则生成。这意味着其他微服务也可以从Redis获取公钥，因为Redis是共享的。

因此，我们可以让每个微服务都从Redis中读取JWK Set，然后使用公钥验证JWT。这样，我们就不需要每个服务都配置公钥，而且密钥轮换时，只需要更新Redis中的JWK Set，各个微服务在下次启动时（或者定时）重新加载即可。

但是，注意：我们目前的代码中，只有auth-service在启动时生成密钥对并存入Redis，其他微服务并没有自动从Redis加载JWK Set的机制。因此，我们需要在其他微服务中也实现类似的JWKSource，从Redis加载JWK Set。

然而，实际上，其他微服务并不需要完整的JWK Set，它们只需要公钥来验证JWT。因此，我们可以创建一个公共的模块，让所有微服务都使用同一个Redis实例来获取公钥。

修改计划：

1. 

   将JWK Set的存储和读取抽象成一个公共组件，比如一个JwkService，它负责从Redis读取JWK Set，并提供公钥。

2. 

   在auth-service中，我们使用这个JwkService来获取JWKSource，并用于签发令牌。

3. 

   在其他微服务中，我们也使用这个JwkService来获取公钥，并配置JwtDecoder。

但是，我们注意到当前的代码中，auth-service已经将JWK Set存入Redis，并且有一个方法从Redis加载。我们可以将这部分代码提取到公共模块中。

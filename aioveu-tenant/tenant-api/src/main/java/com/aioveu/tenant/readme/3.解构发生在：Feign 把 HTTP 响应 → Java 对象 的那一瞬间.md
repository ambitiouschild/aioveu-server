## 一、先给你一句“终极答案”

> ✅ 解构不在 API 定义里
>
> ✅ 解构在 FeignDecoder 里
>
> ✅ API 写 `Result<T>` 只是“给解构器一个入口”

👉 API 写 `Result` ≠ 解构
👉 FeignDecoder 把 `Result.data` 拿出来才是解构

---

## 二、你现在这个“疑惑”的本质（非常对）

你现在的想法是：

> “API 写了 `Result<T>`  
> 调用方也要写 `Result<T>`  
> 那解构在哪？”

答案是：

> 解构发生在：Feign 把 HTTP 响应 → Java 对象 的那一瞬间

## 四、为什么你之前那个接口“没解构”（关键）

### ❌ 你之前这个接口

```java
TenantWxAppInfo getTenantWxAppInfoByClientId(...)
```

Feign 说：

> “返回值不是 Result，我不走 FeignDecoder  
> 直接 Jackson 反序列化”

👉 解构器根本没机会进场

---

## 五、为什么“API 写 Result，调用方也要写 Result”

因为：

> Feign 是“静态代理”，不是“动态转换”

```java
// Feign 生成的代理类（伪代码）
class TenantFeignClientProxy implements TenantFeignClient {

    @Override
    public TenantWxAppInfo getTenantWxAppInfoByClientId(String clientId) {
        // 1. 发 HTTP
        // 2. 用 Result<TenantWxAppInfo> 反序列化
        // 3. 解构
        // 4. 返回 TenantWxAppInfo
    }
}
```

👉 方法签名决定了“调用方看到什么”
👉 不能偷偷把 `Result` 变成 `T`

## 六、你现在这个 FeignDecoder 的“正确心智模型”

| 角色         | 职责                |
| ------------ | ------------------- |
| API 接口     | ✅ 声明“我是 Result” |
| FeignDecoder | ✅ 解构 Result.data  |
| 调用方       | ✅ 拿到 Result       |
| 调用方代码   | ✅ 自己 `.getData()` |

👉 解构是“中间过程”，不是“最终结果”

## 七、你现在这套的“正确写法”（标准模板）

### ✅ Feign API（必须）

```java
@GetMapping("/xxx")

```

### ✅ 调用方（必须）

```java
Result<TenantWxAppInfo> result =

if (result == null || result.getData() == null) {
    throw new OAuth2AuthenticationException("无效的客户端ID");

```

## 一、先给你一句“终极定义”

> Feign 的解构：
>
> HTTP Response（JSON 字符串）
> → Result
> → Result.data
> → Java 对象

👉 从头到尾，只有 HTTP 被“解”

---

## 二、你刚才那句话，拆开来讲（非常准）

> “解构解的是 HTTP 啊，不是 wide 数据”

这里的 wide 数据（我理解你想说的是 VO / DTO） 是 结果，不是 原料。

### ✅ 原料（被解的东西）

```http
HTTP/1.1 200
Content-Type: application/json

{
  "code": 0,
  "msg": "success",
  "data": {
    "tenantId": 3,
    "enabled": 1
  }
}
```

### ✅ 解构器做的事

```java
// 1. 读 HTTP body（字符串）
// 2. 转成 Result
// 3. 取出 data 字段
// 4. 把 data 转成 TenantWxAppInfo
```

👉 HTTP 是唯一被“解构”的东西
👉 VO / DTO 只是反序列化的产物

---

## 三、为什么很多人（包括之前的你）会“误以为解的是 VO”

因为 FeignDecoder 的代码长这样：

```java
Object data = result.getData();
return objectMapper.convertValue(data, targetType);
```

👉 看起来像是“在解 VO”
👉 实际上是：
• HTTP → Result（解构）

• Result.data → VO（反序列化）


---

## 四、你现在这个认知，直接推翻了“错误写法”

### ❌ 错误写法（你现在这个）

```java
TenantWxAppInfo getTenantWxAppInfoByClientId(...)
```

👉 Feign 在干：

```
HTTP(JSON)
 → TenantWxAppInfo（直接反序列化）
```

❌ 根本没有 HTTP → Result 这一步
❌ 解构根本没发生

---

### ✅ 正确写法（你现在应该坚持的）

```java
Result<TenantWxAppInfo> getTenantWxAppInfoByClientId(...)
```

👉 Feign 在干：

```
HTTP(JSON)
 → Result
 → data
 → TenantWxAppInfo
```

✅ HTTP 被解构了
✅ Enum 问题才有机会统一处理

---

## 五、用一句话把你这套认知“封神”

> ✅ Feign 的解构，只认 HTTP
>
> ✅ Result 只是 HTTP 的“外壳”
>
> ✅ VO / DTO 只是 Jackson 的产物
>
> ❌ 不存在“解构 VO”这件事

---

## 六、你现在这套 Feign 接口的“最终正确世界观”

| 层           | 角色                 |
| ------------ | -------------------- |
| HTTP         | ✅ 被解构的对象       |
| Result       | ✅ 解构容器           |
| FeignDecoder | ✅ 解构器             |
| VO / DTO     | ❌ 不参与解构         |
| 调用方       | ✅ 只消费 Result.data |

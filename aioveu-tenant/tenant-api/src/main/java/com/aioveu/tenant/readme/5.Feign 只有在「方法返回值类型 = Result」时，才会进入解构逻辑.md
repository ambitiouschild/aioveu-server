是的，只有看到 `Result` 才会解构。  
而且你这个问题，问到了 FeignDecoder 的“唯一入口条件”。

我直接给你一个不带任何模糊空间的答案。

---

## 一、一句话先给你（死标准）

> ✅ Feign 只有在「方法返回值类型 = Result」时，才会进入解构逻辑
>
> ❌ 只要不是 Result，Feign 就当普通 HTTP 反序列化处理

没有例外，没有隐式解构。

---

## 二、Feign 解构的“唯一判断依据”

Feign 在生成代理类时，只做一件事：

```java
if (method.getReturnType() == Result.class) {
    // ✅ 进入 FeignDecoder 解构
} else {
    // ❌ 直接 Jackson 反序列化 HTTP body
}
```

👉 它不看：
• 方法名

• URL

• HTTP 返回内容

• JSON 里有没有 `data`


👉 它只看：
• Java 方法的返回值类型


---

## 三、你现在这套的“对照表”（你照着抄）

| 方法签名                | 是否解构 | 原因                |
| ----------------------- | -------- | ------------------- |
| `Result<T>`             | ✅        | 返回值类型是 Result |
| `Result<List<T>>`       | ✅        | 返回值类型是 Result |
| `Long`                  | ❌        | 不需要解构          |
| `String`                | ❌        | 不需要解构          |
| `Boolean`               | ❌        | 不需要解构          |
| `TenantWxAppInfo`       | ❌        | 不是 Result         |
| `List<TenantWxAppInfo>` | ❌        | 不是 Result         |
| `Result`                | ❌        | 泛型丢失，不安全    |

---

## 四、为什么“HTTP 返回里有 Result，Feign 也不解构”

因为 Feign 不相信 HTTP，它只相信 Java 类型。

### ❌ 错误理解

> HTTP 返回了 Result → Feign 应该解构

### ✅ 正确理解

> 方法返回值类型是 Result → Feign 才会解构

👉 Feign 是编译期契约
👉 HTTP 是运行期数据

---

## 五、你现在这个 FeignDecoder 的“真实身份”

它不是一个“智能解析器”，而是一个：

> “Result 类型的专用处理器”

```java
if (returnType == Result.class) {
    // 我只负责 Result
}
```

👉 不是 Result → 不解构
👉 这是设计，不是限制

👉 Enum 炸了，它不负责

> ✅ 解构是“类型驱动的”
>
> ✅ 不是“内容驱动的”
>
> ✅ 只有 Result 才是解构入口

>

>
> ✅ 这是 Feign 的硬性契约，不是可选项

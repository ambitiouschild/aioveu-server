## 核心思路：完全以Token为中心

**整个多租户系统的租户ID传递，完全依赖于JWT Token**。这是一个非常标准和优雅的方案：

## 完整的数据流向

```
用户切换租户
    ↓
认证服务生成新Token（包含新tenant_id）
    ↓
前端保存新Token
    ↓
后续请求携带新Token
    ↓
每个微服务解析Token → 获取tenant_id
    ↓
设置到ThreadLocal → 用于数据过滤
```

## 各层实现细节

### 1. **Token生成层**（认证服务）

```
// 生成包含租户ID的Token
public String generateToken(SysUserDetails userDetails) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("tenant_id", userDetails.getTenantId());  // ✅ 核心
    claims.put("user_id", userDetails.getUserId());
    claims.put("username", userDetails.getUsername());
    
    return Jwts.builder()
        .setClaims(claims)
        .signWith(key, SignatureAlgorithm.RS256)
        .compact();
}
```

### 2. **Token解析层**（每个微服务）

```
// 过滤器解析Token
@Component
public class TenantFromTokenFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                   HttpServletResponse response, 
                                   FilterChain filterChain) {
        try {
            String token = extractToken(request);
            if (token != null) {
                Jwt jwt = jwtDecoder.decode(token);
                Long tenantId = jwt.getClaim("tenant_id");  // ✅ 从这里获取
                
                if (tenantId != null) {
                    TenantContextHolder.setTenantId(tenantId);
                }
            }
            filterChain.doFilter(request, response);
        } finally {
            TenantContextHolder.clear();
        }
    }
}
```

### 3. **数据访问层**（MyBatis-Plus）

```
// 从Token获取租户ID进行过滤
@Component
public class TokenTenantHandler implements TenantLineHandler {
    @Override
    public Expression getTenantId() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        
        if (auth != null && auth.getPrincipal() instanceof Jwt jwt) {
            Long tenantId = jwt.getClaim("tenant_id");  // ✅ 从Token获取
            if (tenantId != null) {
                return new LongValue(tenantId);
            }
        }
        
        throw new RuntimeException("Token中缺少租户ID");
    }
}
```

## 为什么这个方案更好？

### 对比其他方案：

| 方案            | 优点                         | 缺点                     |
| --------------- | ---------------------------- | ------------------------ |
| **Token传递**   | ✅ 标准化、安全、无需额外传递 | 需要切换时生成新Token    |
| 请求头传递      | 简单直接                     | ❌ 可能被篡改，需网关处理 |
| ThreadLocal传递 | 无网络开销                   | ❌ 无法跨服务传递         |

### 您的方案优势：

1. **✅ 符合OAuth2/JWT标准** - 租户ID是Token的合法声明
2. **✅ 安全防篡改** - Token有签名验证
3. **✅ 微服务无状态** - 每个服务独立解析Token
4. **✅ 无需额外约定** - 不用定义特殊请求头
5. **✅ 易于监控** - 可以从Token直接追踪租户

## 注意事项

### 1. **Token刷新问题**

```
// 刷新Token时也要保持租户ID
public OAuth2AccessToken refreshToken(String refreshToken) {
    // 解析原Token获取租户ID
    Long tenantId = parseTenantIdFromRefreshToken(refreshToken);
    
    // 生成新Token时包含相同租户ID
    return generateTokenWithTenantId(userDetails, tenantId);
}
```

### 2. **并发请求处理**

切换租户期间，可能有以下情况：

- 请求A：旧Token（tenant_id=1）还在处理
- 用户切换租户
- 请求B：新Token（tenant_id=2）开始处理

**解决方案**：旧Token可以设置较短有效期，或加入黑名单。

### 3. **调试和监控**

```
// 添加调试信息
@GetMapping("/debug/tenant-from-token")

    result.put("token_claims", jwt.getClaims());
    result.put("thread_local_tenant_id", TenantContextHolder.getTenantId());
```

## 总结

**完全以Token为中心的多租户方案**：

1. **切换租户** → 生成新Token（包含新tenant_id）
2. **前端存储** → 保存新Token
3. **请求携带** → 每次请求带Token
4. **微服务解析** → 每个服务从Token获取tenant_id
5. **数据过滤** → MyBatis-Plus用tenant_id过滤数据

这是一个**标准、安全、简洁**的方案，完全避免了微服务间ThreadLocal无法共享的问题，是**推荐的最佳实践**！

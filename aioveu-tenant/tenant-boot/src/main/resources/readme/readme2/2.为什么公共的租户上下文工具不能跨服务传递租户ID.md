我的过滤器和租户id上下文工具和mybatis配置都是公共的，为啥租户上下文工具不能互相传递租户id

您的问题很好！这涉及到**微服务架构中ThreadLocal的局限性**。让我详细解释：

## 问题本质

即使您的过滤器、租户工具类和MyBatis配置是**公共的**（同一个JAR包），在不同微服务中依然**无法共享ThreadLocal**。

## 原因分析

### 1. **JVM进程隔离**

```
// 服务A的JVM进程
TenantContextHolder.setTenantId(1L);  // 在服务A的内存中

// 服务B的JVM进程（完全独立）
Long id = TenantContextHolder.getTenantId();  // null! 不同的内存空间
```

**关键点**：每个微服务运行在独立的JVM进程中，有独立的内存空间。

### 2. **ThreadLocal的工作原理**

```
public class ThreadLocal<T> {
    // ThreadLocalMap存储在Thread对象中
    ThreadLocal.ThreadLocalMap threadLocals = null;
    
    public void set(T value) {
        Thread t = Thread.currentThread();
        ThreadLocalMap map = getMap(t);  // 获取当前线程的Map
        if (map != null) {
            map.set(this, value);
        } else {
            createMap(t, value);
        }
    }
}
```

**关键点**：

- `ThreadLocal`数据存储在 **Thread对象** 中
- 不同JVM进程的Thread对象是**完全独立**的
- HTTP请求在不同服务间传递时，**线程是全新的**

## 租户ID传递流程示例

### 错误理解（您的预期）

```
前端 → 网关 → 服务A → 服务B → 服务C
   ↓      ↓       ↓       ↓       ↓
          设置    设置    设置    设置
   ↓      ↓       ↓       ↓       ↓
          获取    获取    获取    获取
```

### 实际发生的情况

```
前端 → 网关 → 服务A（JVM进程1） → 服务B（JVM进程2） → 服务C（JVM进程3）
   ↓      ↓            ↓               ↓               ↓
          设置ThreadLocal 设置ThreadLocal  设置ThreadLocal
   ↓      ↓            ↓               ↓               ↓
          获取OK      重新获取null    重新获取null    重新获取null
```

## 解决方案对比

### 方案1：通过HTTP请求头传递（推荐）

```
// 在网关或前置服务中
@Component
public class TenantHeaderFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                   HttpServletResponse response, 
                                   FilterChain filterChain) {
        // 解析Token获取租户ID
        Long tenantId = extractTenantIdFromToken(request);
        
        // 创建新请求，添加租户ID到Header
        HttpServletRequest wrapper = new HttpServletRequestWrapper(request) {
            @Override
            public String getHeader(String name) {
                if ("X-Tenant-ID".equalsIgnoreCase(name)) {
                    return String.valueOf(tenantId);
                }
                return super.getHeader(name);
            }
        };
        
        filterChain.doFilter(wrapper, response);
    }
}
```

## 完整的工作流程

### 1. **网关层**（认证+设置Header）

```
// 网关过滤器
public class GatewayTenantFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 解析JWT Token
        String token = extractToken(exchange.getRequest());
        Jwt jwt = jwtDecoder.decode(token);
        
        // 获取租户ID
        Long tenantId = jwt.getClaim("tenant_id");
        
        // 添加到请求头
        ServerHttpRequest newRequest = exchange.getRequest().mutate()
            .header("X-Tenant-ID", String.valueOf(tenantId))
            .build();
        
        return chain.filter(exchange.mutate().request(newRequest).build());
    }
}
```

### 2. **每个微服务的过滤器**（读取Header+设置ThreadLocal）

```
// 公共过滤器（每个服务都使用）
@Component
public class MicroserviceTenantFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                   HttpServletResponse response, 
                                   FilterChain filterChain) {
        try {
            // 从Header获取租户ID
            String tenantIdStr = request.getHeader("X-Tenant-ID");
            if (StringUtils.hasText(tenantIdStr)) {
                Long tenantId = Long.parseLong(tenantIdStr);
                
                // ✅ 设置到本服务的ThreadLocal
                TenantContextHolder.setTenantId(tenantId);
                
                // ✅ 同时设置到SecurityContext（如果需要）
                updateSecurityContextTenantId(tenantId);
            }
            
            filterChain.doFilter(request, response);
            
        } finally {
            // 清理ThreadLocal
            TenantContextHolder.clear();
        }
    }
}
```

## 为什么需要这样设计？

### 对比传统单体应用

| 对比项      | 单体应用            | 微服务架构       |
| ----------- | ------------------- | ---------------- |
| 内存        | 共享JVM进程         | 独立JVM进程      |
| ThreadLocal | 线程内共享          | 进程内共享       |
| 租户传递    | 直接通过ThreadLocal | 必须通过显式传递 |

### 技术限制

1. **进程边界**：HTTP请求跨越进程边界，线程上下文丢失
2. **网络调用**：RPC/HTTP调用创建新线程
3. **异步处理**：`@Async`、消息队列等创建新线程

## 总结

**为什么公共的租户上下文工具不能跨服务传递租户ID**：

1. ✅ **ThreadLocal是进程内共享**，不是跨进程共享
2. ✅ **每个微服务是独立的JVM进程**，有独立的内存空间
3. ✅ **HTTP请求在不同服务间传递时，线程是全新的**

**解决方案**：

1. ✅ **必须通过HTTP请求头显式传递**租户ID
2. ✅ **每个服务独立处理**请求头，设置到自己的ThreadLocal
3. ✅ **服务间调用**通过Feign拦截器自动传递

**简单记忆**：在微服务架构中，**任何需要跨服务传递的数据，都必须通过显式的网络通信（请求头、消息体等）来传递**，不能依赖内存共享。

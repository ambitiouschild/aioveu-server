## 一、先确认你现在的状态（非常关键）

你现在实际上已经具备了：

✅ **多前端小程序**（商家各自注册）

✅ **统一后端服务**

✅ **sys_tenant.wx_appid 作为租户唯一入口**

✅ **sys_user.tenant_id 做数据隔离**

✅ **后端通过 code2Session → wx_appid → tenant_id**

👉 这意味着：

> **前端彻底不再需要知道 tenantId，也不需要写死任何业务参数**

这一步做完，你已经避开了 90% 早期多租户项目的坑。

------

## 二、接下来你必须注意的 3 个关键点（很重要）

### ✅ 1️⃣ 登录接口里，租户识别必须是「第一件事」

**正确顺序（强烈建议）**

```
1️⃣ 接收 code
2️⃣ code2Session → wx_appid
3️⃣ 根据 wx_appid 查 sys_tenant
4️⃣ 拿 tenant_id
5️⃣ 再用 tenant_id + openid 查 / 创建 sys_user
```

❌ 错误顺序（很多人会犯）：

- 先创建 user
- 再补 tenant
- 或允许 user 没有 tenant_id

👉 **tenant_id 必须在 user 创建时就确定**

------

### ✅ 2️⃣ 所有业务 SQL，必须隐式带 tenant_id

你现在已经有：

```
INDEX idx_tenant_id(tenant_id)
```

但**真正决定安全的，是代码习惯**：

✅ 正确（MyBatis / JPA / SQL）：

```
SELECT * FROM order_info
WHERE id = ?
AND tenant_id = ?
```

✅ MyBatis-Plus / JPA 多租户插件（如果你用）：

- 自动拼接 tenant_id
- 防止误写漏写

⚠️ **不要相信“只靠前端过滤”**

------

### ✅ 3️⃣ 禁止任何“跨 tenant 查询”的通用接口

比如这种接口 **必须封死**：

```
GET /api/users
```

✅ 只允许：

```
GET /api/users?tenantId=当前登录租户
```

或者直接：

```
@GetMapping("/users")
public List<User> list() {
    Long tenantId = CurrentUser.getTenantId();
    return userService.listByTenant(tenantId);
}
```

## 三、你现在这个模型已经支持的「未来能力」

你现在这套表 + wx_appid，已经天然支持：

✅ 商家换小程序（只改 sys_tenant）

✅ 一个租户多小程序（稍加扩展）

✅ 平台级管理后台（tenant = 0）

✅ 后期升级为“聚合小程序”（不改前端）

也就是说：

> **你现在做的不是“临时方案”，而是一个“可平滑演进的 SaaS 底座”**

首先是数据库所有表添加租户id字段吗

**不完全是**。并不是所有表都需要加tenant_id字段，需要根据表的类型和用途来决定。

## 需要添加 tenant_id 的表（业务数据表）：

### 1. **所有业务实体表** ✅

这些表存储租户的业务数据，必须加tenant_id：

```
-- PMS 产品相关
pms_product         -- 产品表
pms_category        -- 产品分类
pms_sku             -- SKU表
pms_inventory       -- 库存表

-- OMS 订单相关
oms_order           -- 订单主表
oms_order_item      -- 订单明细
oms_order_log       -- 订单日志
oms_shipping        -- 发货单

-- UMS 用户相关
ums_customer        -- 客户表
ums_address         -- 地址表
ums_member          -- 会员表
ums_integral        -- 积分表

-- SMS 消息相关
sms_template        -- 消息模板
sms_record          -- 发送记录
sms_blacklist       -- 黑名单

-- PAY 支付相关
pay_transaction     -- 交易记录
pay_refund          -- 退款记录
pay_settlement      -- 结算记录
```

### 2. **租户自定义配置表** ✅

```
tenant_config       -- 租户配置
tenant_menu         -- 租户菜单
tenant_role         -- 租户角色
tenant_dept         -- 租户部门
```

## 不需要添加 tenant_id 的表（系统表）：

### 1. **系统基础表** ❌

```
-- 系统管理表（全局）
sys_tenant          -- 租户信息表 ❌（这个表记录租户本身，不需要tenant_id）
sys_user            -- 系统用户表（可能有tenant_id，但逻辑特殊）
sys_dict            -- 字典表
sys_config          -- 系统配置
sys_log             -- 操作日志
sys_file            -- 文件存储
```

### 2. **代码生成相关表** ❌

```
gen_table           -- 代码生成表
gen_table_column    -- 代码生成字段
```

### 3. **全局数据表** ❌

```
area                -- 地区表
country             -- 国家表
currency            -- 货币表
language            -- 语言表
```

## 特殊情况分析：

### 1. **sys_user 用户表** ⚠️

用户表比较特殊，有多种处理方式：

**方案A：全局用户表**（推荐用于SaaS）

```
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,  -- 所属租户
    username VARCHAR(50),
    email VARCHAR(100),
    -- 用户只能访问自己租户的数据
    INDEX idx_tenant (tenant_id)
);
```

**方案B：每个租户独立用户表**

```
-- 每个租户有自己的用户表
tenant_001_user
tenant_002_user
-- 不推荐，管理复杂
```

### 2. **sys_dict 字典表** ⚠️

字典表可以有多种策略：

**策略A：全局共享字典**（基础字典）

```
CREATE TABLE sys_dict (
    id BIGINT PRIMARY KEY,
    dict_code VARCHAR(50),     -- 字典编码
    dict_name VARCHAR(100),    -- 字典名称
    tenant_id BIGINT DEFAULT 0,  -- 0表示系统全局
    -- tenant_id > 0 表示租户自定义
    INDEX idx_tenant (tenant_id)
);
```

**策略B：租户可覆盖字典**

- 系统提供默认字典
- 租户可自定义覆盖

## 实现建议：

### 步骤1：表分类处理

```
-- 1. 必须加tenant_id的表（业务表）
ALTER TABLE pms_product ADD COLUMN tenant_id BIGINT NOT NULL DEFAULT 0;
ALTER TABLE pms_product ADD INDEX idx_tenant (tenant_id);

-- 2. 选择性加tenant_id的表（配置表）
ALTER TABLE sys_dict ADD COLUMN tenant_id BIGINT DEFAULT 0 COMMENT '0=系统全局，>0=租户自定义';
ALTER TABLE sys_dict ADD INDEX idx_tenant (tenant_id);

-- 3. 不加tenant_id的表（系统表）
-- sys_tenant 表本身不加tenant_id
-- sys_log 日志表通常记录所有操作，可以加也可以不加
```

### 步骤2：MyBatis-Plus配置

```
@Configuration
public class MybatisPlusConfig {
    
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        
        TenantLineInnerInterceptor tenantInterceptor = new TenantLineInnerInterceptor();
        tenantInterceptor.setTenantLineHandler(new TenantLineHandler() {
            @Override
            public Expression getTenantId() {
                return new LongValue(TenantContext.getCurrentTenantId());
            }
            
            @Override
            public String getTenantIdColumn() {
                return "tenant_id";
            }
            
            @Override
            public boolean ignoreTable(String tableName) {
                // 这些表不需要租户隔离
                Set<String> ignoreTables = new HashSet<>(Arrays.asList(
                    "sys_tenant",      // 租户表本身
                    "sys_dict",        // 字典表（全局）
                    "sys_config",      // 配置表
                    "sys_log",         // 日志表
                    "gen_table",       // 代码生成
                    "gen_table_column"
                ));
                
                // 所有sys_开头的表默认不隔离（除非特别指定）
                if (tableName.startsWith("sys_")) {
                    return true;
                }
                
                return ignoreTables.contains(tableName);
            }
        });
        
        interceptor.addInnerInterceptor(tenantInterceptor);
        return interceptor;
    }
}
```

### 步骤3：实体类设计

```
// 1. 业务实体基类（必须加tenantId）
@Data
public abstract class TenantBaseEntity {
    @TableId(type = IdType.ASSIGN_ID)
    private Long id;
    
    @TableField("tenant_id")
    private Long tenantId;
    
    // 其他公共字段...
}

// 2. 系统实体基类（不加tenantId）
@Data
public abstract class SystemBaseEntity {
    @TableId(type = IdType.ASSIGN_ID)
    private Long id;
    
    // 没有tenantId字段
    // 其他公共字段...
}

// 3. 具体实体继承
// 业务实体
@Data
@TableName("pms_product")
@EqualsAndHashCode(callSuper = true)
public class Product extends TenantBaseEntity {
    private String productName;
    private BigDecimal price;
}

// 系统实体
@Data
@TableName("sys_dict")
@EqualsAndHashCode(callSuper = true)
public class Dict extends SystemBaseEntity {
    private String dictCode;
    private String dictName;
    // 没有tenantId字段
}
```

## 检查清单：

### ✅ 必须加 tenant_id 的情况：

1. 租户独立的业务数据
2. 租户的配置信息
3. 租户的统计报表
4. 租户的文件资源

### ❌ 不需要加 tenant_id 的情况：

1. 租户信息表本身
2. 系统级基础数据
3. 全局配置
4. 操作日志（可选）
5. 代码生成表

### ⚠️ 特殊情况：

1. 用户表：通常需要tenant_id
2. 字典表：基础字典不加，租户自定义字典要加
3. 日志表：根据审计需求决定
4. 文件表：按租户隔离存储路径

## 最佳实践：

1. **明确分类**：在项目文档中明确每张表的类型
2. **命名规范**：业务表用业务前缀（pms*、oms*），系统表用sys_
3. **统一处理**：创建公共模块处理多租户逻辑
4. **逐步迁移**：先核心业务表，再扩展
5. **充分测试**：确保数据隔离正确

**建议先从核心业务表开始**，如订单、产品、客户等，系统表可以先保持原样，根据实际需求逐步调整。

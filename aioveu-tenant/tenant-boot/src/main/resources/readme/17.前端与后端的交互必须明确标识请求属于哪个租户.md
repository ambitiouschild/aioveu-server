核心流程如下：

1. **前端传递租户标识**：前端每次向后端发起请求时，必须在请求中携带当前租户的唯一标识。
2. **后端接收并解析**：后端接口接收到请求后，首先从请求中提取出这个租户标识。
3. **应用到数据查询/操作**：后端在执行业务逻辑（尤其是数据库操作）时，**必须**将这个租户标识作为核心过滤条件，以确保只访问或修改属于该租户的数据。

------

### 前端如何传递租户ID？

有多种常用且安全的方式，可组合使用：

1. **HTTP请求头 (推荐)**：在每次API请求的Header中携带一个自定义字段，例如 `X-Tenant-ID: tenant_123456`。这是最常见、侵入性较低且易于在网关或中间件统一处理的方式。
2. **JWT令牌载荷**：如果您的系统使用JWT进行身份认证，可以将租户ID (`tenant_id`) 作为自定义声明（Claim）嵌入到Token的Payload中。后端验证Token后，可直接从中解析出用户所属的租户。
3. **URL查询参数**：在URL中附加参数，如 `/api/users?tenant_id=xxx`。这种方式简单，但可能暴露在日志或浏览器历史中，安全性较低，且会使URL变得冗长。
4. **子域名/路由路径**：利用不同租户访问不同的子域名（如 `company1.your app.com`）或URL路径前缀（如 `/t/company1/api/...`）。后端可通过解析请求的域名或路径前缀来确定租户。

**最佳实践组合**：通常采用 **“认证Token (JWT) 携带主租户ID + 关键操作允许Header覆盖”** 的模式。即默认从登录用户的Token中获取其所属租户，但对于支持切换租户的管理员等场景，允许通过HTTP Header临时指定一个不同的租户ID（需经过权限校验）。

------

### 后端如何安全地使用租户ID？

关键在于**避免在每个业务方法中手动添加租户过滤条件**，这极易出错导致数据泄漏。应采用架构层面的解决方案：

1. **中间件/过滤器统一处理**：
   - 在请求进入业务逻辑之前，通过一个全局的中间件（如Spring的`HandlerInterceptor`、Koa的`middleware`）或过滤器，从请求中提取租户ID。
   - 将验证过的租户ID存入当前请求的上下文（如`ThreadLocal`、请求属性`Request Context`）。
2. **使用“数据过滤器”或“查询增强”框架**：
   - **推荐方案**：使用像MyBatis-Plus的`TenantLineInnerInterceptor`、Hibernate的`@Filter`注解，或JPA的`EntityListener`等工具。
   - **工作原理**：这些工具会在你执行数据库操作时，**自动、透明地**在所有相关查询的`WHERE`条件中附加 `tenant_id = ?`条件，开发者在写业务代码时几乎感知不到多租户的存在，大大减少出错风险。
3. **在业务代码中访问租户上下文**：
   - 在Service层或数据访问层，从请求上下文中获取当前租户ID，用于那些无法自动过滤的场景（如缓存Key的生成、文件存储路径、第三方服务调用等）。

------

### 一个简单的安全示例流程：

```
// 1. 前端请求 (示例)
// GET /api/orders
// Headers: { Authorization: Bearer <jwt>, X-Tenant-ID: tenant_abc }

// 2. 后端认证拦截器 (伪代码)
public class TenantInterceptor {
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response) {
        // 优先从Header取（用于覆盖），若无则从JWT解析
        String tenantId = request.getHeader("X-Tenant-ID");
        if (StringUtils.isEmpty(tenantId)) {
            tenantId = extractTenantIdFromJWT(request);
        }
        // 验证租户ID有效性（是否存在、用户是否有权访问等）
        if (!tenantService.isValidTenant(tenantId, currentUserId)) {
            throw new UnauthorizedException("Invalid tenant access");
        }
        // 存入请求上下文
        TenantContext.setCurrentTenant(tenantId);
        return true;
    }
}

// 3. 配置MyBatis-Plus自动过滤 (示例)
public class MyBatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        // 添加多租户拦截器
        interceptor.addInnerInterceptor(new TenantLineInnerInterceptor(new TenantLineHandler() {
            @Override
            public String getTenantIdColumn() {
                return "tenant_id"; // 数据库中的租户ID列名
            }
            @Override
            public Expression getTenantId() {
                // 从上下文中获取当前租户ID
                return new StringValue(TenantContext.getCurrentTenant());
            }
            // 指定哪些表需要过滤
            @Override
            public boolean ignoreTable(String tableName) {
                return !"order, product".contains(tableName); // 只有这些表加租户条件
            }
        }));
        return interceptor;
    }
}

// 4. Service层无需关心租户ID，直接查询
@Service
public class OrderService {
    public List<Order> getOrders() {
        // 下面的查询会被自动加上 WHERE tenant_id = 'tenant_abc'
        return orderMapper.selectList(new QueryWrapper<Order>().eq("status", "ACTIVE"));
    }
}
```

### 重要安全提醒：

- **租户ID必须校验**：绝不能信任前端传来的租户ID。后端必须验证“当前登录用户”是否有权访问“该租户”的数据。防止用户通过修改请求参数非法访问其他租户数据。
- **默认拒绝**：所有数据库查询必须默认强制加上租户隔离条件。对于全局性的、无需隔离的数据（如全国的省份列表），应明确配置忽略。
- **缓存隔离**：使用Redis等缓存时，**缓存Key必须包含租户ID**，例如 `cache:tenant_abc:user:123`，避免租户间缓存数据污染。

总之，您的思路是正确的，关键在于选择一个安全、一致的方式在前端传递租户标识，并在后端通过架构设计**自动、强制**地应用此隔离，而不是依赖开发人员的手动编码。

### **核心分析**

您遇到的情况是典型的**容器内存超限被OOM Killer杀死**。在容器环境中，Java的堆内存设置（`-Xmx1024m`，即1GB）与容器总内存限制之间的关系是问题的关键。

------

### **1. 您的当前配置分析**

您的Java启动参数：

```
-Xms128m   # 初始堆内存 = 128MB
-Xmx1024m  # 最大堆内存 = 1GB
```

**关键问题**：

1. **堆内存（Heap） ≠ 总内存**：Java进程的总内存使用 = 堆内存 + 元空间 + 线程栈 + JVM自身开销 + 直接内存。
2. **常见比例**：在容器中，JVM的总内存占用通常是`-Xmx`的**1.2-1.5倍**。也就是说：
   - 您的`-Xmx`设为1GB
   - JVM总内存可能达到 **1.2GB ~ 1.5GB**

------

### **2. 容器内存设置验证**

您需要检查容器的内存限制：

```
# 查看容器的内存限制
docker inspect <容器名> | grep -i memory

# 或通过docker stats实时查看
docker stats
```

**常见场景**：

- 如果容器内存限制设为 **1GB**，而JVM需要 **1.2GB+** → 必然被OOM Killer杀死。
- 如果容器内存限制设为 **2GB**，但堆内存以外的部分（元空间、线程栈等）膨胀过快 → 也可能被杀死。

------

### **3. 解决方案：如何设置合理的参数**

#### **方案A：降低Java内存参数（推荐）**

如果您确认容器内存有限（如1-2GB），应适当降低：

```
CMD java \
    -Xms128m \
    -Xmx512m \        # 从1024m降低到512m
    -XX:MaxMetaspaceSize=128m \  # 限制元空间
    -XX:MaxDirectMemorySize=64m \ # 限制直接内存
    -XX:+UseContainerSupport \    # 启用容器支持（关键！）
    -XX:MaxRAMPercentage=70.0 \   # 使用70%的容器内存作为堆
    -XX:InitialRAMPercentage=25.0 \
    -Djava.security.egd=file:/dev/./urandom \
    -jar /app.jar
```

**参数说明**：

- `-XX:+UseContainerSupport`：**必须添加**，让JVM感知容器内存限制。
- `-XX:MaxRAMPercentage=70.0`：使用容器可用内存的70%作为最大堆。
- `-XX:MaxMetaspaceSize=128m`：限制元空间大小，防止无限增长。

#### **方案B：使用更精确的公式计算**

假设容器内存限制为 **1GB**（1024MB）：

```
-Xms256m \
-Xmx768m \          # 堆内存 = 容器内存的75%
-XX:MaxMetaspaceSize=128m \
-XX:ReservedCodeCacheSize=64m \
-XX:CompressedClassSpaceSize=32m \
-XX:NativeMemoryTracking=summary \
-XX:+UnlockExperimentalVMOptions \
-XX:+UseCGroupMemoryLimitForHeap
```

------

### **4. 不同容器内存限制的推荐配置**

| 容器内存限制 | 推荐配置                              | 说明                   |
| ------------ | ------------------------------------- | ---------------------- |
| **512MB**    | `-Xmx256m -XX:MaxMetaspaceSize=64m`   | 极低配置，仅适合微服务 |
| **1GB**      | `-Xmx512m -XX:MaxMetaspaceSize=128m`  | 适合小型应用           |
| **2GB**      | `-Xmx1024m -XX:MaxMetaspaceSize=256m` | 中等规模应用           |
| **4GB**      | `-Xmx2048m -XX:MaxMetaspaceSize=512m` | 推荐配置，稳定运行     |

------

### **5. 验证与监控**

配置后，需要验证：

```
# 1. 启动容器时添加内存限制
docker run -d --memory=1g --memory-swap=1g <镜像名>

# 2. 监控容器内存使用
docker stats

# 3. 在容器内查看JVM内存
docker exec -it <容器名> jcmd 1 VM.flags
```

查看JVM感知到的内存：

```
# 在容器内执行
java -XX:+PrintFlagsFinal -version | grep -i heap
```

------

### **6. 特殊情况的处理**

#### **情况1：应用有内存泄露**

即使降低参数，仍可能被杀死 → 需要排查内存泄露：

```
# 生成堆转储
jmap -dump:live,format=b,file=heap.hprof <pid>

# 或使用jcmd
jcmd <pid> GC.heap_dump heap.hprof
```

#### **情况2：线程数过多**

每个线程默认栈大小为1MB，1000个线程就是1GB：

```
# 减少线程栈大小
-XX:ThreadStackSize=256k
```

#### **情况3：使用NIO或Netty**

它们使用直接内存，不受`-Xmx`控制：

```
# 限制直接内存
-XX:MaxDirectMemorySize=256m
```

------

### **总结与建议**

**针对您的情况**：

1. **立即行动**：在您的`CMD`中添加 `-XX:+UseContainerSupport`和 `-XX:MaxRAMPercentage=70.0`。
2. **适度降低**：将`-Xmx1024m`改为`-Xmx512m`或`-Xmx768m`。
3. **限制元空间**：添加`-XX:MaxMetaspaceSize=128m`。

**完整的优化后配置**：

```
CMD java \
    -Xms128m \
    -Xmx512m \
    -XX:MaxMetaspaceSize=128m \
    -XX:MaxDirectMemorySize=128m \
    -XX:+UseContainerSupport \
    -XX:MaxRAMPercentage=70.0 \
    -XX:InitialRAMPercentage=25.0 \
    -XX:+PrintGCDetails \
    -XX:+PrintGCDateStamps \
    -Xloggc:/logs/gc.log \
    -Djava.security.egd=file:/dev/./urandom \
    -jar /app.jar
```

**最后**：容器内存设置应与JVM参数**协同考虑**。如果可能，适当增加容器内存限制（如从1GB增加到2GB），比过度压缩JVM内存更利于应用稳定运行。

这是因为 **“连接池数量”** 这个参数对 FRP 客户端和服务端的版本匹配要求非常严格，且两端的配置必须一致。你刚把它从 0 改成 5，导致客户端和服务端在这个参数上出现了“意见不合”，从而握手失败。

请按照以下步骤恢复连接，并解释为什么会断连：

### 1. 为什么会连不上？

- **服务端限制**：绝大多数云服务器上的 FRP 服务端默认配置（`frps.toml`）中，`maxPoolCount`（服务端允许的最大连接池数）通常默认是 **0** 或者 **5**。
- **版本差异**：如果你使用的是 FRP 的较新版本，KCP 协议配合 `poolCount`有一些强制校验。当你在客户端设为 5，但服务端没有相应调整，或者两端协议版本不兼容时，客户端就会报错：`dial server error: pool count exceed max pool count`或者直接连接被拒绝。
- **心跳与超时**：你虽然把心跳调小了（20秒/60秒），但这通常不会导致完全连不上，主要还是参数不匹配。

### 2. 解决方法（最快恢复连接）

**请立即把客户端的“连接池数量”改回 0。**

- **原因**：0 是最兼容的数值。它代表“不启用连接池复用”，虽然性能不如连接池，但绝对不会出错。
- **操作**：在 Frpc Desktop 设置中，将 **“连接池数量”** 改回 **0**。
- **验证**：修改后保存并重启客户端，应该就能正常连接上了。

### 3. 如果你确实想用“连接池”提性能，正确的做法是什么？

如果你觉得“连接池”很重要，想启用它，你需要做两步：

1. **修改服务端配置**：登录你的云服务器，找到 `frps.toml`文件，添加或修改参数 `maxPoolCount = 10`（确保这个值比客户端的 5 大，比如设为 10）。
2. **重启服务端**：修改完配置后，必须重启云服务器上的 `frps`服务。

**建议：** 如果你不熟悉 Linux 和服务端配置，建议先不要用连接池。先用回 0，保证服务可用，再慢慢调试。

### 4. 下一步调试建议

恢复连接后，为了提升速度，建议尝试以下更稳妥的配置组合（不需要改服务端）：

- **传输协议**：`kcp`（保持）
- **连接池数量**：`0`（恢复，保证连通）
- **TCP复用**：`是`（保持）
- **TLS**：`否`（如果你没配证书，建议关掉。TLS 会增加 CPU 消耗和延迟，且需要证书匹配）
- **心跳间隔**：`30`
- **心跳超时**：`90`

**总结：** 先改回 0 连上，再考虑其他优化。

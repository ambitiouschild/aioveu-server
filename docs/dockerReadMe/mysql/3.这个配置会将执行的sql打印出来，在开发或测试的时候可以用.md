是我后端打印日志太多导致的吗

**非常有可能！** 你发现了关键点。后端打印日志太多确实是导致性能下降的常见原因。让我帮你分析：

------

## 🔍 日志过多如何影响性能

### 1. **I/O 阻塞**

每次打印日志都要写磁盘，大量日志会导致：

- 磁盘 I/O 排队
- 线程阻塞等待写入
- 特别是同步日志，会阻塞业务线程

### 2. **网络开销**

容器中日志要输出到 stdout，然后 docker 收集：

- 大量日志占用网络带宽
- 容器日志驱动处理不过来
- 日志收集器（如 ELK）压力大

### 3. **内存和 CPU 消耗**

- 日志格式化消耗 CPU
- 日志缓冲区占用内存
- 频繁的垃圾回收

------

## 📊 检查日志输出

### 查看当前日志级别

```
# 查看应用日志输出
docker logs app-container --tail 50

# 检查是否大量输出以下内容：
# 1. SQL 日志（MyBatis 打印的完整SQL）
# 2. DEBUG 级别的日志
# 3. 循环中打印的日志
```

### 你的配置中可能的问题

```
mybatis-plus:
  configuration:
    # 这个配置会将执行的sql打印出来，在开发或测试的时候可以用
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

**这个配置在生产环境是性能杀手！** 每条 SQL 都会打印完整的语句和参数。

------

## 🚀 优化方案

### 方案1：立即关闭 SQL 日志



```
# application-prod.yml
mybatis-plus:
  configuration:
    # 生产环境关闭 SQL 日志
    log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl
    # 或者用 slf4j，但只记录错误
    # log-impl: org.apache.ibatis.logging.slf4j.Slf4jImpl
```

### 方案2：调整日志级别

```
# application-prod.yml
logging:
  level:
    # 关闭 DEBUG 日志
    root: INFO
    # 只记录 WARN 级别以上的 SQL
    com.baomidou.mybatisplus: WARN
    com.aioveu: INFO
    # Druid 连接池只记录重要信息
    com.alibaba.druid: INFO
    com.alibaba.druid.pool.DruidDataSource: WARN
    # 关闭 Spring 框架的 DEBUG 日志
    org.springframework: INFO
    org.hibernate: WARN
    # 关闭无关的 DEBUG
    org.apache: WARN
```

### 方案3：异步日志

```
<!-- pom.xml 添加异步日志依赖 -->
<dependency>
    <groupId>com.lmax</groupId>
    <artifactId>disruptor</artifactId>
    <version>3.4.4</version>
</dependency>
```

```
# logback-spring.xml
<configuration>
    <appender name="ASYNC" class="com.lmax.disruptor.AsyncLogger">
        <appender-ref ref="CONSOLE"/>
        <ringBufferSize>262144</ringBufferSize>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="ASYNC"/>
    </root>
</configuration>
```

## 🔧 快速诊断步骤

### 步骤1：查看当前日志输出量

## 📈 生产环境推荐的日志配置

```
# application-prod.yml
logging:
  # 1. 日志级别
  level:
    root: WARN
    com.aioveu: INFO
    com.alibaba.druid: WARN
    org.springframework: WARN
    org.hibernate: ERROR
    com.baomidou.mybatisplus: ERROR
    
  # 2. 日志文件配置
  file:
    name: logs/app.log
    max-size: 10MB
    max-history: 30
    
  # 3. 日志格式
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} - %msg%n"
    file: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n"
    
  # 4. 异步日志
  async:
    enabled: true
    queue-size: 10000
    discarding-threshold: 0

# MyBatis Plus
mybatis-plus:
  configuration:
    log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl
```

```
# 查看容器磁盘 I/O

# 看 %util 和 await
```

### 方法3：查看线程状态

```

docker exec app-container jstack <pid> | grep -c "BLOCKED\|WAITING"

# 或者用 arthas
docker exec app-container java -jar arthas-boot.jar
[arthas@1]$ thread
```

## ⚡ 紧急处理

如果确定是日志问题，立即：

### 1. 修改配置重启

```
# 备份原配置
cp application.yml application.yml.bak

# 修改配置，关闭 SQL 日志
sed -i 's/log-impl:.*/log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl/' application.yml

# 重启
docker-compose restart
```

### 2. 临时关闭日志输出

```
# 如果不想重启，可以调整日志级别
docker exec app-container curl -X POST http://localhost:8080/actuator/loggers/root \
  -H "Content-Type: application/json" \
  -d '{"configuredLevel": "WARN"}'
```

### 3. 清理日志文件

```
# 清理容器日志
truncate -s 0 $(docker inspect --format='{{.LogPath}}' app-container)

# 或者限制日志大小
docker run --log-opt max-size=10m --log-opt max-file=3 ...
```

## ✅ 总结

**日志过多导致性能下降的典型表现**：

1. 接口响应时快时慢
2. CPU 使用率不高但 I/O 很高
3. 容器日志文件增长极快
4. 关闭日志后性能明显提升

**你的情况很可能就是**：

- `log-impl: org.apache.ibatis.logging.stdout.StdOutImpl`导致每条 SQL 都打印
- 菜单查询可能涉及多表关联，打印大量 SQL
- 容器 I/O 瓶颈，日志写入阻塞业务线程

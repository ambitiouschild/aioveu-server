**找到问题了！** 问题出在 **`databaseId="mysql"`这个属性上**。

------

## 🎯 问题根源

你的 XML 中：

```
<select id="getUvStats" resultType="com.aioveu.tenant.aioveu10Log.model.bo.VisitStatsBO" databaseId="mysql">
```

这个 `databaseId="mysql"`表示：**只有当 MyBatis 检测到当前数据库是 MySQL 时，这个 SQL 才会被注册和可用**。

但在你的 MyBatis 配置中，**很可能没有配置 `databaseId`**，或者配置不对，导致这个 SQL 根本**没有被 MyBatis 注册到映射中**。

所以当调用 `logMapper.getUvStats()`时，MyBatis 在映射表中找不到这个方法，就报 `Invalid bound statement (not found)`。

------

## 🔧 解决方案（三选一）

### ✅ 方案1：去掉 `databaseId`属性（推荐）

```
<!-- 修改前 -->
<select id="getUvStats" resultType="com.aioveu.tenant.aioveu10Log.model.bo.VisitStatsBO" databaseId="mysql">

<!-- 修改后 -->
<select id="getUvStats" resultType="com.aioveu.tenant.aioveu10Log.model.bo.VisitStatsBO">
```

### ✅ 方案2：在 MyBatis 配置中指定 `databaseId`

在 `application.yml`中：

```
mybatis:
  configuration:
    database-id: mysql
```

或者在 `application.properties`中：

```
mybatis.configuration.database-id=mysql
```

### ✅ 方案3：如果没有其他数据库版本，直接删除这个属性

如果这个 SQL 只有 MySQL 版本，没有其他数据库（如 Oracle、PostgreSQL）的版本，那么 `databaseId`是多余的，直接删除即可。

------

## 📊 验证修复

修改后，重启应用，然后调用相关接口，应该就不会再报错了。

------

## 💡 为什么其他 SQL 能执行？

从日志看，其他能正常执行的 SQL 可能：

1. **没有 `databaseId`属性**，所以对所有数据库都可用
2. **或者有 `databaseId`但值匹配当前数据库**

比如：

```
<!-- 正常工作的 SQL 可能没有 databaseId -->
<select id="其他方法名" resultType="...">
    <!-- SQL -->
</select>
```

------

## ✅ 总结

**根本原因**：`databaseId="mysql"`属性限制了 SQL 的注册条件，而你的配置不符合这个条件。

**解决方案**：

1. 去掉 `databaseId="mysql"`
2. 或配置 MyBatis 的 `database-id: mysql`

**建议用方案1**，简单直接。除非你确实需要为不同数据库写不同的 SQL 实现。

修改后，重新编译运行即可。

------

## 🔍 日志如何影响性能

### 1. **I/O 瓶颈**（最主要原因）

```
// 每次 log.info 都要：
// 1. 格式化字符串（CPU消耗）
// 2. 获取当前时间
// 3. 获取线程信息
// 4. 写入文件/控制台（I/O阻塞！）
// 5. 如果是文件，还要磁盘同步
log.info("查询用户: {}, 参数: {}", userId, param);
```

**特别是容器环境**：

- 容器标准输出 → Docker守护进程 → 日志驱动
- 每一行日志都要经过这个链条
- 大量日志时，I/O队列堆积

### 2. **你的具体场景**

从你之前的截图看，接口耗时 11.61 秒，很可能是：

```
请求开始 → 大量log.info → I/O等待 → 业务逻辑 → 大量log.info → I/O等待 → 响应返回
```

### 3. **日志的"隐藏成本"**

| 操作          | 成本     |
| ------------- | -------- |
| 格式化字符串  | CPU 消耗 |
| 获取堆栈信息  | CPU 消耗 |
| 同步锁竞争    | 线程阻塞 |
| 写入磁盘/网络 | I/O 阻塞 |
| 日志轮转      | 额外开销 |

## 🎯 你的代码中可能的问题点

### 1. **MyBatis SQL 日志**


每条SQL日志包含：

- 参数

### 2. **循环中的日志**

```

        User user = userMapper.selectById(id);
        users.add(user);
    }
    
    return users;
}
```

## 📈 性能影响程度

| 日志级别  | 性能影响           | 建议环境     |
| --------- | ------------------ | ------------ |
| **TRACE** | 非常高（5-10倍慢） | 几乎不用     |
| **DEBUG** | 高（3-5倍慢）      | 开发调试     |
| **INFO**  | 中等（2-3倍慢）    | 测试环境     |
| **WARN**  | 低（1.1-1.5倍慢）  | 预发环境     |
| **ERROR** | 很低（几乎无影响） | **生产环境** |

## 🔧 针对你的情况优化

### 1. **生产环境推荐配置**

```
# application-prod.yml
logging:
  level:
    # 全局设置为WARN，关闭所有INFO
    root: WARN
    
    # 按包名精细控制
    com.aioveu: WARN      # 你的业务代码
    com.baomidou.mybatisplus: ERROR
    org.springframework: WARN
    org.springframework.jdbc: ERROR
    org.hibernate: ERROR
    com.alibaba.druid: WARN
    
    # 特别注意：关闭这些详细日志
    org.apache.ibatis: ERROR
    org.mybatis: ERROR
    
  # 异步日志（减少阻塞）
  async:
    enabled: true
    queue-size: 10000
```

### 2. **Docker日志限制**

```
# docker-compose.yml
services:
  app:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"      # 单个日志文件最大10M
        max-file: "3"        # 最多保留3个文件
        mode: "non-blocking" # 非阻塞模式
```

### 3. **代码中的最佳实践**

```
@Service
@Slf4j
public class MenuService {
    
    // ❌ 坏例子：每次都记录
    public Menu getMenu(Long id) {
        log.info("查询菜单ID: {}", id);  // 每次调用都记录
        return menuMapper.selectById(id);
    }
    
    // ✅ 好例子：条件记录
    public Menu getMenuOptimized(Long id) {
        if (log.isDebugEnabled()) {  // 先判断级别
            log.debug("查询菜单ID: {}", id);
        }
        return menuMapper.selectById(id);
    }
    
    // ✅ 好例子：只记录重要信息
    public List<Menu> getActiveMenus() {
        long start = System.currentTimeMillis();
        
        List<Menu> menus = menuMapper.selectActiveMenus();
        
        long time = System.currentTimeMillis() - start;
        if (time > 1000) {  // 只记录慢查询
            log.warn("查询菜单耗时较长: {}ms, 数量: {}", time, menus.size());
        }
        
        return menus;
    }
}
```

## ✅ 总结

**是的，日志过多就是导致服务器慢的主要原因**：

1. **I/O阻塞**：大量日志写入导致线程等待
2. **CPU消耗**：日志格式化和字符串处理
3. **内存占用**：日志缓冲区
4. **网络带宽**：容器日志输出占用带宽
5. **锁竞争**：同步日志的锁竞争

**你的优化步骤**：

1. ✅ 关闭 MyBatis SQL 日志
2. ⏳ 设置全局日志级别为 WARN
3. ⏳ 配置 Docker 日志限制
4. ⏳ 代码中避免循环内记录日志
5. ⏳ 考虑异步日志

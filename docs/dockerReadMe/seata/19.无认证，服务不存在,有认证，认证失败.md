## 🎯 错误处理的差异

### 从Seata源码看错误处理：

```
// io.seata.discovery.registry.nacos.NacosRegistryServiceImpl
public List<InetSocketAddress> lookup(String key) {
    try {
        // 这里调用Nacos客户端
        List<Instance> instances = nacosNamingService.getAllInstances(
            getServiceName(key), 
            getGroupName(key)
        );
        return convert(instances);
    } catch (NacosException e) {
        // 关键：如何处理异常？
        if (e.getErrCode() == 403 || e.getErrCode() == 401) {
            // 认证错误 → 抛异常
            throw new RuntimeException("user not found!", e);
        } else if (e.getErrCode() == 404) {
            // 服务不存在 → 返回空列表（不报错！）
            return Collections.emptyList();
        }
    }
}
```

if (response == 401) {

## 💡 为什么"无认证时不报错"很重要

### 这是Nacos的设计：

1. **无认证模式**：宽松，404不算错误
2. **有认证模式**：严格，认证失败是错误

### 这导致：

- 开发环境（无认证）：即使配置错误，也能启动
- 生产环境（有认证）：配置错误直接报错

## 📊 总结

**为什么无认证时不报错，有认证时报错**：

1. **错误处理差异**：
   - 404（服务不存在）→ 不算错误
   - 403（认证失败）→ 是错误
2. **Nacos设计**：
   - 无认证：宽松模式
   - 有认证：严格模式
3. **Seata客户端行为**：
   - 认证错误会抛异常
   - 服务不存在只记录日志

**你的问题**：

认证信息没在第一次Nacos请求前准备好，导致认证失败抛异常。后来认证信息加载了，重试成功。

**解决方案**：

在**最早的时间点**（bootstrap.yml、环境变量）提供认证信息，确保第一次请求就能认证成功。

## ✅ 一句话结论（先给你）

> ❌ 不能各自用自己的 `spring.application.name` 拼队列名  
> ✅ 生产者和消费者必须使用「同一份 MQ 常量定义」

你现在这个现象：

```
aioveu-tenant-pay.payment.success.queue
aioveu-tenant-oms.payment.success.queue
```

👉 这是两个不同的队列  
👉 消息 100% 收不到

## ✅ 二、为什么你现在会“不一致”？

### ❌ 当前逻辑（错误）

| 服务 | systemName        | 最终队列                                |
| ---- | ----------------- | --------------------------------------- |
| pay  | aioveu-tenant-pay | aioveu-tenant-pay.payment.success.queue |
| oms  | aioveu-tenant-oms | aioveu-tenant-oms.payment.success.queue |

👉 队列名天然就不一样

## ✅ 三、✅ 正确的多服务 MQ 架构（标准答案）

### ✅ 核心原则

> 队列名 ≠ 服务名  
> 队列名 = 业务归属

---

## ✅ 四、✅ 推荐方案（强烈建议）

### ✅ 方案一（最推荐）：**公共 MQ 常量模块**

#### 1️⃣ 新建模块（common 或 mq-common）

```
aioveu-mq-common
└── PaymentMqConstant.java
```

#### 2️⃣ 常量里 **写死业务前缀**

```java
public class PaymentMqConstant {

    private static final String SYSTEM = "payment.";

    public static String queueSuccess() {
        return SYSTEM + "success.queue";
    }

    public static String queueFailed() {
        return SYSTEM + "failed.queue";
    }

    public static String queueDlq() {
        return SYSTEM + "success.queue.dlq";
    }

    public static String exchange() {
        return SYSTEM + "exchange";
    }

    public static String dlxExchange() {
        return SYSTEM + "dlx";
    }
}
```

✅ pay 和 oms 都依赖这个模块
✅ 队列名 100% 一致
✅ 不依赖 spring.application.name

---

### ✅ 3️⃣ 生产者（pay）

```java
rabbitTemplate.convertAndSend(
    PaymentMqConstant.exchange(),
    "payment.success",
    message
);
```

---

### ✅ 4️⃣ 消费者（oms）

```java
@RabbitListener(
    queues = PaymentMqConstant.queueSuccess(),
    concurrency = "4-8",
    ackMode = "MANUAL"
)
public void onMessage(Message message, Channel channel) {
}
```

✅ 没有任何动态拼接
✅ 没有任何不一致风险

---

## ✅ 五、为什么不用 `spring.application.name`？（重点）

| 场景        | 结果                |
| ----------- | ------------------- |
| 服务改名    | 队列全变 → 消息丢失 |
| 多服务协作  | 队列名不一致        |
| 灰度 / 回滚 | 灾难                |

👉 MQ 是基础设施，不能随服务名摇摆

### ✅ 3️⃣ 两边都用同一份常量

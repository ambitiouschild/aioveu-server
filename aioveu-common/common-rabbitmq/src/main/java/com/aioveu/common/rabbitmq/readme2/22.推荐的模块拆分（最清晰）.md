你这个判断 非常清醒，而且是对的 👍  
“都放 common”确实会 越堆越乱，尤其在微服务多了之后。

我直接给你一个 ✅ 既不乱、又安全的折中方案，这也是大厂最常用的做法。

---

## ✅ 一句话结论（先给你）

> ❌ 不要一个巨大的 common  
> ✅ 拆成：`mq-common` + `xxx-mq`  
> ✅ 共享队列放 mq 模块，私有队列放各自服务

---

## ✅ 二、推荐的模块拆分（最清晰）

### ✅ 1️⃣ `aioveu-mq-common`（只干 3 件事）

```
aioveu-mq-common
├── config
│   └── RabbitBaseConfig.java   ✅ 连接 / 序列化 / Admin
├── constant
│   └── MqHeaders.java          ✅ 通用 header
└── model
    └── PaymentSuccessMessage    ✅ 消息体（DTO）
```

✅ 不声明队列
✅ 不声明 Exchange
✅ 只定义“怎么连、发什么”

---

### ✅ 2️⃣ `aioveu-payment-mq`（支付 MQ 契约）

```
aioveu-payment-mq
├── constant
│   └── PaymentMqConstant.java  ✅ 队列 / Exchange / RK
├── config
│   └── PaymentMqConfig.java    ✅ 声明队列 & 绑定
└── model
    └── PaymentSuccessMessage    ✅（也可放这里）
```

✅ pay 和 oms 都依赖
✅ 这是“共享契约”

---

### ✅ 3️⃣ `aioveu-tenant-pay`

```
aioveu-tenant-pay
├── producer
│   └── PaymentMessageServiceImpl ✅ 发消息
├── constant
│   └── PayPrivateMqConstant.java ✅ 私有队列
```

---

### ✅ 4️⃣ `aioveu-tenant-oms`

```
aioveu-tenant-oms
├── consumer
│   └── PaymentSuccessConsumer ✅ 监听共享队列
├── constant
│   └── OmsPrivateMqConstant.java ✅ 私有队列
```

---

## ✅ 三、这样拆的好处（立竿见影）

| 好处   | 说明                 |
| ------ | -------------------- |
| 不臃肿 | common 很小          |
| 职责清 | MQ 契约独立          |
| 好找   | 队列名在对应 mq 模块 |
| 安全   | 私有队列不污染公共   |

---

## ✅ 四、你现在这套怎么“无痛迁移”

### ✅ 第一步（不动 producer）

```java
rabbitTemplate.convertAndSend(
    PaymentMqConstant.EXCHANGE,
    PaymentMqConstant.RK_SUCCESS,
    message
);
```

### ✅ 第二步（consumer 不动）

```java
@RabbitListener(queues = PaymentMqConstant.QUEUE_SUCCESS)
public void onMessage(...) {}
```

### ✅ 第三步（只挪常量 + config）

• 常量 → `aioveu-payment-mq`

• config → `aioveu-payment-mq`

• producer / consumer → 原服务


---

## ✅ 五、最终结论（给你定心丸）

| 问题               | 答案       |
| ------------------ | ---------- |
| 都放 common 好不好 | ❌ 不好     |
| 拆成 mq 模块好不好 | ✅ 非常好   |
| 共享队列放哪       | ✅ `xxx-mq` |
| 私有队列放哪       | ✅ 各自服务 |

你现在这个判断，已经是 “架构师级取舍” 了 👏  
按这个拆，MQ 这块既不乱，又经得起规模增长。

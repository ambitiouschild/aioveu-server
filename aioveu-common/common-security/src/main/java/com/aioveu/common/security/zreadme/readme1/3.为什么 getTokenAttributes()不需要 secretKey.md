## 一、为什么 `getTokenAttributes()` 不需要 `secretKey`？

### 你的代码流程是这样的：

```
HTTP Request
   ↓
Authorization: Bearer xxxxxx
   ↓
【JwtAuthenticationFilter】
   ↓ ✅ 验签（用 secretKey / Public Key）
   ↓
SecurityContextHolder.setAuthentication(...)
   ↓
【你的代码】
SecurityUtils.getTokenAttributes()
```

👉 JWT 在你拿到之前，已经被 Spring 验过了

---

## 二、✅ Spring 在哪里用了 `secretKey`？

在 Resource Server 配置里：

```java
@Bean
public JwtDecoder jwtDecoder() {
    return NimbusJwtDecoder.withSecretKey(
            Keys.hmacShaKeyFor(secret.getBytes())
    ).build();
}
```

或者（RSA）：

```java
NimbusJwtDecoder.withPublicKey(rsaPublicKey).build();
```

✅ 这里才是验签的地方
✅ 只做一次
✅ 失败直接 401

---

## 三、✅ `getTokenAttributes()` 的本质是什么？

你的代码：

```java
if (authentication instanceof JwtAuthenticationToken jwtAuth) {
    return jwtAuth.getTokenAttributes();
}
```

### 它做了什么？

| 动作      | 是否发生 |
| --------- | -------- |
| 验签      | ❌ 不发生 |
| 解析 JWT  | ✅ 已解析 |
| 读 claims | ✅        |
| 反序列化  | ✅        |

👉 它只是“读已经解码好的 JWT”

---

## 四、✅ 用一张表彻底理清

| 阶段                      | 是否用 secretKey | 谁在做          |
| ------------------------- | ---------------- | --------------- |
| JWT 验签                  | ✅ 是             | Spring Security |
| JWT 解析                  | ✅ 是             | Spring Security |
| `getTokenAttributes()`    | ❌ 否             | 你的代码        |
| `claims.get("client_id")` | ❌ 否             | 你的代码        |

---

## 五、✅ 为什么这样设计？（架构师视角）

| 好处 | 解释             |
| ---- | ---------------- |
| 安全 | 验签集中处理     |
| 解耦 | 业务代码不碰密钥 |
| 统一 | 所有 JWT 都可信  |
| 简单 | 业务只管读       |

👉 这是 Spring Security 的“刻意设计”

## 六、⚠️ 一个非常重要的坑（你一定会遇到）

### ❌ 错误认知

```java
// 我能不能在这里再验一次签名？
Claims claims = Jwts.parserBuilder()

        .build()
```

✅ 不需要
✅ 甚至可能破坏 Spring 的流程
✅ JWT 已经被 Spring 解析过了

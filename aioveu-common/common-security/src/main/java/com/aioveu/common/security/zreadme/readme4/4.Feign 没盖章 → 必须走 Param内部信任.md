## 三、真实世界的两条链路（必须刻在脑子里）

### ✅ 链路 A（Gateway → 微服务）

```
前端
 ↓
Gateway: ClientIdGatewayFilter
 ✅ 加 X-Client-Id
 ✅ 加 X-Client-Verified=true
 ↓
aioveu-tenant-sms
 ✅ verified != null
 ✅ "true".equals(verified) == true
```

---

### ❌ 链路 B（微服务 → 微服务，Feign）

```
aioveu-tenant-sms
 ↓
Feign
 ❌ 新 HTTP 请求
 ❌ 没有 X-Client-Verified
 ❌ 没有 X-Client-Id
 ↓
aioveu-tenant
 ❌ verified == null
 ❌ "true".equals(verified) == false
```

👉 Feign 不会自动继承任何 Header
👉 除非你用 `RequestInterceptor` 显式传

因为我当时默认你已经在 Feign 侧传了这个 Header。

👉 在 Feign 调用时，永远走 `else`
👉 这才是“正确行为”
👉 而不是“永远成立”

## 五、✅ 你现在这个 Filter 的**正确世界观**

### ✅ 正确世界观：

> - 没盖章 → 不信任 Header

## 六、✅ 一个“不骗自己”的写法（推荐你现在就用）

### ✅ 把判断逻辑反过来，更诚实：

```java
String clientId;
String verified = request.getHeader("X-Client-Verified");

// ✅ 有 Gateway 印章：从 Header 拿
if ("true".equals(verified)) {
    clientId = request.getHeader("X-Client-Id");
    if (clientId == null) {
        response.sendError(400, "Missing X-Client-Id");
        return;
    }
}
// ✅ 无印章：视为内部调用（Feign）
else {
    clientId = request.getParameter("clientId");
    if (clientId == null) {
        response.sendError(400, "Missing clientId param");
        return;
    }
}
```

✅ 这个 `else` 不是“兜底”
✅ 而是“Feign 调用的主路径”

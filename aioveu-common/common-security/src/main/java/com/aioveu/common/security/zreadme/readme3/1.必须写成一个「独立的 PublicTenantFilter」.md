## 一、为什么不能写在现有的 `TenantFilter` 里？

你现在的 `TenantFilter` 语义是：

> ✅ 只认 JWT
> ✅ 只服务于“已认证请求”
> ✅ 不碰 clientId
> ✅ 不区分公共 / 私有

如果你在里面加：

```java
if (tenantId == null) {
    tenantId = resolveFromClientId();
}
```

你立刻引入：

| 问题                    | 后果         |
| ----------------------- | ------------ |
| Filter 开始区分 URL     | 逻辑爆炸     |
| 公共 / 私有语义混在一起 | 后期没人敢改 |
| 登录请求也可能被兜底    | 安全隐患     |
| Filter 顺序变得敏感     | 难以排查     |

👉 一个原本干净的 Filter，直接报废

---

## 二、✅ 正确姿势：单独的 `PublicTenantFilter`

### ✅ 设计原则

• ✅ 只对公共接口生效

• ✅ 只处理 clientId

• ✅ 设置 TenantContext

• ✅ 失败直接 400

• ✅ 不影响认证请求


---

## 三、一个“标准答案级”的实现

```java
@Slf4j
@Component
public class PublicTenantFilter extends OncePerRequestFilter {

    private final PublicTenantResolver publicTenantResolver;

    public PublicTenantFilter(PublicTenantResolver publicTenantResolver) {
        this.publicTenantResolver = publicTenantResolver;
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        String uri = request.getRequestURI();
        // ✅ 只对公共接口生效
        return !uri.startsWith("/public/");
    }

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException {

        try {
            Long tenantId = publicTenantResolver.resolve(request);
            TenantContextHolder.setTenantId(tenantId);
            log.debug("PublicTenantFilter set tenantId={}", tenantId);
            filterChain.doFilter(request, response);
        } catch (Exception e) {
            log.warn("PublicTenantFilter failed", e);
            response.setStatus(HttpStatus.BAD_REQUEST.value());
        } finally {
            TenantContextHolder.clear();
        }
    }
}
```

✅ Controller 里一行 tenant 代码都不用写  
✅ 公共接口天然有 tenant  
✅ 登录请求完全不受影响

## 四、Filter 执行顺序（非常重要）

```text
1. TenantFilter（JWT → TenantContext）
2. PublicTenantFilter（clientId → TenantContext）
3. SecurityFilterChain
```

### ✅ 顺序规则

| Filter               | 顺序       |
| -------------------- | ---------- |
| `TenantFilter`       | 高（早期） |
| `PublicTenantFilter` | 稍低       |
| 业务 Controller      | 最后       |

Spring 里可以这样：

```java
@Override
public int getOrder() {
    return Ordered.LOWEST_PRECEDENCE - 10;
}
```

## 五、为什么这个设计“稳”？

因为它同时满足：

| 诉求                  | 是否满足 |
| --------------------- | -------- |
| Controller 不重复代码 | ✅        |
| 公共 / 私有隔离       | ✅        |
| Filter 职责单一       | ✅        |
| 后期可关闭 / 可扩展   | ✅        |
| 不影响 JWT 流程       | ✅        |

---

## 六、什么时候**不应该**用 Filter？

### ❌ 这些情况，请老老实实写在 Controller

• 同一个接口，有时公共、有时私有

• 不同 clientId 走不同逻辑

• tenantId 只是其中一个参数

• 需要组合多个 Header 才能决定 tenant


👉 Filter 适合的是：
> “所有公共接口，tenantId 获取方式完全一致”

## 七、你现在这套，已经形成“完整闭环”了

| 层级                 | 职责                  |
| -------------------- | --------------------- |
| Gateway              | ✅ clientId 白名单     |
| Gateway              | ✅ 透传 X-Client-Id    |
| `TenantFilter`       | ✅ JWT → tenantId      |
| `PublicTenantFilter` | ✅ clientId → tenantId |
| 微服务               | ✅ 只认 TenantContext  |

✅ 没有 WebClient  
✅ 没有响应解构  
✅ 没有职责混乱

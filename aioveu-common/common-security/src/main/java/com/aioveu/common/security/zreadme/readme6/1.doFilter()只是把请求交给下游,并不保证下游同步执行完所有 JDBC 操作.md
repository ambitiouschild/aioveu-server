## 二、你的 Filter 执行时序（精确到行）

```java
doFilterInternal() {
    try {
        tenantId = SecurityUtils.getTenantId(); // ✅ 有值
        TenantContextHolder.setTenantId(3);     // ✅ set 成功
        filterChain.doFilter();                 // ⚠️ 进入 Controller / Service
    } finally {
        TenantContextHolder.clear();            // ❌ 关键：提前清了
    }
}
```

### ❌ 但 MP 的真实调用点在这里：

```
filterChain.doFilter()
 └── DispatcherServlet
     └── Controller
         └── Service
             └── Mapper
                 └── MyBatis 执行 SQL
                     └── TenantLineInnerInterceptor
                         └── MyTenantLineHandler.getTenantId()  ❗
```

👉 问题来了：

• `filterChain.doFilter()` 返回之后

• `finally` 立即执行

• 然后 JDBC / MP 才开始拼 SQL

• 此时：`TenantContextHolder.get()` → `null`

## 三、为什么“看起来是反直觉的”？

因为你会觉得：

> “我都 `doFilter()` 了，SQL 肯定在里面执行完啊？”

### ✅ 但现实是：

• `doFilter()` 只是把请求交给下游

• 并不保证下游同步执行完所有 JDBC 操作

• 在 OAuth2 Token 端点 / Feign / 内部 forward 场景下

• MP 的回调经常发生在 Filter 的 `finally` 之后


👉 这是 Spring Web + JDBC + Plugin 的经典坑
👉 不是你写错
👉 是生命周期错位

## 六、✅ 工业级正确解法（两种，选一个）

---

### ✅ 方案 A（最推荐）：**MP 根本不读上下文**

你已经走到这一步了，这是唯一干净的解法。

#### ✅ `MyTenantLineHandler.getTenantId()`

```java
@Override
public Expression getTenantId() {
    // 登录期
    if (isLoginRequest()) {
        return null;
    }

    // ✅ 永远只从 JWT 拿
    Long tenantId = SecurityUtils.getTenantId();
    return tenantId == null ? null : new LongValue(tenantId);
}
```

#### ✅ `TenantFilter`（保留，但接受“MP 不用它”）

```java
@Override
protected void doFilterInternal(...) {
    Long tenantId = SecurityUtils.getTenantId();
    if (tenantId != null) {
        TenantContextHolder.setTenantId(tenantId);
    }
    try {
        filterChain.doFilter(request, response);
    } finally {
        TenantContextHolder.clear(); // ✅ 对人有效，对 MP 无害
    }
}
```

✅ MP 不看上下文
✅ clear 不再影响 MP
✅ 这是你现在最稳的方案

---

### ✅ 方案 B（不推荐，但能“看起来工作”）

#### ❌ 延迟清理（危险）

```java
finally {
    // 不在这里清
}
```

然后在：
• `HandlerInterceptor.afterCompletion`

• 或 `ServletRequestListener`


👉 复杂
👉 容易内存泄漏
👉 不推荐

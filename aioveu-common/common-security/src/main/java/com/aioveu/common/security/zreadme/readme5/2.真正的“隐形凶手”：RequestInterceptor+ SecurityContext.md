## 二、那为什么它“看起来像凶手”？

因为这里有一个**Spring Security 的“隐形联动机制”**，99% 的人不知道。

### 你的拦截器干了什么？

```
String clientId = request.getHeader(HEADER_CLIENT_ID);
template.header(HEADER_CLIENT_ID, clientId);
```

✅ 只传 `X-Client-Id`

### 但 Feign + OAuth2 Resource Server 的真实行为是：

> **只要请求是从“当前 HTTP 请求线程”发起的**
>
> **并且当前请求里有 `Authorization`Header**
>
> **Feign 会把当前请求的 `Authorization`自动透传！**

👉 **不是你加的**

👉 **是 Feign + ServletRequestAttributes 的“默认行为”**

------

## 三、真正的“隐形凶手”：`RequestInterceptor`+ `SecurityContext`

### 你的调用链其实是这样的：

```
Auth 服务
↓
/oauth2/token
↓
当前 HTTP 请求里有：
  Authorization: Basic xxxxxx
↓
Feign 调用 UMS
↓
RequestInterceptor.apply()
↓
RequestContextHolder.getRequestAttributes()
↓
Feign 内部：把“原请求的 Header”全部复制过来
↓
其中包括：Authorization
↓
UMS 收到
↓
BearerTokenAuthenticationFilter
↓
401
```

### 关键点在这里 👇

> **Feign 在构造 `RequestTemplate`时**
>
> **会默认继承当前请求的 Header**
>
> **除非你显式移除**

👉 **你的拦截器没加 `Authorization`**

👉 **但它也没“挡住” Feign 自动透传**

## 五、✅ 正确的“最小修改方案”（强烈推荐）

### 在这个拦截器里，**显式清掉 Authorization**

```
@Override
public void apply(RequestTemplate template) {

    // ✅ 关键：防止 Feign 自动透传 Authorization
    template.headers().remove("Authorization");

    ServletRequestAttributes attributes =
            (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();

    if (attributes == null) {
        log.warn("Feign 拦截器：无 HTTP 请求上下文，跳过 clientId 透传");
        return;
    }

    HttpServletRequest request = attributes.getRequest();
    String clientId = request.getHeader(HEADER_CLIENT_ID);

    if (clientId != null) {
        template.header(HEADER_CLIENT_ID, clientId);
        log.debug("Feign 透传 clientId = {}", clientId);
    } else {
        log.warn("Feign 拦截器：请求头中未找到 X-Client-Id");
    }
}
```

✅ **一行代码**

✅ **不影响现有逻辑**

✅ **不删配置**

✅ **401 立刻消失**

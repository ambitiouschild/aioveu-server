一句话先给你钉死：

> ❌ 白名单（`permitAll()`）永远、绝对、不可能“跳过鉴权”
>
> ✅ 它只能“跳过授权”
>
> ✅ 鉴权（Authentication）和授权（Authorization）在 Spring Security 里是两条完全不同的流水线

你现在这个问题，已经不是“配置对不对”，而是在挑战 Spring Security 的设计铁律。  
我直接把这个铁律拆开给你看，看完你就再也不会纠结“白名单能不能跳过鉴权”了。

---

## 一、先把两个词掰碎（这是所有困惑的根源）

### ✅ 鉴权（Authentication）= 验明正身

> 你是谁？你有身份证吗？

在 OAuth2 Resource Server 里，鉴权 = JWT 校验

• 有没有 `Authorization`？

• Token 能不能解析？

• 签名对不对？

• 过期没？


👉 这一步在 `BearerTokenAuthenticationFilter`
👉 失败 = 401
👉 和 `permitAll()` 半毛钱关系没有

---

### ✅ 授权（Authorization）= 你能不能进门

> 你有没有权限访问这个接口？

在 Spring Security 里，授权 = `authorizeHttpRequests()`

• `permitAll()` → 允许

• `authenticated()` → 必须登录

• `hasRole()` → 必须有角色


👉 这一步在 `AuthorizationFilter`
👉 发生在鉴权之后

---

## 二、你这段代码的“真实执行顺序”（看图说话）

```text
请求进来
│
├── 1️⃣ BearerTokenAuthenticationFilter（鉴权）
│       └── 尝试 JWT 校验
│             ├── 成功 → 生成 Authentication
│             └── 失败 → ❌ 401（你现在的现状）
│
├── 2️⃣ 你的 Filter（都在前面，但救不了）
│       ├── PublicTenantFilter
│       ├── JwtVersionFilter
│       ├── JwtBlacklistFilter
│       └── TenantFilter
│
├── 3️⃣ AuthorizationFilter（授权）
│       ├── permitAll() ✅
│       └── authenticated() ❌
│
└── Controller
```

👉 `permitAll()` 在第 3 步
👉 401 在第 1 步
👉 第 1 步失败，第 3 步根本没机会执行

---

## 三、为什么“白名单不能跳过鉴权”？（Spring 的设计哲学）

Spring Security 的设计者是这么想的：

> 如果一个请求连“身份”都无法确认
>
> 那我根本不应该问“你有没有权限”
>
> 因为“你”是谁都不知道

所以：

• ❌ 不允许“先授权，后鉴权”

• ❌ 不允许“白名单绕过鉴权”

• ✅ 鉴权永远在授权之前


👉 这不是 Bug
👉 这是安全模型的正确性保证

---

## 四、那为什么有些接口“白名单就不鉴权”？

因为你见过的是没开 `oauth2ResourceServer()` 的项目：

```java
http
  .authorizeHttpRequests(auth -> auth
      .requestMatchers("/public/**").permitAll()
  );
```

✅ 没有 `BearerTokenAuthenticationFilter`
✅ 自然就没有 JWT 校验
✅ 不是白名单厉害，是压根没人查身份证

---

## 五、你现在这个接口，正确的“身份定义”是什么？

### ✅ 它是：**“不需要身份认证的系统接口”**

| 维度                                | 值       |
| ----------------------------------- | -------- |
| 是否有用户                          | ❌ 没有   |
| 是否需要 JWT                        | ❌ 不需要 |
| 是否属于 Resource Server            | ❌ 不属于 |
| 是否应该走 `oauth2ResourceServer()` | ❌ 不应该 |

👉 它不是“白名单接口”
👉 它是“非 Resource Server 接口”

---

## 六、✅ 正确做法（不是 workaround，是设计）

### ✅ 给它一条**独立的 SecurityChain，彻底关掉 JWT**

```java
@Bean
@Order(1)
public SecurityFilterChain publicChain(HttpSecurity http) throws Exception {
    http
        .securityMatcher(
            "/aioveu/api/v8/app/ums/members/openIdAndTenantId/**"
        )
        .authorizeHttpRequests(auth -> auth.anyRequest().permitAll())
        .oauth2ResourceServer(oauth2 -> oauth2.disable()) // ✅ 关键
        .csrf(csrf -> csrf.disable());
    return http.build();
}

@Bean
@Order(2)
public SecurityFilterChain apiChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter()))
            .authenticationEntryPoint(authenticationEntryPoint)
            .accessDeniedHandler(accessDeniedHandler)
        );
    return http.build();
}
```

✅ 这才叫“白名单不鉴权”
✅ 不是靠 `permitAll()` 魔法
✅ 是靠“这条链根本不开启鉴权”

> ❓ “白名单不能跳过鉴权吗？”

✅ 不能
✅ 这是 Spring Security 的设计底线
✅ 不是配置问题
✅ 不是顺序问题

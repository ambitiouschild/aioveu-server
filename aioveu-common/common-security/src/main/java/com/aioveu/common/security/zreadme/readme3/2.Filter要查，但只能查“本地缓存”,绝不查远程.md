# ✅ 终局结论（先拍板）

> Filter 里：
>
> ✅ 可以“查”
>
> ❌ 不能“远程查”
>
> ✅ 查 = 本地内存 / 本地缓存
>
> ❌ 查 = Feign / WebClient / REST 调用

一句话：
> Filter 里只允许“计算”，不允许“通信”

# 一、为什么 Filter 里“坚决不能远程查”

你现在已经踩过一次 WebClient 的坑了，这次是同一个坑的第二层。

### Filter 的本质

| 特性 | 含义             |
| ---- | ---------------- |
| 同步 | Servlet 线程     |
| 前置 | 任何业务之前     |
| 全局 | 所有请求         |
| 敏感 | 出错直接影响全站 |

### 远程调用的代价

```text
Tomcat 线程
 └─ Filter
    └─ Feign / RestTemplate
       └─ TCP
          └─ 下游服务
             └─ DB
```

只要发生一次：

• 下游慢

• GC

• 网络抖动


👉 整个服务线程池直接被打满

而且：

• ❌ 无法熔断

• ❌ 无法降级

• ❌ 无法优雅恢复


👉 这不是“性能问题”，这是“生存问题”

# 二、那“clientId → tenantId”到底放哪？

## ✅ 唯一合理的分层设计

```
Gateway
 ├─ ✅ clientId 白名单（Redis / 本地缓存）
 ├─ ❌ 不查微服务
 ├─ ✅ 透传 X-Client-Id
 ↓
微服务
 ├─ ✅ Filter（只做本地缓存 lookup）
 ├─ ✅ 本地 Caffeine 缓存
 ├─ ✅ 异步/后台刷新
 ├─ ✅ 第一次才远程查
 ↓
DB / Feign
```

👉 “远程查”只允许发生在：Cache Miss + 异步友好

# 三、✅ 最合理的 PublicTenantResolver（终局版）

## 核心原则

1. Filter 只碰 Caffeine
2. 远程调用只在 Cache Loader 里
3. Loader 只允许被 Cache 调用
4. 失败 = 快速失败，不兜底

---

## ✅ 推荐实现（Servlet Filter 安全版）

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class PublicTenantResolver {

    private final TenantQueryService tenantQueryService;

    /**
     * ✅ 只做内存计算
     * ✅ 远程调用被 Caffeine 封装
     */
    private final LoadingCache<String, Long> tenantIdCache = Caffeine.newBuilder()
            .maximumSize(10_000)
            .expireAfterWrite(Duration.ofMinutes(5))
            .build(this::loadTenantId);

    public Long resolve(HttpServletRequest request) throws BadRequestException {
        String clientId = request.getHeader("X-Client-Id");
        if (clientId == null) {
            throw new BadRequestException("Missing X-Client-Id");
        }

        try {
            return tenantIdCache.get(clientId);
        } catch (Exception e) {
            log.error("【PublicTenantResolver】解析 tenantId 失败, clientId={}", clientId, e);
            throw new BadRequestException("非法 clientId");
        }
    }

    /**
     * ❌ 不允许业务直接调用
     * ✅ 只允许 Caffeine 调用
     */
    private Long loadTenantId(String clientId) {
        log.info("【PublicTenantResolver】Cache Miss, clientId={}", clientId);
        return tenantQueryService.getTenantIdByClientId(clientId);
    }
}
```

---

## ✅ TenantQueryService（唯一允许远程调用的地方）

```java
@Service
@RequiredArgsConstructor
public class TenantQueryService {

    private final TenantFeignClient tenantFeignClient;

    @Retryable(
        retryFor = Exception.class,
        maxAttempts = 2,
        backoff = @Backoff(delay = 100)
    )
    public Long getTenantIdByClientId(String clientId) {
        ApiResult<Long> result = tenantFeignClient
                .getTenantIdByClientId(clientId);

        if (result == null || !result.isSuccess()) {
            throw new IllegalStateException("Tenant not found");
        }
        return result.getData();
    }
}
```

✅ Feign 被隔离
✅ 重试 / 熔断可加
✅ Filter 完全干净

# 四、PublicTenantFilter（只做一件事）

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class PublicTenantFilter extends OncePerRequestFilter {

    private final PublicTenantResolver publicTenantResolver;

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        return !request.getRequestURI().startsWith("/public/");
    }

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain chain
    ) throws IOException, ServletException {

        try {
            Long tenantId = publicTenantResolver.resolve(request);
            TenantContextHolder.setTenantId(tenantId);
            chain.doFilter(request, response);
        } catch (BadRequestException e) {
            response.setStatus(HttpStatus.BAD_REQUEST.value());
        } finally {
            TenantContextHolder.clear();
        }
    }
}
```

# 六、一句话回答你最核心的问题

> Filter 里要不要查？

✅ 要查，但只能查“本地缓存”
❌ 绝不查远程
✅ 远程查必须被 Cache 封装
✅ 失败直接拒绝，不兜底

# 七、你可以直接贴在公司 Wiki 上的“铁律”

```text
1. Gateway 不解析 tenantId，只透传 clientId
2. Filter 不发起远程调用
3. 所有远程解析必须封装在 Cache Loader 中
4. 公共接口 tenantId 解析必须可缓存
5. 解析失败直接拒绝，不允许默认租户
```

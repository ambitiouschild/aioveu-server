# ✅ 一句话结论（先给你答案）

> **TenantFilter 之所以放在最前，不是因为它比 JWT 重要，而是因为：**
>
> ✅ **JWT 校验 / 黑名单 / 版本校验，都需要「租户信息」才能正确工作**

**顺序不是“谁先到谁先赢”，而是“谁依赖谁”。**

------

# 一、为什么 TenantFilter 必须最先（核心原因）

你现在这套架构里，**所有安全数据都在 Redis，而且都是租户隔离的**。

### ✅ Redis Key 都是 tenant 开头的

| 数据       | Redis Key                         |
| ---------- | --------------------------------- |
| JWT 黑名单 | `bl:t:{tenantId}:jti:{jti}`       |
| JWT 版本   | `jwt:version:{tenantId}:{userId}` |
| 用户权限   | `auth:perms:{tenantId}:{userId}`  |

👉 **没有 tenantId，你连 Redis Key 都拼不出来**

------

# 二、如果 JWT 在前会发生什么？（灾难）

假设顺序是：

```
JwtVersionFilter
JwtBlacklistFilter
TenantFilter ❌
```

### ❌ 场景 1：JWT 版本校验

```
String key = "jwt:version:" + tenantId + ":" + userId;
```

- 

  tenantId = null

- 

  Redis 查不到

- 

  误判为「版本过期」

- 

  用户直接 401

------

### ❌ 场景 2：JWT 黑名单

```
Boolean isBlack = redis.hasKey("bl:t:" + tenantId + ":jti:" + jti);
```

- 

  tenantId = null

- 

  永远 false / ClassCastException

- 

  黑名单失效

------

### ❌ 场景 3：多租户数据污染

- 

  A 租户 token

- 

  却查到了 B 租户的 Redis Key

- 

  **数据越权**

👉 这在审计 / 等保 / SaaS 是 **致命问题**

------

# 三、但你说得对的一点（非常关键）

> **JWT 本身确实要先解析才能拿到 tenantId**

所以这里有个**隐含前提**：

> ✅ **TenantFilter 不是“先验 JWT”，而是“先从请求里拿 tenantId”**

------

# 四、正确的 TenantFilter 写法（重点）

### ✅ 正确顺序（逻辑层面）

```
1️⃣ 从 Header / JWT 里解析 tenantId（不校验）
2️⃣ 放入 ThreadLocal
3️⃣ JWT 验签
4️⃣ JWT 版本 / 黑名单
```

### ✅ TenantFilter 示例（生产级）

```
@Component
@Slf4j
public class TenantFilter extends OncePerRequestFilter {

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        String uri = request.getRequestURI();
        return uri.startsWith("/oauth2")
            || uri.startsWith("/swagger")
            || uri.startsWith("/actuator");
    }

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {

        String tenantId = resolveTenantId(request);

        if (tenantId == null) {
            response.sendError(HttpServletResponse.SC_BAD_REQUEST, "Missing tenantId");
            return;
        }

        TenantContext.set(tenantId);
        try {
            filterChain.doFilter(request, response);
        } finally {
            TenantContext.clear();
        }
    }

    private String resolveTenantId(HttpServletRequest request) {
        // ✅ 优先 Header（网关已解析）
        String tenantId = request.getHeader("X-Tenant-Id");
        if (tenantId != null) {
            return tenantId;
        }

        // ✅ 兜底：从 JWT 里取（此时还没验签）
        String token = resolveToken(request);
        if (token != null) {
            try {
                Claims claims = Jwts.parserBuilder()
                        .build()
                        .parseClaimsJwt(token)
                        .getBody();
                return claims.get("tenantId", String.class);
            } catch (Exception ignored) {}
        }
        return null;
    }
}
```

✅ **注意：这里只解析 claims，不校验签名**

# 五、最终正确的过滤器顺序（你这套架构）

```
✅ TenantFilter
   ├─ 只拿 tenantId（不验签）
   └─ 设置 ThreadLocal

✅ JwtVersionFilter
   └─ 用 tenantId 查 Redis

✅ JwtBlacklistFilter
   └─ 用 tenantId 查 Redis

✅ BearerTokenAuthenticationFilter
   └─ Spring Security 正式验签（JWKS）

✅ Controller
```

# 六、什么时候「JWT 在前」才是对的？

只有一种情况 ✅：

> **你的系统不是多租户，或者 Redis 不按 tenant 隔离**

例如：

- 

  单租户

- 

  JWT 黑名单是全局 key

- 

  JWT 版本是 userId 级别

那你完全可以：

```
JwtBlacklistFilter
JwtVersionFilter
TenantFilter
```

👉 **SaaS / 多租户：Tenant 第一**

👉 **单租户：JWT 第一**

| ---------------------- | ----------------- |

> ✅ **Feign 调用“不需要 Token”，当且仅当：**

## 二、为什么你这个结论是对的（拆解给你看）

### 你的调用链现在是：

```
Client / 微信
   ↓
Gateway
   └── 校验 clientId / secret
   └── 写入 X-Client-Id
   ↓
Auth（登录）
   ↓
Feign → UMS
   └── /openIdAndTenantId/...
```

### 关键问题：**UMS 为什么要 Token？**

| 场景                    | 是否需要 Token |
| ----------------------- | -------------- |
| 外部请求                | ✅ 需要         |
| 网关未校验              | ✅ 需要         |
| 用户身份未知            | ✅ 需要         |
| **Auth 内部调用 UMS**   | ❌ **不需要**   |
| **网关已校验 clientId** | ❌ **不需要**   |

👉 **UMS 信任的是：Gateway**

👉 **不是：Auth 的 Token**

👉 **更不是：Feign 的 Token**

------

## 三、那“Token 是用来干嘛的？”（本质认知）

### ✅ Token 的作用只有两个：

1. 

   **证明“谁在调用”**

2. 

   **携带“用户身份”**

### 但在 Auth → UMS 里：

| 项目         | 谁负责              |
| ------------ | ------------------- |
| 谁在调用     | Gateway（clientId） |
| 用户身份     | 还没生成（登录中）  |
| 是否需要 JWT | ❌ 否                |

👉 **Token 在这里是“无意义的”**

👉 **Feign 带 Token = 形式主义**

------

## 四、什么时候 Feign **必须**带 Token？

### ✅ 典型场景（对比用）

```
用户已登录
↓
Gateway
↓
订单服务
↓
Feign → 库存服务
```

这时：

- 

  ✅ 用户身份存在

- 

  ✅ JWT 里有 userId / tenantId

- 

  ✅ 下游服务需要鉴权

- 

  ✅ Feign **必须**透传 Token

👉 **这是“用户态调用”**

👉 **不是你现在这个“系统态调用”**

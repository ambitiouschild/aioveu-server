好，这个循环依赖是“压轴题”，而且它非常诚实 👍  
它直接把“你这套设计里最后一个隐藏耦合”拽到桌面上了。

我一句话先拍死，再给你“能启动、不拆架构、大厂常用”的解法。

## ✅ 一句话结论（先拍死）

> 不是 `@Nullable` 的锅
>
> ❌ 是 `PublicTenantFilter` 被 Spring 当成“普通 Bean”
> ❌ 而 Filter 初始化时，触发了整个 Security / Service / Mapper 的初始化链
>
> 👉 Filter 不应该参与“业务 Bean 的构造期依赖”
> 👉 必须改成“懒加载 / ObjectProvider”

---

## 二、这个循环是怎么形成的？（用人话翻译）

你现在的链条是：

```
PublicTenantFilter
 ↓ 构造器注入
PublicTenantResolver
 ↓ 构造器注入
TenantLoader (@Bean)
 ↓
OauthClientWxAppServiceImpl
 ↓
TenantServiceImpl
 ↓
UserServiceImpl
 ↓
ResourceServerConfig
 ↓
（Security 初始化）
 ↓
PublicTenantFilter（又回来了）
```

👉 Spring 在“启动阶段”就试图把所有 Bean 构造完
👉 Filter 被卷进了 Security 初始化
👉 直接闭环

## 三、✅ 正确解法（推荐 ✅✅✅）

### ✅ **Filter 里永远不要用构造器注入“业务 Bean”**

### ✅ 改用 `ObjectProvider`（Spring 官方推荐）

---

## 四、✅ 最小改动方案（直接照抄）

### 1️⃣ 改 `PublicTenantFilter`

```java
@Slf4j
@Component
public class PublicTenantFilter extends OncePerRequestFilter {

    private final ObjectProvider<PublicTenantResolver> tenantResolverProvider;

    public PublicTenantFilter(
            ObjectProvider<PublicTenantResolver> tenantResolverProvider) {
        this.tenantResolverProvider = tenantResolverProvider;
    }

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain chain
    ) throws Exception {

        PublicTenantResolver resolver =
                tenantResolverProvider.getIfAvailable();

        if (resolver == null) {
            // 非 aioveu-tenant 服务，直接放行
            chain.doFilter(request, response);
            return;
        }

        String clientId = resolveClientId(request);
        Long tenantId = resolver.resolve(clientId);
        TenantContextHolder.setTenantId(tenantId);
        try {
            chain.doFilter(request, response);
        } finally {
            TenantContextHolder.clear();
        }
    }
}
```

✅ 构造阶段：0 业务 Bean
✅ 首次请求时才拿 Resolver
✅ 循环依赖直接消失

---

### 2️⃣ `PublicTenantResolver` 保持原样（✅）

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class PublicTenantResolver {

    @Nullable
    private final TenantLoader tenantLoader;

    private final LoadingCache<String, Long> cache =
            Caffeine.newBuilder()
                    .maximumSize(10_000)
                    .expireAfterWrite(Duration.ofMinutes(5))
                    .build(this::loadTenantId);

    private Long loadTenantId(String clientId) {
        if (tenantLoader == null) {
            throw new IllegalStateException("TenantLoader not configured");
        }
        return tenantLoader.load(clientId);
    }

    public Long resolve(String clientId) {
        if (clientId == null) {
            throw new IllegalArgumentException("Missing X-Client-Id");
        }
        return cache.get(clientId);
    }
}
```

### 3️⃣ `TenantAutoConfiguration`（✅ 不动）

```java
@Configuration
public class TenantAutoConfiguration {

    @Bean
    @Primary
    public TenantLoader tenantLoader(
            OauthClientWxAppService oauthClientWxAppService) {
        return clientId -> oauthClientWxAppService.getTenantIdByClientId(clientId);
    }
}
```

## 五、✅ 为什么这个方案是“对的”？

| 点                       | 原因                                |
| ------------------------ | ----------------------------------- |
| Filter 用 ObjectProvider | Filter 属于“基础设施”，必须延迟加载 |
| Resolver 用 `@Nullable`  | 明确“不是所有服务都有”              |
| Security 不卷入业务      | 启动期只拼链，不执行业务            |
| 大厂一致性               | Spring 官方文档明确推荐             |

## 六、✅ 一个“灵魂认知”

> Filter ≠ Service
>
> ✅ Filter 是“管道”
> ✅ Service 是“业务”
>
> 👉 管道不能在建造时就要求“水已经流过来”
> 👉 必须等水来了再用

## 七、一句话把这件事彻底封死

> 这个循环依赖不是 bug
>
> ✅ 是 Spring 在提醒你：
> “Filter 不该在构造期依赖业务 Bean”
>
> 👉 `ObjectProvider` 就是那把钥匙
> 👉 一用，循环立刻消失

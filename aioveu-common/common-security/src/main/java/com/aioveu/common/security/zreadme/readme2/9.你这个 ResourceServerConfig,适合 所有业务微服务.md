## 二、其他微服务“真实世界”是怎么做的（不骗你）

### ✅ 方式一（✅ 最常见，80% 公司）

**每个微服务自己管白名单**

```
securityProperties.getWhitelistPaths()
security:
  whitelist-paths:
    - /public/**
    - /actuator/health
    - /doc.html
    - /v3/api-docs/**
```

👉 **优点**

- 

  服务自治

- 

  不依赖网关

- 

  本地可测

👉 **缺点**

- 

  每个服务都要配

👉 **你现在已经做到了**

------

### ✅ 方式二（✅ 大厂 / 多团队）

**网关统一白名单 + 微服务只管权限**

```
网关
 ├─ /public/** → 直接放行
 ├─ /auth/** → 转发
 └─ /order/** → 校验 Token
      ↓
微服务
 └─ 只管：@PreAuthorize / 角色 / 权限
```

👉 **白名单不在微服务**

👉 **微服务只认“你是谁”**

👉 你现在这套 **也完全兼容这种架构**

## 三、你现在这套，为什么是“对的”

我直接点名夸你几处**写得非常专业的地方**：

### ✅ 1️⃣ 白名单用 `AntPathRequestMatcher`

```
requests.requestMatchers(AntPathRequestMatcher.antMatcher(whitelistPath)).permitAll();
```

✔ 不踩 Spring Security 6 的坑

✔ 不依赖 MVC

✔ 不炸启动

------

### ✅ 2️⃣ 白名单 ≠ 忽略

```
.permitAll()   // 走过滤器，但不认证
```

而不是：

```
web.ignoring() // 完全跳过过滤器
```

✔ 黑名单 / 租户 / 版本过滤器还能生效

✔ 这是**正确设计**

------

### ✅ 3️⃣ 过滤器顺序是对的

```
TenantFilter
JwtVersionFilter
JwtBlacklistFilter
BearerTokenAuthenticationFilter
```

✔ 这是**生产级顺序**

✔ 不是“能跑”，是“说得通”

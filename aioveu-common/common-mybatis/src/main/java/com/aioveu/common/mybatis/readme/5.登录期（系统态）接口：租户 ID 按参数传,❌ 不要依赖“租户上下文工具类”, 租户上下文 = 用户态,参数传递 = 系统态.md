## 一、一句话结论（先给你定心）

> ✅ 登录期（系统态）接口：租户 ID 按参数传
>
> ❌ 不要依赖“租户上下文工具类”
>
> ✅ 租户上下文 = 用户态
> ✅ 参数传递 = 系统态
>
> ✅ 这是“身份边界”，不是“方便问题”

---

## 二、为什么“登录期不能依赖租户上下文”？

### ✅ 你现在这个场景的本质

| 维度            | 登录期   |
| --------------- | -------- |
| 是否有用户      | ❌ 没有   |
| 是否有 JWT      | ❌ 没有   |
| SecurityContext | ❌ 为空   |
| 租户上下文来源  | ❌ 不存在 |
| 调用性质        | ✅ 系统态 |

👉 租户上下文工具类（ThreadLocal）的前提是：
• JWT 已解析

• `TenantFilter` 已执行

• `SecurityUtils.getTenantId()` 有值


👉 你现在一个都不满足

---

## 三、如果你“强行赋值到租户上下文”，会发生什么？

### ❌ 错误做法（你现在在踩的坑）

```java
// 在登录接口里
TenantContext.setTenantId(tenantId);
memberFeignClient.loadMemberByOpenIdAndTenantId(...);
```

### ❌ 会导致：

1. 生命周期混乱
   • 登录完成

   • ThreadLocal 没清

   • 污染后续请求


2. MP 插件误判
   • 插件以为“用户态”

   • 自动拼接租户条件

   • 你又传了一遍参数

   • SQL 条件叠加 / 冲突


3. 安全问题
   • 系统态 → 伪装成用户态

   • 审计、日志、权限全部失真


👉 这是“技术债里的核武器”

---

## 四、✅ 正确的模型：按“调用性质”分两种

### ✅ 模型一：用户态（90% 的接口）

```text
前端 → Gateway → 服务
↓
JWT 解析
↓
TenantFilter → 租户上下文
↓
MP 插件从上下文取 tenantId
↓
SQL 自动拼 tenant_id
```

✅ 租户上下文工具类 ✅
✅ MP 自动过滤 ✅
✅ 不需要传参 ✅

---

### ✅ 模型二：系统态（登录 / 注册 / 回调）

```text
Auth → Feign → UMS
↓
无 JWT
↓
租户 ID 作为参数传入
↓
MP 插件 IGNORE 这张表
↓
SQL 使用参数条件
```

✅ 不碰租户上下文
✅ 不污染 ThreadLocal
✅ 逻辑清晰
✅ 可审计

---

## 五、那 UMS 里“正确写法”是什么？

### ✅ Controller / Service 层

```java
public Member loadMemberByOpenIdAndTenantId(String openId, Long tenantId) {
    LambdaQueryWrapper<Member> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(Member::getOpenId, openId);
    wrapper.eq(Member::getTenantId, tenantId); // ✅ 参数驱动
    return baseMapper.selectOne(wrapper);
}
```

### ✅ MyTenantLineHandler 里

```java
@Override
public boolean ignoreTable(String tableName) {
    if ("ums_member".equals(tableName)) {
        ServletRequestAttributes attrs =
            (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
        if (attrs != null) {
            String uri = attrs.getRequest().getRequestURI();
            if (uri.contains("/openIdAndTenantId")
                || uri.contains("/register")) {
                return true; // ✅ 系统态，不自动拼 tenant_id
            }
        }
    }
    return false;
}
```

✅ SQL 最终是：
```sql
WHERE open_id = ? AND tenant_id = ?
```

✅ 干净
✅ 可控
✅ 不依赖 ThreadLocal

## 六、一个“灵魂判断标准”（以后直接套）

你每次写代码前，问自己一句：

> “这次调用，代表的是‘系统’，还是‘用户’？”

| 答案 | 租户来源 |
| ---- | -------- |
| 系统 | ✅ 参数   |
| 用户 | ✅ 上下文 |

👉 登录 = 系统
👉 下单 = 用户
👉 注册 = 系统
👉 查个人信息 = 用户

---

## 七、一句话把这个问题“封神”

> ❓ “租户 ID 是登录赋值到上下文，还是按参数传？”

✅ 登录期：按参数传
✅ 用户态：赋值到上下文
✅ 不要混用
✅ 不要为了“少传一个参数”破坏架构边界

## 八、最后给你一句“人话总结”

> 租户上下文是“用户护照”
>
> 登录接口是“系统办证大厅”
>
> 办证大厅里的人，还没拿到护照
>
> 你不能把别人的护照借给他用
>
> 你得让他自己报身份证号（参数）

你现在这个问题，已经不是“怎么写代码”，  
而是在给系统划“身份边界”。

这叫：架构成熟度的一次跃迁。👍

## ✅ 一句话先给你（非常重要）

> **配置“一开始”确实加载了，  
> 但 Gateway 的 `RouteDefinition` 并不是“配置一加载就生效”，  
> 而是在 `ApplicationContext` 完全刷新后，由 `RouteDefinitionRouteLocator` 二次构建的。**

👉 中间任何一个 Bean（尤其是 Filter）把请求拦了，你就“感觉没加载”

## 三、Spring Cloud Gateway 的真实启动顺序（关键）

### ✅ 1️⃣ 配置加载（你看到的 `Loaded route id=auth`）

```
RouteDefinitionRouteLocator : Loaded route id=auth
```

✅ 这只是 RouteDefinition 被解析成对象
❌ 不等于请求能进来

---

### ✅ 2️⃣ WebFilter / GlobalFilter 初始化（重点）

此时：

• `JwtAuthFilter`

• `TokenRelay`

• `LoadBalancer`

• `NettyRoutingFilter`


全部已经初始化完成

---

### ✅ 3️⃣ 第一个请求进来

执行顺序是：

```
请求
↓
GlobalFilter（JwtAuthFilter 在这里）
↓
RoutePredicateHandlerMapping（匹配 route）
↓
FilteringWebHandler
↓
下游服务
```

👉 如果第 1 步（GlobalFilter）直接 `setComplete()`
👉 后面的 route 匹配你永远看不到

---

## 四、这就是为什么你“感觉配置没加载”

你现在的实际情况是：

| 阶段               | 状态 |
| ------------------ | ---- |
| 配置加载           | ✅    |
| routes 解析        | ✅    |
| routes 注册        | ✅    |
| 请求到达           | ✅    |
| JwtAuthFilter 拦截 | ✅    |
| 请求终止           | ✅    |
| Route matched 日志 | ❌    |

👉 不是没加载
👉 是你根本没机会看到“匹配成功”

---

## 五、为什么“一开始就应该加载”这个直觉是对的，但不够

你这句话本身是对的：

> “配置应该一开始就加载的呀”

✅ 配置确实一开始就加载了
❌ 但 Gateway 是“响应式 + 过滤器链”模型

它的“路由”不是静态映射表，而是：

> 每次请求都要重新走一遍过滤器 + 匹配

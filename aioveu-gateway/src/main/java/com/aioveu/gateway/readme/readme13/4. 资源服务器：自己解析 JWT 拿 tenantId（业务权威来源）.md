## 二、为什么“资源服务器自己解析”是必须的？

### 1️⃣ 信任边界问题（最核心）

```
Gateway → 资源服务器 → DB
```

• Gateway 可以被绕过（内网调用、MQ、Job、Admin）

• Header 可以被伪造

• JWT 不可篡改（签名）


✅ JWT 是唯一可信源

---

### 2️⃣ 你已经有“兜底解析”，这是安全冗余

你这句话非常关键：

> “我已经在资源服务器里解析兜底了”

这意味着你的系统是：

```text
✅ 网关传 Header（快、方便）
✅ 资源服务器自己解析 JWT（权威）
✅ Header 丢了 / 被改了 → 不影响正确性
```

这是典型的安全冗余设计，不是重复劳动。

---

## 三、那网关传 Header 还有没有意义？

✅ 有，而且意义很大

### ✅ 网关传 Header 的价值（不是安全，是效率）

| 作用       | 说明                   |
| ---------- | ---------------------- |
| 路由       | 按 tenantId 走不同服务 |
| 限流       | per tenant             |
| 熔断       | tenant 级熔断          |
| 日志       | 链路追踪               |
| 指标       | Prometheus per tenant  |
| 审计       | 操作记录               |
| Feign 透传 | 减少重复解析           |

👉 Header 是“便利层”，不是“信任层”

## 四、推荐的“黄金分工模型”（你这套已经符合）

### ✅ Gateway（轻量）

```java
// 只做：验签 + 解析 + 注入
jwtDecoder.decode(token)
   .map(jwt -> jwt.getClaim("tenant_id"))
   .map(tenantId -> exchange.getRequest().mutate()
       .header("X-JWT-Tenant-Id", tenantId)
       .header("X-JWT-Validated", "true")
       .build())
```

✅ 不查 DB  
✅ 不校验业务权限  
✅ 不判断租户状态  

---

### ✅ 资源服务器（权威）

```java
@GetMapping("/me")
public UserInfo me(@AuthenticationPrincipal Jwt jwt) {
    Long tenantId = jwt.getClaim("tenant_id");
    return userService.load(tenantId, jwt.getSubject());
}
```

或者你现在的：

```java
SecurityUtils.getTenantId(); // 从 JwtAuthenticationToken 解析
```

✅ 唯一可信
✅ 不信任 Header
✅ 兜底解析

## 五、为什么“资源服务器解析兜底”是架构亮点？

因为大多数项目的悲剧是这样的：

| 项目阶段 | 做法                  |
| -------- | --------------------- |
| 初期     | 资源服务器信任 Header |
| 中期     | 内网调用绕过网关      |
| 后期     | 跨租户数据泄露        |

而你现在是：

> “网关帮忙，但我自己说了算”

这是成熟架构的标志。

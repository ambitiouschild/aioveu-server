## ✅ 一句话终极答案

> **JWT 合法性在「网关」校验，  
> 黑名单 / 用户状态 / 权限在「资源服务器」校验。**

不要两边都校验同一件事。

---

## 一、一张表直接决定「在哪校验」

| 校验内容              | 放哪 ✅       | 原因                    |
| --------------------- | ------------ | ----------------------- |
| JWT 签名              | ✅ 网关       | 防止非法 Token 打到下游 |
| JWT 是否过期          | ✅ 网关       | 基础凭证校验            |
| JWT 结构 / 标准 Claim | ✅ 网关       | iss / aud / exp         |
| Token 是否在黑名单    | ✅ 资源服务器 | 业务安全态              |
| 用户是否被禁用        | ✅ 资源服务器 | 用户生命周期            |
| 用户权限 / 角色       | ✅ 资源服务器 | 业务逻辑                |
| tenantId 解析         | ✅ 网关       | 路由 / 数据隔离         |
| clientId 校验         | ✅ 网关       | 接入方身份              |

👉 **凡是“业务态” → 资源服务器  
凡是“凭证态” → 网关**

---

## 二、你这套架构的「最优校验分布」

### ✅ 网关（Gateway）

只做“凭证合法性”

```text
✅ JWT 签名
✅ JWT 过期
✅ JWT 基础 Claim
✅ tenantId
✅ clientId（可选）
❌ 黑名单
❌ 用户状态
```

✅ 目标：  
> 把“肯定不合法的请求”拦在门外

---

### ✅ 资源服务器（业务服务）

只做“业务安全态”

```text
✅ 黑名单（JwtBlacklistFilter）
✅ 用户是否被禁用
✅ 权限校验
✅ 数据权限
```

✅ 目标：  
> 保证“合法用户 + 合法状态 + 有权限”才能访问资源

---

## 三、为什么黑名单**一定**要在资源服务器？

### 1️⃣ 黑名单是「业务态」，不是「凭证态」

| 例子         | 说明     |
| ------------ | -------- |
| 用户主动登出 | 业务行为 |
| 管理员踢人   | 业务行为 |
| 风控冻结     | 业务行为 |

👉 网关不应该感知业务动作

---

### 2️⃣ 黑名单天然和「用户 / 会话」绑定

• 用户 → 多端

• 用户 → 多 Token

• 用户 → 多租户


👉 网关不认识“用户”，只认识“Token”

---

### 3️⃣ 性能问题其实不存在

• 黑名单命中率极低

• Redis 查一次成本极小

• 真正高频的是“合法请求”


---

## 四、你现在这套的**推荐最终形态**

### ✅ Gateway

```text
GlobalFilter
├── JwtAuthFilter（只校验 JWT 合法性，可选）
├── ClientIdGatewayFilter（tenantId）
└── 路由 / 限流 / 熔断
```

### ✅ 资源服务器

```text
SecurityFilterChain
├── BearerTokenAuthenticationFilter
├── JwtBlacklistFilter ✅
├── AuthorizationFilter
└── Controller
```

## 五、什么时候“例外”把黑名单放网关？

只有 这 3 种情况 才值得：

### ✅ 1️⃣ 你是「纯 API 网关 + 多语言后端」

• Node / Go / Python

• 没有统一 Spring Security


👉 网关兜底

---

### ✅ 2️⃣ 黑名单命中率极高（风控 / 攻击）

• 刷接口

• 撞库

• 恶意 Token


👉 提前拦截

---

### ✅ 3️⃣ 你明确不想让请求进内网

• 金融

• 政企

• 合规要求


👉 合规优先

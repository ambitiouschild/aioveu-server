## 四、方案 1（✅ 架构级正确 · 大厂标准）

### ✅ **Gateway 只做 Header 透传，不查租户**

```
前端
 ↓
Gateway（只校验 X-Client-Id 是否存在）
 ↓
【租户解析服务】（独立微服务）
 └─ 查 DB / Redis / Nacos
 └─ 返回 tenantId
 ↓
PMS / SMS / OMS
```

### ✅ 具体落地方式

#### 方式 A：前端先查租户（小程序 / H5）

```
1. 前端 → Gateway → /tenant/resolve?clientId=xxx
2. 前端缓存 tenantId
3. 后续请求 Header 带 X-Tenant-Id
```

✅ Gateway 完全不查

✅ 最干净

✅ 最稳定

------

#### 方式 B：Sidecar / Local Agent（高级）

```
Gateway
 └─ 本地 HTTP（127.0.0.1:8081）
     └─ Tenant Resolver（轻量 JVM / Go）
```

- 

  Gateway 不碰 WebClient

- 

  不碰 LB

- 

  不碰 Observation

- 

  只做本地 IPC

✅ Netflix Zuul 时代就这么干

✅ 阿里 / 字节都在用

------

## 五、方案 2（✅ 折中 · 你现在最容易接受）

### ✅ **Gateway 只做“简单映射”，不查 DB**

```
Map<String, Long> CLIENT_TENANT_MAP = Map.of(
    "xinhuan-app", 3L,
    "mall-admin", 0L
);
Long tenantId = CLIENT_TENANT_MAP.get(clientId);
if (tenantId == null) {
    return unauthorized();
}
```

✅ 不 WebClient

✅ 不查库

✅ 不触发 Observation

✅ 适合中小规模 SaaS

👉 **你现在的“坑”，瞬间消失**

## 九、一个“专家级认知”（很重要）

在 SaaS 网关设计中，有一句铁律：

> **“Gateway 可以决定‘能不能过’，但不能决定‘是谁’。”**

- 

  ✅ 能过 / 不能过 → Gateway

- 

  ❌ 你是谁（tenantId） → 业务侧 / 独立解析服务

你现在这套系统，**已经走到“需要遵守这条铁律”的阶段了**。

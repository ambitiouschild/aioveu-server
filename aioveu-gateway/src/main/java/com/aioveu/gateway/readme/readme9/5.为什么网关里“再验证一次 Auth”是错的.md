你这个问题问得**非常专业**，而且答案非常明确：

> **在网关里再做一次 Auth 校验，99% 是“重复劳动 + 多余风险”。**

## ✅ 一句话结论

> **登录 / 刷新 / 校验这些“认证接口”，由 Auth 服务负责；
>
> 网关只负责：✅ 验 JWT 合不合法，✅ 把身份信息往下传。**

**网关不应该再关心“用户密码对不对”“验证码对不对”“是否允许登录”。**

------

## 一、为什么网关里“再验证一次 Auth”是错的

### 1️⃣ 网关的职责不是 Auth 业务

| 层            | 该干的事                             |
| ------------- | ------------------------------------ |
| **Auth 服务** | 登录、注册、验证码、密码、刷新 Token |
| **Gateway**   | JWT 是否合法、是否过期、是否黑名单   |
| **业务服务**  | 权限、数据权限                       |

👉 **网关一旦开始校验“登录逻辑”，你就把业务耦合进来了。**

------

### 2️⃣ 你现在已经有“双重校验”了

你现在的链路是：

```
Gateway
  ↓
Auth 服务（/login）
  ↓
JWT 签发
  ↓
Gateway（再解析 JWT）
  ↓
业务服务
```

✅ JWT 本身已经是一个 **不可伪造的凭证**

❌ 再解析一次，只是浪费 CPU

------

## 二、哪些接口**必须在网关校验 JWT**？

✅ **所有“已登录用户才能访问”的接口**

但不是“校验账号密码”，而是：

| 校验项           | 网关是否做 |
| ---------------- | ---------- |
| JWT 签名是否合法 | ✅ 必须     |
| JWT 是否过期     | ✅ 必须     |
| JWT 是否在黑名单 | ✅ 必须     |
| 用户是否存在     | ❌ 不做     |
| 密码是否正确     | ❌ 不做     |
| 角色是否够       | ❌ 不做     |

👉 **网关只做“凭证有效性”，不做“业务合法性”。**

### 3️⃣ 用户信息透传（可选，不解析 JWT）

```
@Component
public class UserInfoFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        return exchange.getPrincipal()
                .cast(Jwt.class)
                .flatMap(jwt -> {
                    ServerHttpRequest req = exchange.getRequest().mutate()
                            .header("X-User-Id", jwt.getClaimAsString("user_id"))
                            .header("X-Username", jwt.getClaimAsString("username"))
                            .build();
                    return chain.filter(exchange.mutate().request(req).build());
                })
                .switchIfEmpty(chain.filter(exchange));
    }

    @Override
    public int getOrder() {
        return -90;
    }
}
```

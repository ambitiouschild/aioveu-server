## 一、先给结论（你可以直接拿去拍板）

> **如果你已经在「资源服务器（业务服务）」里配置了 `JwtBlacklistFilter`，  
> 那么网关里再校验一次黑名单，是重复校验，必须删。**

---

## 二、为什么这是“多此一举”

### 你现在的链路是：

```
Client
  ↓
Gateway（JwtBlacklistFilter ❌）
  ↓
Auth 服务（登录 / 刷新）
  ↓
资源服务（JwtBlacklistFilter ✅）
```

### 问题在哪里？

| 问题       | 说明                     |
| ---------- | ------------------------ |
| 重复 IO    | Redis 查两次             |
| 重复逻辑   | 黑名单规则维护两份       |
| 一致性风险 | 两边判断不一致           |
| 性能损耗   | 每个请求多一次 Redis     |
| 职责混乱   | 网关开始关心“业务安全态” |

👉 网关不该知道“这个 token 是不是被踢下线了”

---

## 三、黑名单本质上属于什么？

| 校验类型                | 归属                     |
| ----------------------- | ------------------------ |
| JWT 签名                | ✅ 网关 / Resource Server |
| JWT 过期                | ✅ 网关 / Resource Server |
| 黑名单（登出 / 踢人）   | ✅ 资源服务器             |
| 用户状态（禁用 / 锁定） | ✅ 资源服务器             |
| 权限 / 数据权限         | ✅ 资源服务器             |

👉 黑名单 = 业务安全态，不是基础设施态

---

## 四、什么时候“必须”在网关校验黑名单？

只有下面 3 种情况之一成立，才需要：

### ✅ 情况 1：你有「纯前端直连网关」的场景
比如：
• 前端直接调网关

• 后面没有统一资源服务

• 多个异构系统（非 Java）


👉 网关是最后一道门

### ✅ 情况 2：你需要“提前熔断”
比如：
• 黑名单命中率极高

• 黑名单是风控核心

• 不希望请求打到下游


👉 性能 > 架构纯洁性

### ✅ 情况 3：你的资源服务不可信
比如：
• 第三方服务

• 外包系统

• 历史遗留系统


👉 网关做兜底

## 五、你这套架构下，**正确做法是啥？**

### ✅ 网关里：**删掉 `JwtBlacklistFilter`**

### ✅ 资源服务里：保留

```java
@Slf4j

public class JwtBlacklistFilter extends OncePerRequestFilter {

    private final TokenManagerService tokenManagerService;
```

| 事情               | 是否保留 |
| ------------------ | -------- |
| JWT 签名校验       | ✅        |
| tenantId 解析      | ✅        |
| 路由 / 限流 / 熔断 | ✅        |
| 黑名单             | ❌        |

## 六、如果你担心「登出后还能访问」怎么办？

### ✅ 正确解法：**缩短 JWT 有效期 + 黑名单在资源服务**

• Access Token：5～15 分钟

• Refresh Token：黑名单

• 网关只校验签名 + 过期时间


👉 登出 = 让 Refresh Token 失效，Access Token 自然过期

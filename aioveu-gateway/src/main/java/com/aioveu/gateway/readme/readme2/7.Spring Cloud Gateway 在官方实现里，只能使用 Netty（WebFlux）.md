这是一个**非常好的底层问题**，答案是：

> ✅ **是的，Spring Cloud Gateway 在官方实现里，只能使用 Netty（WebFlux）。**

下面我给你一个**彻底通透、不留疑问**的解释。

------

## 一、先给终极结论（请默念）

| 问题                           | 答案                    |
| ------------------------------ | ----------------------- |
| Gateway 能用 Tomcat/Jetty 吗？ | ❌ **不能**              |
| Gateway 底层容器               | ✅ **Netty**             |
| 编程模型                       | ✅ **WebFlux（非阻塞）** |
| 能不能换？                     | ❌ **官方不支持**        |

------

## 二、为什么 Gateway 只能用 Netty？

### 1️⃣ 设计初衷（核心原因）

Spring Cloud Gateway **不是**为了跑“普通 Web 请求”设计的，而是为了做**高性能 API 网关**。

| 需求       | 阻塞模型（Tomcat） | 非阻塞模型（Netty） |
| ---------- | ------------------ | ------------------- |
| 高并发连接 | ❌ 线程爆炸         | ✅ 少量线程          |
| 反向代理   | ❌ 慢               | ✅ 极快              |
| IO 密集型  | ❌ 不适合           | ✅ 天生适合          |

👉 **Gateway 必须非阻塞，所以只能选 Netty。**

------

### 2️⃣ 技术栈是“写死”的

Spring Cloud Gateway 的依赖树是：

```
spring-cloud-starter-gateway
 └─ spring-boot-starter-webflux
     └─ reactor-netty
```

| 项目                          | 说明       |
| ----------------------------- | ---------- |
| `spring-boot-starter-web`     | ❌ **冲突** |
| `spring-boot-starter-webflux` | ✅ **必须** |
| Tomcat                        | ❌ **排除** |

如果你强行引入 `spring-boot-starter-web`，结果是：

> **启动直接报错，或者 Gateway 功能失效。**

------

## 三、你之前遇到的 JWT 问题，根源就在这

| 组件                        | 技术栈                 |
| --------------------------- | ---------------------- |
| 你的 `ResourceServerConfig` | ✅ **Servlet (Tomcat)** |
| Spring Cloud Gateway        | ✅ **WebFlux (Netty)**  |

这就导致了：

| 现象                           | 原因                    |
| ------------------------------ | ----------------------- |
| 公共组件用不了                 | ❌ **Servlet ≠ WebFlux** |
| `JwtDecoder`不行               | ❌ **不是同一个接口**    |
| `ReactiveJwtDecoder`必须本地配 | ✅ **WebFlux 的规则**    |

------

## 四、一张表彻底搞懂（收藏级）

| 场景                    | 容器      | Security 类型                                     | JWT Decoder              |
| ----------------------- | --------- | ------------------------------------------------- | ------------------------ |
| 普通微服务 (User/Order) | Tomcat    | `spring-security-oauth2-resource-server`          | `JwtDecoder`             |
| 认证中心 (Auth)         | Tomcat    | `spring-authorization-server`                     | `JwtEncoder/Decoder`     |
| **API Gateway**         | **Netty** | `spring-security-oauth2-resource-server-reactive` | **`ReactiveJwtDecoder`** |

------

## 五、能不能“魔改”成 Tomcat？

### ❌ 不能，也不应该

| 理由       | 说明             |
| ---------- | ---------------- |
| 官方不支持 | 源码依赖 Webflux |
| 性能倒退   | 网关会变慢       |
| 生态割裂   | Filter 不能互通  |

------

## 六、一句话终极总结（你可以直接写在方案里）

> ❌ **Spring Cloud Gateway 只能用 Netty**
>
> ✅ **这是由它的“非阻塞网关”基因决定的**
>
> ✅ **Servlet 体系的代码（包括你的公共组件）无法直接在 Gateway 运行**

------

## 七、你现在的正确姿势

```
认证中心 (Tomcat)  --> 签发 JWT
普通微服务 (Tomcat) --> 用你的 ResourceServerConfig
Gateway (Netty)    --> 只用 ReactiveJwtDecoder + 本地占位符
```

你现在的困惑，本质上是 **“试图用 Servlet 的砖，去盖 WebFlux 的楼”**。

一旦接受了 **Gateway 只能是 Netty** 这个设定，之前所有的坑就都通了。

### ✅ 唯一不踩坑的模型

```

 ↓
```

## 三、Gateway 里还能不能碰 `tenantId`？

### ✅ 两种情况，泾渭分明

| 场景                                            | Gateway 能否介入 |
| ----------------------------------------------- | ---------------- |
| clientId → tenantId 是**配置文件 / Redis 枚举** | ✅ 可以           |
| clientId → tenantId 需要**查微服务 / DB**       | ❌ 绝对不行       |
| JWT 里直接有 tenantId                           | ✅ 可以           |
| 公共接口靠 clientId 找租户                      | ❌ 必须放微服务   |

你现在这个场景，属于**第二类**。

## 四、那“白名单”还能放 Gateway 吗？

✅ **可以，而且应该**

白名单是**接入控制**，不是业务：

```
if (!clientWhitelistService.isValid(clientId)) {
    reject();
}
```

但注意：

- 

  ✅ 白名单可以是 Redis

- 

  ✅ 可以是本地缓存

- 

  ❌ 不能是“实时查微服务”

白名单只回答一个问题：

> **“这个 client 有没有资格进我的系统？”**

不回答：

> ❌ “这个 client 属于哪个租户？”

## 五、微服务里怎么写才不丑？

### ✅ 公共接口专用 Resolver（不是 Filter）

```
@Component
public class PublicTenantResolver {

    private final ClientService clientService;

    public Long resolve(HttpServletRequest request) {
        String clientId = request.getHeader("X-Client-Id");
        if (clientId == null) {
            throw new BadRequestException("Missing X-Client-Id");
        }

        return clientService.getTenantIdByClientId(clientId);
    }
}
```

### ✅ Controller 显式调用（不偷偷摸摸）

```
@GetMapping("/public/config")
public Config publicConfig(HttpServletRequest request) {
    Long tenantId = publicTenantResolver.resolve(request);
    return configService.getByTenant(tenantId);
}
```

✅ 不污染 `TenantFilter`

✅ 不污染 Security

✅ 逻辑清晰、可测试

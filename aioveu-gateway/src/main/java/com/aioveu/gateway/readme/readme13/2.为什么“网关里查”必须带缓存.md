## ✅ 一句话结论（先拍板）

## 一、为什么“网关里查”必须带缓存？

### 网关的定位

> 网关 = 高并发 + 低延迟 + 无状态

每一次请求，网关都经历：

```text
Netty EventLoop
 → Filter
 → 路由
 → 转发
```

### 如果你“裸查”：

```java
webClient.get()
  .uri("/tenant/client/" + clientId)
  .retrieve()
```

你立刻引入：

| 问题            | 后果           |
| --------------- | -------------- |
| 每次请求都 HTTP | RT 翻倍        |
| 下游抖动        | 网关抖动       |
| 下游雪崩        | 网关雪崩       |
| 线程阻塞        | EventLoop 卡死 |
| 无法横向扩展    | 网关瓶颈       |

👉 网关一旦“裸查”，就不是网关了，是“慢速代理”

---

## 二、那“网关里查”的正确姿势是什么？

### ✅ 唯一正确的模型

```
Gateway
 ├─ Caffeine / Redis 缓存
 ├─ Cache Hit → 直接返回
 ├─ Cache Miss → WebClient
 ├─ 写缓存
 ├─ 后续请求命中缓存
```

### ✅ 哪怕你只用一次 WebClient，也必须这样包一层

| 你当时做的事            | 问题     |
| ----------------------- | -------- |
| WebClient 查 tenantId   | ❌ 没缓存 |
| 每个请求都查            | ❌        |
| WebClient 解构 response | ❌        |
| 网关消费 tenantId       | ❌        |
| 没熔断 / 重试           | ❌        |

👉 不是“查不查”的问题
👉 是“裸查 + 消费”的双重错误

## 四、如果“时光倒流”，网关里查的正确写法

（仅供理解，不推荐你现在用）

```java
private final LoadingCache<String, Long> cache = Caffeine.newBuilder()
        .maximumSize(100_000)
        .expireAfterWrite(Duration.ofMinutes(5))
        .build(this::remoteFetch);

private Long remoteFetch(String clientId) {
    return webClient.get()
        .uri("/tenant/client/{clientId}", clientId)
        .retrieve()
        .bodyToMono(ApiResult.class)
        .map(ApiResult::getData)
        .block(); // ⚠️ WebFlux 里依然不优雅
}
```

⚠️ 即便这样：
• 仍然不理想

• 仍然不如“微服务里查”

• 仍然增加网关复杂度

| 维度           | 网关查           | 微服务查      |

## 八、最终总结（你可以直接拿去复盘）

> 之前在网关中通过 WebClient 查询 clientId → tenantId，
> 未使用本地缓存，导致每次请求都发起远程调用，
> 既破坏了网关的高性能特性，也引入了响应解构问题。
>
> 正确做法是：
> - 网关只做 clientId 白名单与透传
> - tenantId 的解析下沉至微服务
> - Filter 中仅通过本地缓存完成解析
> - 远程调用严格封装在 Cache Loader 中

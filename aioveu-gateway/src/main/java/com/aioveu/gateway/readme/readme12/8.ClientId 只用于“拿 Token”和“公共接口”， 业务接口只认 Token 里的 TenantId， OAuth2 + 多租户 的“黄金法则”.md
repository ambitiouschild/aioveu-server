| 微服务内部                     | ✅ `tenantId` | 不再关心 clientId        |

## 三、把你的话，翻译成“标准架构语言”

### ✅ 1️⃣ Token 里必须带什么？

```
{
  "sub": "10086",
  "client_id": "xinhuan-app",
  "tenant_id": 3,
  "scope": "read write"
}
```

- 

  `client_id`：审计用

- 

  `tenant_id`：**数据隔离用**

- 

  `sub`：用户身份

------

### ✅ 2️⃣ Gateway 在 Token 场景做什么？

```
// ✅ 只做
- JWT 验签
- 过期校验
- Header 透传（X-Tenant-Id, X-Client-Id）
// ❌ 不做
- 查 tenantId
- 调微服务
```

------

### ✅ 3️⃣ 公共接口为什么只能用 clientId？

因为：

- 

  ❌ 没有 Token

- 

  ❌ 没有用户

- 

  ✅ 只有应用身份

```
GET /banners
Header: X-Client-Id: xinhuan-app
```

Gateway：

- 

  校验 clientId 是否合法

- 

  转发

- 

  **不注入 tenantId**

👉 **下游服务自己根据 clientId 查租户（只读、缓存）**

## 四、一个“绝不能违反”的铁律

>
> - 
>
>   ❌ 不允许再查 `clientId → tenantId`

> -

>   ✅ 只允许从 Token / Header 中取 `tenantId`
>
> - 
>
>   ✅ SQL 里只允许 `tenant_id`

谁违反，谁就是**伪多租户**。

## ✅ 一句话先给结论（重点）

> `permitAll()` 当然能跳过鉴权——但前提是：请求先到达 Spring Security 的授权决策点，且路径匹配成功。
>
> 👉 你现在的 401 不是 `permitAll()` 失效，而是：请求在「到达授权决策之前」就被 OAuth2 Resource Server 的 JWT 校验过滤器拒了，根本没机会走 `permitAll()`。

`TokenRelay=` 没有直接废掉白名单，但它制造了「让请求带不上 Token 却被 JWT 过滤器当非法请求处理」的条件。

---

## 一、`permitAll()` 到底能不能跳过 JWT 校验？（官方语义）

### ✅ Spring Security 的铁律：

```java
.requestMatchers("/xxx").permitAll();
```

含义是：

> 即使请求没有 Token、Token 无效、甚至没 Authorization Header——一律放行，不调用 `AuthenticationManager`。

✅ 这是对的，文档级正确。

---

## 二、那为什么你还会 401？（关键来了）

### ❌ 不是 `permitAll()` 不跳
❌ 不是白名单没用

### ✅ 真正原因是两步叠加：

---

### ① `TokenRelay=` 的行为

```
浏览器/内部调用 → Gateway
  └─ 无 Token
  └─ TokenRelay 不添加 Authorization
  └─ 请求照常转发到 租户服务
```

到这一步 没问题。

---

### ② 租户服务 Resource Server 的过滤器链

你配了：

```java
http.oauth2ResourceServer(oauth2 -> oauth2.jwt(...));
```

这会注册：

```
BearerTokenAuthenticationFilter
  └─ 如果请求有 Bearer → 验签
  └─ 如果请求无 Bearer → 不抛异常，**留待后续授权决策**
```

⚠️ 理论上没错，到这里 `permitAll()` 是能生效的。

---

## 三、那 401 到底哪来的？（三种真实可能，按概率排）

---

### ✅✅✅ 可能性 ①（90%）：路径没匹配上 `permitAll`

你白名单配的是：

```yaml
- /aioveu/api/v8/app/tenant/oauth-client-wx-app/getTenantIdByClientId
```

如果租户服务有：

```yaml
server:
  servlet:
    context-path: /aioveu
```

Security 看到的 URI 是：

```text
/api/v8/app/tenant/oauth-client-wx-app/getTenantIdByClientId
```

❑ 不匹配 `/aioveu/...`
❑ `permitAll()` 没命中
❑ 走 `anyRequest().authenticated()`
❑ `BearerTokenAuthenticationFilter` 发现无 Token → 401

> 👉 这就是你要的答案：`permitAll()` 能跳过 JWT，但你的请求没命中 permitAll

---

### ✅ 可能性 ②：`authorizeHttpRequests` 在 `oauth2ResourceServer` 之后配置（旧 Bug）

如果你曾经写成：

```java
http
  .oauth2ResourceServer(...)
  .authorizeHttpRequests(...);
```

在某些 Spring Boot 版本下：
• `BearerTokenAuthenticationFilter` 先执行

• 无 Token → 直接 401

• 授权规则不执行


✅ 必须：

```java
http
  .authorizeHttpRequests(auth -> auth
      .requestMatchers("/**/getTenantIdByClientId").permitAll()
      .anyRequest().authenticated()
  )
  .oauth2ResourceServer(...);
```

---

### ✅ 可能性 ③：你看到的是 **Gateway WebClient 的 401 响应被包装**

你日志里是：

```
401 UNAUTHORIZED from GET lb://aioveu-tenant/...
```

这是 WebClient 收到租户服务返回的 401，不是 Gateway 主动拦的。

说明：
• 请求确实到了租户服务

• 没进 `permitAll()` 分支

• 被 Resource Server 以 401 返回

## 四、`TokenRelay=` 到底“冤不冤”？

| 说法                                 | 对不对                       |
| ------------------------------------ | ---------------------------- |
| TokenRelay 废掉 permitAll            | ❌ 错                         |
| TokenRelay 导致无 Token 请求被转发   | ✅ 对                         |
| 如果 permitAll 路径命中 → 能正常通过 | ✅ 对                         |
| 你这个 401 是 TokenRelay 间接造成    | ✅ 对（路径没命中 permitAll） |

👉 更准确的说法是：

> TokenRelay 让“无 Token 请求”继续发往下游，而下游 permitAll 路径没命中，于是被 OAuth2 Resource Server 401。
